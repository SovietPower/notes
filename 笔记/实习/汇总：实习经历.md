## 2022.7 华为

### 实习介绍

首先说一下这个实习的背景，我们组招了4个实习生，是为了秋招准备的，没分配什么任务，就给了我们一些东西学习。
然后现在云数据库有一个问题，就是哈希表一般是二倍扩容，涉及所有元素的拷贝，代价比较高，很影响用户体验，需要性能更好的哈希表或好的扩容算法。
...
无锁现在的应用场景？folly 是个比较新、用的比较多的C++库，但里面的并发哈希表都没有用无锁。Java, Go 这样的语言提供的并发map也没用无锁，这样看来应用是非常少的，应该和应用场景关系很大，所以通用库没用。
好像 OceanBase 用的多，有空去看看。
Linux 的 RCU (Read-Copy Update) 数据同步方式也是无锁？

无锁结构不止包含 CAS 重试，还包括操作重试（当一个操作失败时，可能要重新从头执行该操作），仍然很可能会有竞争。并且自选是浪费 CPU 的。
根本还是要减少锁粒度，降低写冲突率。

> 无锁算法本质就是一个乐观锁的思想，我们认为冲突概率很低，直接去执行一个操作，在中间或者最后再检查我们能否做出这个更新 或 能否使用这个结果。如果失败，就不断尝试直到成功。
> 其实本质还是 CPU 去做一个自旋锁，还是锁住了线程，只不过可以很快地确定操作成功还是失败，不需要等待其它线程。
> 还是会浪费 CPU 的，只有一些特别的场景，确定冲突概率很低的时候适合。

### 项目介绍

> [0] - 0000 - 0010 - [2] - 0100 - 0101 - [1] - 1000 - [3] - 1100

基于 split-ordered list、hazard pointer 的无锁哈希表。
其实比较简单，可以分成几部分来说：

首先是无锁链表，这个就参考了一个论文的实现，比较简单。
主要是基于乐观锁的思想设计的，我们认为线程间的冲突概率很低，所以对于某个操作，比如在链表中 insert 某个节点，我们不加锁、直接遍历链表、找到要插入的位置，比如在 pre 与 cur 之间，然后执行 CAS(pre, (0, cur), (0, newNode))。其中 pre 指向一个二元组，第一个值表示该节点有没有被删除，第二个数值是 pre 节点的 next 指针，本来是 cur，要改成新的节点。
如果 CAS 成功，说明没有其它线程对 pre 的状态或 next 进行了更改，也即没有和其它线程冲突，函数返回；否则发生了冲突，因为我们没加锁，要重新执行整个函数。

整体比较简单，难点应该在于：怎么证明这样一个算法 在任何情况下都是正确的。
这个要证明每个函数都是原子的，都存在一个线性化点，在执行到该点之前，函数相当于没有执行；在执行完该语句后，函数就相当于完成了操作。涉及的结论比较多，我没有仔细看。

然后是 hazard pointer。在并发环境下，程序不知道什么时候 自己持有的某个对象，会被其它线程删除。比如某个节点的指针，随时可能指向一个非法区域。
这个问题和 GC 很像，我们要确定一个对象不被任何对象引用，才能回收。所以对拥有 GC 的语言，我们是不需要担心这个问题的。
对于其它语言，我们可以使用 hazard pointer 指向一个对象，来保护对象不被回收。
如果一个对象要被删除，把它交给管理器，但不立刻进行回收。只有这个对象不被任何 hazard pointer 指向时，我们再回收它。
这需要我们维护一个待回收队列，检查轮询检查是否可删除。如果 hazard pointer 保护的对象很多，我们每次检查都回收不了几个，就会导致大量的检查开销。
但事实上，在常用的无锁结构中，我们最多只需要几个 hazard pointer 就可以了。对无锁链表来说，每个线程最多只需要 3 个 hazard pointer，也就是任意时刻，最多只有 3 个可能被其它线程回收、导致程序不正确的对象，我们各用一个 hazard pointer 保护。所以它的开销其实没有那么高，HP 总量不多，大部分对象可以立即回收。
所以我们可以通过 hazard pointer 来保证 线程持有的指针 总是合法的。

然后是无锁哈希表。
我们知道链式哈希，所有操作本质上都是对链表的增删改查。如果我们实现了无锁链表和 hazard pointer，直接将它用在哈希表中，就是一个 线程安全的 无锁哈希表了。
这就是那个论文实现的 lock-free hash table。

最后是 split ordered list。
这是一种哈希表的实现方式，本质上是链式哈希，与锁不锁没关系。
特别点在于，我们扩容时，只需要在链表中插入一个桶节点就可完成，不需要拷贝所有数据到新地址。
但好像没人用这个，可能是因为连续性太差，同一个桶中的数据分布非常乱；如果实现了增量扩容，大多数场景下，扩容影响也没那么大。

### 问题整理

**原子变量的使用**

- 每个节点的 next 指针。对 next 的读取使用 acquire 与 release，而 CAS 操作都使用 consume 和 release。
- 计数器，比如哈希表的节点数量 size。这个不影响同步，只需要保证原子性，所以用 memory_order_relaxed 去操作即可（fetch\_add、fetch\_sub）。

**业务背景 / 组内工作**





**任务**

- 参考 LevelDB 与 RocksDB，实现带引用计数的 LRU cache 和基于时钟置换算法的无锁、并发性能更好的 clock cache。
- 实现 block cache 缓存对磁盘或远程文件的读取，替代操作系统的 page cache，能允许程序更好的控制内存使用；由于之前的索引系统完全通过 mmap 映射本地文件的方式读写索引，而分布式存储系统无法支持 mmap，因此通过 block cache 读取文件能实现存储与计算分离，也便于本地调试。
- 后续可支持读取压缩后的索引文件，并将解压后的结果缓存，减少磁盘访问；可利用磁盘建立次级缓存，进一步减少读取远程文件的消耗。

> 压缩算法：https://github.com/facebook/zstd

参考 page cache，使用 xarray 替代 LRU cache 中的哈希表。





**感受**

1.	虽然架构方向的工作不像业务一样直接接触用户，但是了解使用方的需求、明确工作需要做到什么程度（比如是否严格要求数据安全、是否严格限制内存使用、是否用于生产环境）是非常重要的。如果忽略需求、考虑过多，做一些一两年内可能都用不到的功能和过度优化，很可能是浪费时间且无意义的。
不要毫无理由地优化、不要过早优化、不要优化无关紧要/影响不大的东西、不要假设复杂的代码一定比简单的代码更好更快。
2.	在做底层服务时，必须考虑代码改动对上游服务带来的影响，尤其是已经出现多年、被大量代码使用的服务。能让上游服务不需要做任何改动的方案，才算好方案。
3.	在开始写代码前，应该与组内成员讨论一下遇到的问题和选择的方案，一起验证一下方案是否有问题、是否可能改进，减少写无用代码、浪费时间的可能。
4.	在做一个重要需求之前，应该确保自己对工作涉及的内容和代码足够熟悉、在正式开发前构思好整体流程。如果在设计具体方案时没考虑到某些细节，很可能导致开发时遇到许多细节问题，影响开发效率。
5.	架构方向的工作中，很可能会遇到之前不了解或没做过的内容（比如BlockCache），此时需要多了解和分析相似开源系统中的代码实现，总结它们的特点与区别，以便选择或设计适合自己系统的实现。
6.	在晨会等会议向主管或组长汇报工作时，应该注重工作的结果、产生的价值、后续的安排和预期的完成时间。如果没有问题，工作过程只需要概括地讲下，甚至可以忽略。
7.	开发C++ 项目时，如果不遵循严格的代码规范，很容易导致代码结构混乱、可读性和可维护性变差。还需要多看《C++ Core Guidelines》，学习理解其中的规则（比如尽量用代码和类型直接表达程序的含义，而非依赖注释；尽量避免使用全局对象、增加代码的耦合性）；也要多学现代C++，用更新的语法写出更高效的逻辑（比如BlockCache就可以用C++20的协程实现异步并发读取，优化效率）。
8.	在编写代码时，应注意开放封闭原则：对象对于扩展应该是开放的。代码尽量考虑到后续可能的变化和需求，尽量做到只通过添加代码、不修改已有代码来实现新的功能。这样易于系统维护和扩展，也会降低出现问题的概率。一个好的系统抽象是实现开发封闭的好方式（比如ARTIS分为接口层、核心层、数据结构层和存储层，通过在存储层添加Mempool的子类BlockCache，就能在不影响其它层的情况下新增读写功能）。
