### redis
> https://blog.csdn.net/Oooo_mumuxi/article/details/105903889

是一个基本的链式哈希，不过扩容的方式很特别，是渐进式扩容。

**扩容过程**
首先给`ht[1]`分配空间，然后逐步将原哈希表`ht[0]`迁移到`ht[1]`：

- **active rehashing**
`server.c`中的`serverCron()`周期性被调用，执行各类需要异步处理的事。
每次调用`serverCron()`时，进行一次`incrementallyRehash()`，花费约 1ms 执行一部分迁移（迁移100个桶或遍历了1000个空桶后结束）
- **lazy rehashing**
哈希表执行基本的增删改查（`dictAddRaw`、`dictGenericDelete`、`dictFind`、`dictGetRandomKey`、`dictGetSomeKeys`）时，调用`_dictRehashStep()`进行少量扩容（迁移1个桶或遍历了10个空桶后结束）。

迁移完成后，释放`ht[0]`，将`ht[1]`赋给`ht[0]`。

rehash 可以扩容也可缩容。

### Split-Ordered Lists
所有元素形成一个链表，通过在链表内部添加两倍的桶节点，实现无需迁移的逻辑扩容。

**优点**

- 可以实现 lock-free 扩缩容，无需迁移。
- 几乎不会有浪费、不被使用的空间。扩容是为了增加桶来减少遍历每个桶的时间；不需要缩容。

**缺点**

- 空间连续性差，同一个桶的元素很可能位于不同位置。
- 需要多级地址表来获取要查询桶的地址。需要多约4次的随机访问。

### Extendible Hashing
> https://www.geeksforgeeks.org/extendible-hashing-dynamic-approach-to-dbms/

类似 Trie，哈希值二进制表示的低$k$位相同的元素 会放在同一个桶中。当该桶溢出时，根据$k+1$位是0还是1再分成两个桶。

**优点**

- 单次操作可能导致一个桶的若干次扩缩容，但最多只需迁移该桶一半的元素。如果每个桶不大，扩缩容花费就很低（也相当于被分摊到了每次操作）。
- 可以自动进行扩缩容，空间利用率高。
- 效率不会随元素的增多而降低。

**缺点**

- 需要足够大的数组 **directory** 保存每个桶的地址。
- 不能使用溢出桶。如果每个桶内元素数量过少，会导致 directory 很大，很可能浪费空间；如果桶内元素数多，会导致操作效率降低。
- 很容易导致 directory 过大，浪费空间，即使哈希函数的分布足够均匀。
> Suppose we have only two records per bucket and 3 records have the same key for the last 20 bits. Since no Local depth of a bucket cannot be bigger than the Global Depth, we will have global depth of 20, even though most of the Local Depths are in range from 1 to 5 bits.

### DPDK Hash Library
> https://doc.dpdk.org/guides/prog_guide/hash_lib.html
> https://www.quora.com/Why-is-cuckoo-hashing-not-a-popular-technique-even-though-it-has-worst-case-lookup-guarantees-Is-it-because-general-hashing-used-in-popular-implementation-of-HashMap-and-HashSet-performs-well-in-practice-but-still-no-O-1-guarantees-on-worst-case
> https://www.cs.cmu.edu/~binfan/papers/nsdi13_memc3.pdf

DPDK 用一个数组保存每个`key-value`对，用一个保存了 所有`key`的摘要`signature`和该键值对在数组中的索引`index` 的哈希表进行查找。

每个操作前，对`key`进行哈希产生它的桶编号、摘要和索引。

- 摘要是16位的，用于代替`key`在对应桶中快速检查它是否可能出现。如果该桶中有相同的摘要，则再检查数组中该索引处的`key`是否相等。
- 一个桶的多个摘要可放在同一 cacheline 中，减少读取次数。
- 插入时如果要移动元素，只需移动该摘要和索引，效率会更高

哈希使用`Cuckoo Hashing`：会为每个`key`产生两个桶编号，称为其主要桶和次要桶。

- 对于插入：如果在第一个桶中插入失败，则将桶中不在其次要桶中的随机一项，移动到它的次要桶，将要插入的元素放在此处；如果被移动的项仍不能在次要桶插入，则再随机移动次要桶中的另一项。重复，直到找到空位（成功）或无法移动（失败）。
- 对于查询：依次在第一、第二个桶中查询，都查询不到即为不存在。
- 对于删除：同查询，依次查找两个桶。如果是并发场景，则不会真正删除元素，需由用户负责安全回收。
- 如果需要在插入失败若干次后进行扩容，则在失败的次要桶后连接一个溢出桶。查询、删除都需检查次要桶的溢出桶，且删除非溢出桶后，要将溢出桶中的一个元素移回非溢出桶。

特点基本与`Cuckoo Hashing`相同。
**优点**

- 空间利用率高。
- 查询效率高：最多只需要两次可并行的读。

**缺点**

- 插入效率低：可能需多次移动。
- 并发安全算法的实现难：插入会涉及多个不可预知的桶。链式哈希每次只需在一个桶上加锁或 CAS。
- 需根据应用场景调整参数：组相连度/要尝试桶的数量、插入失败前进行移动的次数 的最优值都可能随场景变化。

### Go map
与 DPDK 相同，key 的哈希值的一半用于选择桶，另一半保存用来快速比较。
同一个桶中的键值分开存储，便于内存对齐和顺序查找 key。

扩容与 redis 类似，每次迁移若干个桶。
分为翻倍扩容和等量扩容，后者用来代替缩容（只减少溢出桶的数量）。

> https://halfrost.com/go_map_chapter_one/#toc-21
> 哈希算法选用高效的 memhash 算法 和 CPU AES指令集。AES 指令集充分利用 CPU 硬件特性，计算哈希值的效率超高。
> key - value 的排列设计成 key 放在一起，value 放在一起，而不是key，value成对排列。这样方便内存对齐，数据量大了以后节约内存对齐造成的一些浪费。
> key，value 的内存大小超过128字节以后自动转成存储一个指针。
> tophash 数组的设计加速了 key 的查找过程。tophash 也被复用，用来标记扩容操作时候的状态。
> Go 1.9 版本以后，Map 原生就已经支持线程安全。

### Google SparseHash
> https://tristanpenman.com/blog/posts/2017/10/11/sparsehash-internals/
> https://github.com/sparsehash/sparsehash

将数据按 key 分组，每组有一个位图表示每个哈希值是否已被映射，有一个外部数组存储该组内的数据。

<img src="https://tristanpenman.com/images/sparsehash-internals/sparsetable@2x.png" style="zoom: 62%;" />

这张图里每个组的大小为8，有一个大小为8的 bitmap。key 在该范围内的数据都会保存在它指向的数组中。
实际使用是，组大小为 48。

本质也是链式哈希，但通过将 48 个 key 放到同一个用 bitmap 表示的桶中，减少了桶数组带来的额外开销。
原先每个桶包含一个指针指向桶链表，所以每个桶的代价是32/64位。
而 sparse hashmap 在数据块之外的开销是：48位位图，8位 count 记录指向数组的大小，32/64位指针指向数据数组，这一共可以表示 48 个桶，所以每个桶 的代价只有 2 位。

**优点**

- 每个桶只需要很小的额外开销 (2 bit)，在存储对象不大时空间利用率非常高（存储对象大时利用率也高，但因为对象占比太大，所以该优点并不显著）。

**缺点**

- 由于它把 48 个 key 对应的数据放在了同一个桶里，当元素很密集时，访问效率就非常低（冲突概率高 48 倍）。
    所以只适合稀疏的情况。

### Folly F14Map
> https://github.com/facebook/folly/blob/main/folly/container/F14Map.h

F14 实现了三类哈希表：

- `F14NodeMap`：链式哈希，~~（对象较大时）内存利用率高~~。
- `F14ValueMap`：开放寻址，将键值对一起排列在数组内，适合小型对象。
- `F14VectorMap`：开放寻址，但数据不放在数组内，数组内只有 key 和指针。适合大型对象。

一般使用`F14FastMap`，会根据键值对类型选择使用`F14ValueMap`或`F14VectorMap`（即默认选择开放寻址，但根据键值对大小决定，是将键值对放到哈希数组内，还是放到外面再用指针访问）。

F14 的每个键值对都有 1B 的 tag：最高位为1，低7位为二次计算的哈希值（或哈希值的高7位，反正不与映射位置重合）。
14个 tag 和 2B 的辅助信息，组成一个 16B 对齐的`chunk`。
查询以`chunk`为单位，利用 SIMD 指令集，可将给定哈希值一次与14个 tag 进行比较，然后才检查相同的键值对，以减少 key 的比较次数。

使用`chunk`为单位，大部分查询所需的探测次数会很少。~~而如果使用链式动态分配`chunk`，`chunk`很可能填不满，会比较浪费。~~

### Abseil Swiss Table
> https://github.com/abseil/abseil-cpp/blob/master/absl/container/flat_hash_map.h
> https://abseil.io/blog/20180927-swisstables
> Rust 也使用了该实现

与 folly 类似，提供 链式的`node_hash_map`和开放寻址的`flat_hash_map`。

`flat_hash_map`同样为每个条目记录 1B 信息（称为 Control Byte），低7位为部分哈希值，高位用来标识该位置空闲或被删除。
16个 Control Byte 称为一个 Group。查询利用 SIMD，以 Group 为单位。
一点不同是，`flat_hash_map`会预先分配与哈希表同样大小的`slot_type[]`存储键值对，所以无需保存索引。其大致结构为：
```cpp
struct raw_hash_set
{
	ctrl_t ctrl[capacity+kWidth-1]; // ctrl_t: int8_t
	slot_type slots[capacity];
	// raw_hash_map 实际就是 slot_type 为 map_slot_type（包含 std::pair<const K, V>）的 raw_hash_set
};
```
里面`kWidth = 16`为 Group 大小。多分配的`kWidth-1`个是对`ctrl[0..kWidth-1]`的拷贝，可保证在对`ctrl`的末尾几个元素读取整块时不会越界。

### Parallel Hashmap
> https://github.com/greg7mdp/parallel-hashmap

Abseil 或其它常见的开放寻址哈希表，在`resize()`时，会分配原数组2倍大小的新数组进行迁移，导致临时3倍的内存占用。
为此，PHMap 在 Abseil 的基础上将哈希表分为16个`submap`，扩容时逐个扩容，将`resize()`时的内存峰值从`3x`降低到`(2+1/16)x`。

![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220825180146620-731187589.png)

其实也是很常见的分片方法。通过分片，可实现简单的无锁并发：每个线程只执行属于自己的`submap`的操作。
这会略微提高内存峰值，但效率相比单线程可提升三倍。

如果需要在内部加锁（允许每个线程任意操作`submap`），多个`submap`也减少了线程间的锁竞争。

### Boost flat_map unordered_flat_map

分别是 std::map 与 std::unordered_map 的替代。

flat_map：是一个有序数组，每个位置是 pair<key, value>，查找时二分，插入和删除时都需要移动后面的数据。所以查询复杂度为 O(logn)，插入删除为 O(n)。

unordered_flat_map：开发寻址的哈希表，使用二次探测。查询插入为 O(1)。

**benchmark**

由于这两个哈希表的局部性更好、常数更小，所以在数据量较小的情况下会比 std 实现更好。而且很多情况下哈希表都不会存太多数据。
但是，std 的两个哈希表不要求 value_type 是可复制赋值和可复制构造的，这两个需要。

flat_map：

- 当元素数量较少、对象较小（移动代价低）时，flat_map 的插入删除依旧能优于 map，查询甚至能优于 unordered_map。
- 遍历效率很高，优于 std。
- 因为需要二分，所以 key 应该可高效地比较大小，当 key 为字符串时会影响查询性能。
- POD 类型的拷贝效率更高，所以非 POD 可能会影响插入和删除性能。

unordered_flat_map：

- 查询更新性能都优于 std 和 flat_map。

**哈希调研总结**

哈希表的实现没有很多样化，主要都是在传统做法上做小部分改进。
且大部分实现都以优化插入、查询效率、空间利用率、内存友好为主，能否支持回收、迭代器是否会失效、扩容带来的影响等并不关键。

链式哈希：

- 将所有节点组织成一个全局链表，可提高遍历的效率（std::unordered_map）；通过修改桶指针指向指定节点，可实现无需迁移的扩容（[Split-Ordered Lists](http://www.cs.ucf.edu/~dcm/Teaching/COT4810-Spring2011/Literature/SplitOrderedLists.pdf)）。
- 分段锁：并发环境下，将哈希表分为比如16段，对每段都单独加锁和操作。更进一步，我们可以为每个桶加一个锁。
  由于每个桶内的操作应该是非常少的，很快就能完成，用 CAS 替代锁应该会取得更好的效果。

开放寻址：

- 利用 SIMD 指令集，一次比较多个元素，减少探测次数。
- 当元素较大时，将键值对存储在外部数组，哈希表内存指针或索引，减少扩容时移动开销。根据元素大小选择使用哪类 map。

并发哈希：

- 目前看的主流实现并不复杂，是在传统哈希上通过 CAS 加锁，利用 HazardPointer 或 不回收元素 保证并发访问安全。没有使用 lock-free。

其它：

- 对哈希值进行混合，保证即使默认或用户提供的哈希函数效果较差时，也不会出现严重的性能下降，提高稳定性。
- 随机初始化哈希种子，保证非确定性的迭代顺序，可部分[避免 DoS 攻击](https://github.com/rust-lang/rust/issues/36481)。
- 优化局部性。

**任意倍扩容的尝试思路**

- （不太理解桶的数量会不会很影响空间占用）如果用户经常少量扩容，则最初分配$2n$个桶（$n$为容量），当扩容总量不足$2n$时，无需扩容；如果用户可能进行大幅度扩容，预先使用$4n$或$8n$个桶。
在需要下次扩容前，可允许用户在任意相对空闲的时刻，手动或定期执行可能影响性能的扩容。
- 把快速的扩容/迁移作为一项付费服务，分配更多的资源。
