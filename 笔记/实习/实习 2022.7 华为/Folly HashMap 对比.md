## benchmark 总结
benchmark 涉及内容：

- 插入、查询、删除、遍历效率。
- 发生大量插入/删除后的查询效率。
- POD 和非 POD 对象的影响（通常影响不大，但涉及多次拷贝时非 POD 可能很影响效率）。
- 适用性，如：std 的两个哈希表不要求 value_type 是可复制赋值和可复制构造的，但有些哈希表需要（[vector](https://zh.cppreference.com/w/cpp/container/vector) 也要）。

测试注意点：

- 由于测试一般会多次运行取平均值，所以每次测试前要刷新 cache，以更符合实际场景。

**刷新 cache 中旧内容的方式**

> https://zhuanlan.zhihu.com/p/661418250

1. malloc 分配 cache 大小（通常就是 L3 cache 大小）两倍的空间。
    如果只有一倍，部分区域可能重复映射到一个 cacheline 上，导致并非所有 cache 都被 flush。两倍大小可以尽可能保证所有 cache 都被 flush。
2. memset 设置整个空间为非 0 值，如`memset(mem, 1, 2 * L3_CACHE)`。
    如果设为 0 值，操作系统可能直接分配 zero-page 以减少分配开销（值为 0 的预设页，这样读时可以正确返回 0、不会触发 page fault；写时触发 page fault，使用 copy on write 的方式新分配一个页写）。
3. 将内存赋值给 volatile 全局变量，避免做某些优化（甚至直接优化掉）。
4. 随机访问空间中的某个 cacheline（如：cacheline 为64B，那么每次随机访问 rand * 64 起始位置的 int64），加到一个 volatile 全局变量里。
    重复 10 * cacheline 总数次，以保证每个 cacheline 都被访问过（此外淘汰策略可能不是严格 LRU，而是伪 LRU，所以一次遍历不一定能保证全部旧数据被淘汰）。
    需要随机访问而非顺序访问，因为有些 CPU 在遇到顺序访问时，可能将淘汰策略从 LRU 换为 MRU（淘汰最近访问的数据），导致 cache 中老的数据依旧保留。

指令 cache 不太好清空。icache 比较大，需要执行一大段程序才能 flush。但测试中不同的测试对象和轮次都能满足 icache 总是命中（除了第一次执行），所以相对影响不大。

## 结果总结

AHMap：AtomicHashMap
CHMap：ConcurrentHashMap

CHMap 适合的场景（以下条件满足任一即可）：

- 需要删除元素：AHMap 不会删除元素，这些元素仍会占据空间并影响后面的操作。
  开放寻址不方便删除元素，也是为了安全。用链式的话，就必须动态分配节点、改变链表结构，只通过简单的 CAS 是不行的。
- 需要修改值，且不便于在外部加锁：AHMap 不支持并发修改。
- 需要高空间利用率：CHMap 只需在插入每个元素时新建该节点，AHMap 需在最初初始化整个数组。
- 哈希表大小不确定：AHMap 必须设置足够大的初始大小，因为其扩容后的容量最多为初始大小的19倍。CHMap 没有限制。

AHMap 更适合的场景：

- 表内数据量较多但又不太多，比如 1e6~1e8 左右：数据量小时优势不明显，但扩容方式又限制了其最大容量。
- 需要多次遍历整个哈希表：CHMap 的遍历效率非常低。
- 操作类型单一，如插入后再大量查询，插入后再遍历。

AHMap 在重复执行同一种操作时效率很高，一般能快 2~4 倍；即使效率随扩容次数线性降低，但最终仍不差于 CHMap（也因为它扩容量有限）。
但当多种操作一起进行，且数据量不大（如1e6以内）时，AHMap 表现的并没那么理想。
此外支持的操作少，所以其应用场景比较有限。
所以即使 AHMap 用简单的方式实现了很高的效率，这种简单导致的多种限制，也大大降低了它的实用性。

## 测试
> https://martin.ankerl.com/2019/04/01/hashmap-benchmarks-01-overview/

### **构造**（不使用）
**测试方式**
测试$2^{10},\ 2^{18},\ 2^{26}$大小的 map 初始化：

**结果**
![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101055397-869631250.png)

用时都不大。
由于哈希方法不同，AHMap 在创建时就需初始化每个节点，用时更多；CHMap 是在插入时再初始化新的节点。


### **并发查询+少量插入**
**测试方式**
初始的哈希表预先分配足够大空间。
每 4 次串行插入进行 40K/400 次并发查询。
分别测试 0%/25%/50%/75%/100% 的查询成功率。
一共进行 2500/250000 轮（最终为 1e4/1e6 个数据，1e8 次查询）。

**结果**
![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101137047-1082715298.png)

查询成功率越低，AHMap 用时越低，但此影响对 CHMap 不明显。
查询成功率较低($\leq 50\%$)时，AHMap 快约2倍；较高时差距不大。
如果哈希表初始大小足够大，数据量的多少对查询的影响不明显。


### **并发查询2**
测试 AHMap 并发查询效率 和查询成功率、初始大小/最终大小（两者相关）的关系。

**测试方式**
定义刚好够大的 AHMap，插入若干个元素，然后测试 5e7 次查询。

**结果**
![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101204257-1334344105.png)

当表内元素密集时，查询成功率、表内元素的多少，对查询操作的影响不大。

**分析**
AHMap 使用线性探测，当哈希表元素少、比较稀疏 或 冲突率很低时（如前一个测试的结果），低查询成功率能使`find`能很快找到空元素 然后返回 empty；反之如果元素密集（如本次的多数测试），失败的查询可能会比成功的查询遍历更多元素，导致效率降低。
所以当表稀疏且查询成功率低时，AHMap 效率会提高，否则影响不大。

因为 AHMap 的 find 在遍历到空元素时停止，表内空元素比例一定(`1 - LoadFactor`)，所以 表内元素数量 对 每次遍历的最多元素数量 影响也不大，对整个操作的影响不大。


### **并发插入**
**测试方式**
分别测试初始哈希表较小和足够大（预分配空间）的情况。
由于 AHMap 总大小的限制，初始哈希表大小为 $最终大小/18$，可能并不符合 CHMap 的实际情况。
8线程共进行 2e5/2e6/2e7 次插入。
8线程各进行 $25\times 10^{3/4/5}$ 次插入和 5 次查询，共 2e5/2e6/2e7 次。

**结果**
![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101233005-1598299232.png)
\*这里的时间包含了扩容过程所需的时间，而不只有插入。

扩容会严重影响插入的效率。
即使不扩容，随着数据量的增长，CHMap 效率降低的很快，但 AHMap 几乎不受影响。


### **并发插入2**
上一个测试 AHMap 在扩容情况下，`2e6`次插入用时远小于`2e5`，而`2e5`却与`2e7`接近。所以做了下面这个测试。

**测试方式**
定义不同初始大小的 AHMap，分别并发插入不同量的数据。

**结果**
![插入总量为`finalSize`](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101253951-437527287.png)

初始大小越大，扩容次数越少，效率越高。

**问题**
当初始大小为最终大小的固定比例时，效率与数据量的关系更像正弦函数，而非正相关。
不太理解原因，可能是哈希函数的问题。毕竟如果只是几次扩容导致的，不会相差这么大。

此外该测试结果浮动很大，如果不是机器问题，那说明 AHMap 的性能比较容易受数据影响、不太稳定。

### **并发删除**
**测试方式**
先插入 1e4/1e7 个元素，然后测试 1e4/1e7 次的删除。
与查询类似，分别测试 0%/25%/50%/75%/100% 的删除成功率。

**结果**
![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101334589-829982305.png)

AHMap 要快 4~10 倍。

删除流程与查询类似，所以元素稀疏时，删除成功率越低，AHMap 用时越低。CHMap 则受此影响更小。


### **并发修改**
AHMap 仅能通过修改迭代器更新值，不保证并发安全。
CHMap 除了迭代器，还提供了`assign_if_equal`，类似 CAS 的线程安全的更新。


### **插入+查询+扩容**
**测试方式**
初始哈希表大小为 $最终大小/16$（受 AHMap 限制）。
8个线程分别执行同样数量的插入和查询，总共 1e6/1e7/1e7 次插入和 1e7/1e6/1e7 次查询。
插入查询次序随机，在都未执行完的情况下 $p_{Insert}:p_{Find}=3:1$。
分别测试 0%/50%/100% 的查询成功率。

**结果**
![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101348012-112444120.png)

对 CHMap，插入+扩容的耗时远大于查询。
当需要进行扩容且插入量较少时，CHMap 要优于 AHMap，否则更差。

**问题**
测试 AHMap 进行1e6次插入、1e7次查询时，结果波动幅度极大（最少4s，最多3.5min，且大部分时候都要2min以上），但每次的种子是一样的，也不像是机器问题（只有这部分测试有波动）。
没有找到原因。

只进行1e4次插入、1e6次查询，用时也很高。
当只进行1e6次插入，或先插入再只进行1e7次查询时，用时均很稳定（几秒）。


### **插入+查询+删除**
**测试方式**
8个线程分别执行相同次数的插入、查询和删除，总共 1e7 次插入/查询 和 元素总数的 0%/25%/50%/75%/100% 次的删除。
操作次序随机，在都未执行完的情况下 $p_{Insert}:p_{Erase}:p_{Find}=6:5:5$。
查询成功率取100%，但可能查询已删除的`key`。

**结果**
![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101400156-1107209838.png)

没有删除操作时，AHMap 优于 CHMap。
一旦包含删除操作，AHMap 的用时上升，而 CHMap 的用时大幅下降，最后远优于 AHMap。


### **扩容后效率**
**测试方式**
先进行 1e7 次插入，然后8个线程均执行 1250'000 次 插入/删除/查询，共 1e7 次。

**结果**
![\*Insert 中包含因插入导致的扩容的花费时间](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101411323-1504588476.png)

扩容对 AHMap 的所有后续操作影响都很大，但对 CHMap 影响不大。
但在 1e7 范围内，AHMap 的效率在扩容后仍然更高。


### **遍历**
**测试方式**
测试初始哈希表较大时：若容器已有 25%/50%/100% 元素，8线程并发遍历所有元素的时间。
测试初始哈希表较小时：插入 1e5/2e6/4e7 个元素后，8线程并发遍历所有元素的时间。

**结果**
![](https://img2022.cnblogs.com/blog/1143196/202208/1143196-20220822101424893-1095697010.png)

AHMap 的效率很高，比 CHMap 高10倍左右。
哈希表内的元素比例 对遍历 AHMap 的影响不大，但对 CHMap 影响很大：只有元素很少时，遍历 CHMap 才比较可行。
扩容次数对遍历效率没有影响。

**分析**

- AHMap 迭代器的`++` 是枚举每个哈希数组（SubMap）的所有位置，直到找到下个值；CHMap 迭代器的`++` 是枚举每个段的每个桶，直到找到下个非空桶。<br>
而 SubMap 元素总量和桶数量都约等于元素总量，所以不管表内元素多少，两者都相当于枚举了最大容量个元素。
- CHMap 枚举每个元素时，都需使用 Hazptr 的`reset_hazptr()`保护，需要多个函数调用和对共享变量的原子操作。
- AHMap 遍历无效元素时很快（只需`if(!valid()) ++id;`），且访问空间连续；但 CHMap 即使在遍历空指针，也需原子重设 Hazptr。<br>
所以即使 AHMap 的 `Erase`不能真正删除元素，也不会很影响它的遍历效率。


### 其它影响因素

- 线程数量：8
- `Key, Value`数值范围：`int64_t`
- `Key`低8位的分布（CHMap 使用低8位进行分段）：随机。
- `Key`的数据范围：整个`int64_t`内随机。
