**CPU usage/utilization 与 CPU load**

[CPU 负载](https://en.wikipedia.org/wiki/Load_(computing)) (**CPU load**)：使用或等待使用 CPU 的进程数（或线程数？）（或 CPU 运行和等待队列的长度）。
进程有多种状态，负载统计的是处于 runnable 和 running 状态的进程。

当负载始终小于1时，每个需要使用 CPU 的进程都可以立即使用它，而无需等待；大于等于1时，如果有进程需要使用 CPU，会由于 CPU 不可用而等待。如果 cpu.load 的值大于 CPU 的核心数，那么可能代表系统的 CPU 负载过高，有些进程需要等待 CPU 资源。如果 cpu.load 的值小于 CPU 的核心数，那么意味着系统的 CPU 资源还有剩余，可以处理更多的进程。
负载可作为 CPU 瓶颈的参考指标。

> 虽然多数系统的负载统计的是 runnable 和 running 进程数，但 Linux 也会统计处于 uninterruptible sleep 状态的进程（通常是等待硬盘活动的进程）。因此当许多进程因为 IO 阻塞时（比如硬件较慢或使用的 NFS 出现问题），此时的负载不能真正表示 CPU 使用的情况，即使负载较大，CPU 可能也是够用的（但还是能表明进程大概要等多久）。
> uninterruptible sleep 通常是很短暂的 sleep，因此 linux 将其近似认为是 runnable，但当 IO 出问题时这个睡眠就会很长了。

CPU 利用率 (**CPU usage**, busy)：CPU 处理任务（非空闲状态）占总时间的百分比，即 使用中的时间/(使用时间 + 空闲时间)。

> CPU 某一时刻的状态可分为多种：
> us (user space)：运行非内核代码花费的时间。
> sy (system, kernel space)：运行内核代码花费的时间。
> id (idle)：空闲时间。
> wa：等待 I/O 花费的时间。
> st：从虚拟机中窃取时间。
>
> 可以简单认为 cpu usage = (us + sy) / (us + sy + id)。

两个值都是定期采样得到的。

**两者的关系：**

- CPU 负载是比利用率更好的指标。
    比如机器 A, B 的利用率都是 100%，但 A 的负载是3，B 的负载是 6，两者的 CPU 需求肯定是不同的。
- 通常负载高就代表利用率高。
    但在 Linux 上，如果是 IO 密集型任务导致的高负载，则利用率不一定高。这种情况下，即使 CPU 利用率低，但负载较高，CPU 也可能是瓶颈？因为进程潜在的要与大量进程竞争？
- 利用率高不代表负载高。即使利用率是 100%，只要负载为1（对单核），就不会影响执行。这也是理想状态。
- 吞吐量 (throughout，单位时间内处理任务的数量) 是优化的根本目标。

总之需要根据实际情况分析：当任务没有高吞吐量要求时，运行它的 CPU 利用率或负载低也不是坏事，因为可以腾出 CPU 供其它进程运行；当利用率高时，只要负载不高，也是 ok 的。

> 高负载低利用率可能原因：
> - 严重的 I/O 等待：如果许多任务在等待磁盘或网络 I/O，那么 CPU 可能处于空闲状态，但 Load 仍然很高。
> - 锁争用：进程在争用资源（如文件锁、网络锁），导致大量任务不能立即得到 CPU 执行。
> - 不均衡负载：某些核心可能忙于处理，而其他核心处于空闲状态，这种情况下可以优化任务分配来平衡负载。
>
> 可以问 GPT 及可能排查方式。

### perf

> https://zhuanlan.zhihu.com/p/642276891
> https://zhuanlan.zhihu.com/p/445260558
> https://zhuanlan.zhihu.com/p/471379451
>

>
> perf 的原理是以固定的频率产生中断，然后窥探函数调用栈的状态。如果连续多次都发现调用栈状态相同，那么就可以假设在这一段时间内该函数在持续执行。

**常用**

- 看 cpu：perf record -e cpu-clock -g -F 1000 ./run.sh
- 看分支预测率和 page fault 数： perf stat ./run.sh
- 看 cache 命中率（各层分别输出，可能不支持）：perf stat -e L1-dcache-loads,L1-dcache-load-misses,L1-dcache-stores,L1-dcache-store-misses,L1-icache-load-misses,LLC-loads,LLC-load-misses,LLC-stores,LLC-store-misses ./run.sh
- 看 cache 命中率（不区分缓存层级，累计所有）：perf stat -e cache-references,cache-misses ./run.sh

**常用事件**

- cycles：默认事件，统计指令或函数消耗的 CPU 周期数。
- cpu-clock：统计消耗的 CPU 时间，可能比 cycles 更准确。
- instructions：统计 CPU 执行的指令数，可与 cycles 计算出每条指令的平均周期数 (CPI)。
- cache-references, cache-misses：cache 总访问次数与 miss 次数。
- branch-instructions, branch-misses：表示分支指令的执行次数与分支预测失败次数。

**perf report**

`perf report -i perf.data`查看采样数据（注意必须要 -i。似乎默认会以 perf.data 作为输入）。

> Self：是最后一列的符号（可以理解为函数）本身所占比例。
> Children ：是这个符号调用的其他符号（可以理解为子函数，包括直接和间接调用）占用的比例之和。
> [.]：代表该调用属于用户态，若自己监控的进程为用户态进程，那么这些即主要为用户态的cpu-clock占用的数值，[k]代表属于内核态的调用。
> 通过方向键和回车，可以看到函数的调用关系，同时以汇编代码的形式展示资源的消耗情况。

#### 使用

**mov/cmp 等普通指令占比高**

> https://stackoverflow.com/questions/63990981/why-does-cmp-instruction-cost-too-much-time
> https://stackoverflow.com/questions/57319960/why-mov-instruction-cost-so-much-time

这类普通的、周期短的指令应该耗时很短，但会因为一些原因占比很高：

- 等待操作数被写入：操作数可能依赖其它耗时高的指令完成（数据依赖）（比如 intel 下的 div）。
- 等待使用的寄存器被释放：使用的寄存器被其它耗时高的指令占用？
- 需要访存但未命中 cache：发生了 cache miss（注意不一定是在这条指令访存，可能是前面有 mov 访存，而当前指令是第一条用 mov 结果的指令）。
  通常 cpu 可以继续执行其它指令，但如果它是 cmp + j 就没法继续执行（需要决定控制流），只能在这里等待。
- 分支预测失败：CPU 需要重新执行被错误预测的指令，但和该场景有关吗？

实际会因为乱序执行和优化很乱。



**怎么看**

child 高说明该函数占用高（但不一定是这个函数）；self 高说明该函数耗时高（需要优化）。

### 火焰图

> Linux 下有 perf 命令，可以分析程序的性能、函数的耗时和调用关系。通过`perf record`提供的信息，可以用应用生成程序执行的火焰图。

> [火焰图](https://blog.csdn.net/m0_65931372/article/details/126539238)可以直观地展示程序性能的瓶颈在哪个函数上。
> 横轴是时间，反映函数的耗时（但不完全是）；纵轴是调用栈，反映函数的调用关系。
> 越高的位置，函数调用栈越深；越宽的位置，函数执行时间越长。
> 通常要优化的点是平顶（某个最深层的函数执行了太长时间），或者执行时间过长的函数；也可优化一下层数过深的部分（包括过多的函数调用）。
>
> 火焰图是定期采样当前 cpu 在执行的函数，因此体现的是函数占用 cpu 的比例（最耗 cpu 的函数）而不一定是最耗时的函数（比如在等待 io）。

> 粒度应该更细一些，精确到代码块而非函数，但不好实现，所以应该划分好函数。



**使用**

1. 采集 cpu 信息：`perf record -e cpu-clock -g -a -- sleep 15`（可以不加 -a，最后指定要测试的程序，或用 -p 指定 pid）（如果不指定 sleep 就 ctrl c 结束）。
2. 用 perf script 对 perf.data 解析：`perf script -i perf.data &> perf.unfold`。
3. 获取 FlameGraph 库：`git clone https://github.com/brendangregg/FlameGraph.git`。
4. 将 perf.unfold 中的符号进行折叠：`./stackcollapse-perf.pl perf.unfold &> perf.folded`。
5. 生成 svg：`./flamegraph.pl perf.folded > perf.svg`，用浏览器打开。

y 轴表示调用栈，每一层都是一个函数。调用栈越深，火焰就越高，顶部就是正在执行的函数，下方都是它的父函数。
x 轴表示抽样数，如果一个函数在 x 轴占据的宽度越宽，就表示它被抽到的次数多，即执行的时间长。
注意，x 轴不代表时间，而是所有的调用栈合并后，按字母顺序排列的。

火焰图就是看顶层的哪个函数占据的宽度最大。只要有"平顶"（plateaus），就表示该函数可能存在性能问题。

**性能优化 - PGO**

> https://zhuanlan.zhihu.com/p/652814504

Profile-Guided Optimization (PGO) / feedback-directed optimization (FDO) 是指利用程序运行过程中采集到的 profile 数据，来重新编译程序以达到优化效果的 post-link 优化技术。它是一种通用技术，不局限于某种语言。
它首先要对程序进行剖分(Profile)，收集程序实际运行的数据生成 profiling 文件，根据此文件来进行性能优化：通过缩小代码大小，减少错误分支预测，重新组织代码布局减少指令缓存问题等方法。PGO 向编译器提供最常执行代码区域，编译器知道这些区域后可以对这些区域进行针对性和具体的优化。

### 编译器优化

**fopt-info**

[gcc](https://gcc.gnu.org/onlinedocs/gcc/Developer-Options.html#index-fopt-info) 可通过`-fopt-info`查看编译器做了哪些优化，以及未做某些优化的原因。
与`-fopt-info-optimized-optall`等价。

通过`-fopt-info-options`选择和筛选要看哪一类的优化信息。
options 可由三部分组成：优化类别（ipa、loop、inline、omp、vec、optall）；信息类别（optimized、missed、note、all）；输出量（包括 默认、internals。后者输出更详细、并不常用的信息）。
比如：`-fopt-info-vec-missed`、`-fopt-info-inline-optimized-missed=inline.txt`（输出到指定文件）。

**Rpass**

[clang](https://clang.llvm.org/docs/UsersManual.html#options-to-emit-optimization-reports) 也提供了输出编译器优化信息的方式：

- -Rpass：在哪做了优化。
- -Rpass-missed：在哪优化失败。
- -Rpass-analysis：在哪决定不做或做一个优化。
- -Rpass=xx：查看某种类型的优化。包括：inline、loop-vectorize 等。

gcc 包含一系列内存检查工具 sanitizers，可以在程序运行时发现非常多的问题（如检测 data race（记录每次内存访问）、UB）。

sanitizer 可分为三类：

- [AddressSanitizer](https://clang.llvm.org/docs/AddressSanitizer.html)：检查内存泄露、越界读写、使用未初始化内存、重复释放等。
- [MemorySanitizer](https://clang.llvm.org/docs/MemorySanitizer.html)：检查使用未初始化内存、使用已释放内存、使用重复释放的内存等。
- ThreadSanitizer。

> ThreadSanitizer (TSan) 是动态检查，所以只有实际触发了的 data race 才能被 TSan 检测，存在但是没有触发的无法检测。
>
> valgrind 需要插桩，实际运行起来是单线程的，会慢很多，所以只适合用来测试；tsan 对性能影响不是很大，可以放在正常的 CI 里。

> https://stackoverflow.com/questions/48994494/how-to-properly-use-prefetch-instructions

```cpp
void __builtin_prefetch (const void *addr, ...);
```

指针 addr 指向要预取的数据（我们判定这些数据是很快能访问到的，或它们就在最近的内存中）。

可选参数（两个必须都是编译时常数）：

- rw：1 表示写(w)，0 表示读(r)。默认为 0。
- locality：指明数据的时间局部性 (temporal locality)。该值在 0 - 3 之间，值越高代表数据的时间局部性越强。为 0 时表示它没有时间局部性，即要访问的数据被访问后的不长时间里不会再被访问；为 3 时表示被访问的数据具有高时间局部性，即被访问后很有可能再次访问。默认为 3。
  时间局部性：某一条指令执行后，在不久后该指令会被再次执行；某数据被访问后，在不久后该数据会被再次访问。

示例：

```cpp
for (int i = 0; i < n; i++) {
    a[i] = a[i] + b[i];
    __builtin_prefetch (&a[i+j], 1, 1);
    __builtin_prefetch (&b[i+j], 0, 1);
}
```

> 预取一般是在循环 i 时 prefetch i + k 的数据？
> 对连续的内存，根据 cacheline / 元素大小 决定预取的位置即 k？如果 k > 1 最好还要循环展开 k 次以便每次循环 prefetch 一次不同的 cacheline？
> 对不同的内存，k 取一个较小值？
>
> 现在硬件也有 prefetcher，手动的 prefetch 未必更好。

**_mm_prefetch**

> https://stackoverflow.com/questions/46521694/what-are-mm-prefetch-locality-hints

```cpp
void _mm_prefetch(char *p, int _hint);
```

Intel SSE (Streaming SIMD Extensions) 指令集中提供的函数。

第二个参数 _mm_hint 是预取方式，包括：

- _MM_HINT_T0：将数据取到所有缓存。适用于数据会被频繁访问的情况，尤其是短时间内。
- _MM_HINT_T1：将数据取到 L2 和 L3。适用于数据会被访问，但不是立即需要或访问频率较低的情况。
- _MM_HINT_T2：将数据取到 L3。适用于数据会被访问，但较长时间后才访问或访问频率较低的情况。
- _MM_HINT_NTA：读到非临时缓存中，防止 cache pollution。适用于数据只会被访问一次，或数据量非常大、不适合保留在缓存中。

> hint 只是 hint，不一定严格遵守。
>
> gcc 内可能会将`_mm_hint`定义为直接使用`__builtin_prefetch`的宏，因为前者必须要平台支持 SSE，所以这样可以实现不同平台的兼容。

**||, &&**

短路运算`||`和`&&`存在分支，无法应用 SIMD，对某些不需要短路的情况，用`|`和`&`可能更快：

```cpp
bool find(const int x, int* a) {
	return x == a[0] || x == a[1] || x == a[2] || x == a[3]
		|| x == a[5] || x == a[6] || x == a[7] || x == a[4];
}
bool find2(const int x, int* a) {
	return (x == a[0]) | (x == a[1]) | (x == a[2]) | (x == a[3])
		| (x == a[5]) | (x == a[6]) | (x == a[7]) | (x == a[4]);
}
```
