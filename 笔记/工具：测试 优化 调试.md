**CPU usage/utilization 与 CPU load**

[CPU 负载](https://en.wikipedia.org/wiki/Load_(computing))：使用或等待使用 CPU 的进程数（或线程数？）（或 CPU 运行和等待队列的长度）。
进程有多种状态，负载统计的是处于 runnable 和 running 状态的进程。

当负载始终小于1时，每个需要使用 CPU 的进程都可以立即使用它，而无需等待；大于等于1时，如果有进程需要使用 CPU，会由于 CPU 不可用而等待。（如果有 n 个核，则是判断负载是否小于 n）
负载可作为 CPU 瓶颈的参考指标。当平均负载大于核数时，说明 CPU 可能不够用。

> 虽然多数系统的负载统计的是 runnable 和 running 进程数，但 Linux 也会统计处于 uninterruptible sleep 状态的进程（通常是等待硬盘活动的进程）。因此当许多进程因为 IO 阻塞时（比如硬件较慢或使用的 NFS 出现问题），此时的负载不能真正表示 CPU 使用的情况，即使负载较大，CPU 可能也是够用的（但还是能表明进程大概要等多久）。
> uninterruptible sleep 通常是很短暂的 sleep，因此 linux 将其近似认为是 runnable，但当 IO 出问题时这个睡眠就会很长了。

CPU 利用率：CPU 处理任务（非空闲状态）占总时间的百分比，即 使用中的时间/(使用时间 + 空闲时间)。

> CPU 某一时刻的状态可分为多种：
> us (user space)：运行非内核代码花费的时间。
> sy (system, kernel space)：运行内核代码花费的时间。
> id (idle)：空闲时间。
> wa：等待 I/O 花费的时间。
> st：从虚拟机中窃取时间。
>
> 可以简单认为 cpu usage = (us + sy) / (us + sy + id)。

两个值都是定期采样得到的。

**两者的关系：**

- CPU 负载是比利用率更好的指标。
    比如机器 A, B 的利用率都是 100%，但 A 的负载是3，B 的负载是 6，两者的 CPU 需求肯定是不同的。
- 通常负载高就代表利用率高。
    但在 Linux 上，如果是 IO 密集型任务导致的高负载，则利用率不一定高。这种情况下，即使 CPU 利用率低，但负载较高，CPU 也可能是瓶颈？因为进程潜在的要与大量进程竞争？
- 利用率高不代表负载高。即使利用率是 100%，只要负载为1（对单核），就不会影响执行。这也是理想状态。
- 吞吐量 (throughout，单位时间内处理任务的数量) 是优化的根本目标。

总之需要根据实际情况分析：当任务没有高吞吐量要求时，运行它的 CPU 利用率或负载低也不是坏事，因为可以腾出 CPU 供其它进程运行；当利用率高时，只要负载不高，也是 ok 的。

**测量 CPU 状态**

top、uptime 可以显示系统的平均负载，其结果有三个值，分别表示以 1、5 和 15 分钟为间隔的平均负载。
在多核机器中，负载可能统计的是所有核上的负载之和，也可能是负载/核数。核数量可通过`cat /proc/cpuinfo | grep core`查看。

vmstat 可显示 CPU 活动。
/proc/stat 文件中也可提取 CPU 活动。它包含自启动以来有关系统的各种指标。

### 其它

**性能优化 - PGO**

> https://zhuanlan.zhihu.com/p/652814504

Profile-Guided Optimization (PGO) / feedback-directed optimization (FDO) 是指利用程序运行过程中采集到的 profile 数据，来重新编译程序以达到优化效果的 post-link 优化技术。它是一种通用技术，不局限于某种语言。
它首先要对程序进行剖分(Profile)，收集程序实际运行的数据生成 profiling 文件，根据此文件来进行性能优化：通过缩小代码大小，减少错误分支预测，重新组织代码布局减少指令缓存问题等方法。PGO 向编译器提供最常执行代码区域，编译器知道这些区域后可以对这些区域进行针对性和具体的优化。

**火焰图**

Linux 下有 perf 命令，可以分析程序的性能、函数的耗时和调用关系。
通过`perf record`提供的信息，可以用应用生成程序执行的火焰图。

[火焰图](https://blog.csdn.net/m0_65931372/article/details/126539238)可以直观地展示程序性能的瓶颈在哪个函数上。
横轴是时间，反映函数的耗时（但不完全是？）；纵轴是调用栈，反映函数的调用关系。
越高的位置，函数调用栈越深；越宽的位置，函数执行时间越长。
通常要优化的点是平顶（某个最深层的函数执行了太长时间），或者执行时间过长的函数；也可优化一下层数过深的部分（包括过多的函数调用）。

粒度应该更细一些，精确到代码块而非函数，但不好实现，所以应该划分好函数。

### 编译器优化

**fopt-info**

[gcc](https://gcc.gnu.org/onlinedocs/gcc/Developer-Options.html#index-fopt-info) 可通过`-fopt-info`查看编译器做了哪些优化，以及未做某些优化的原因。
与`-fopt-info-optimized-optall`等价。

通过`-fopt-info-options`选择和筛选要看哪一类的优化信息。
options 可由三部分组成：优化类别（ipa、loop、inline、omp、vec、optall）；信息类别（optimized、missed、note、all）；输出量（包括 默认、internals。后者输出更详细、并不常用的信息）。
比如：`-fopt-info-vec-missed`、`-fopt-info-inline-optimized-missed=inline.txt`（输出到指定文件）。

**Rpass**

[clang](https://clang.llvm.org/docs/UsersManual.html#options-to-emit-optimization-reports) 也提供了输出编译器优化信息的方式：

- -Rpass：在哪做了优化。
- -Rpass-missed：在哪优化失败。
- -Rpass-analysis：在哪决定不做或做一个优化。
- -Rpass=xx：查看某种类型的优化。包括：inline、loop-vectorize 等。

**sanitizer**

>https://github.com/google/sanitizers
>
>`-fsanitize=address -lasan`
>`-fsanitize=undefined`

gcc 包含一系列内存检查工具 sanitizers，可以在程序运行时发现非常多的问题（如检测 data race（记录每次内存访问）、UB）。
例：https://blog.csdn.net/Al_xin/article/details/108218727

sanitizer 可分为三类：

- [AddressSanitizer](https://clang.llvm.org/docs/AddressSanitizer.html)：检查内存泄露、越界读写、使用未初始化内存、重复释放等。
- [MemorySanitizer](https://clang.llvm.org/docs/MemorySanitizer.html)：检查使用未初始化内存、使用已释放内存、使用重复释放的内存等。
- ThreadSanitizer。

> ThreadSanitizer (TSan) 是动态检查，所以只有实际触发了的 data race 才能被 TSan 检测，存在但是没有触发的无法检测。
