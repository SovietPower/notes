## 编译原理

**CFG**

CFG 由终结符号、非终结符号和一组产生式组成。
终结符号为语言中会出现的各个符号。
非终结符号可看过表示特定字符串的变量或别名。非终结符号中包含一个开始符号。
产生式的格式为`左部 -> 右部`，左部为非终结符，右部为若干个终结符或非终结符。
可由上下文无关文法生成的语言 称为 上下文无关语言。

**CFG 与 CSG**

即上下文无关文法（context-free grammar，2型文法）与上下文有关文法（context-sensitive grammar，1型文法）

https://www.zhihu.com/question/21833944/answer/307309365

主要在于，CSG 的产生式左边可以加终结符，以限制产生式的选择？

CSG 允许产生式左边有终结符，但不是必须有，所以所有 CFG 文法都属于 CSG 文法。所以1型文法比2型文法识别的语言集合更大。还存在一个0型文法，描述能力更强。

3型文法更弱，例子：正则表达式。

> 3型文法：正则文法/正规文法。正则文法描述的语言构成了正则语言类，正则语言类中的语言也可以由有限状态自动机或正则表达式来表达

> 正则文法表达不了递归，上下文无关文法就可以。编程语言语法一般都有递归结构，所以不能用DFA。
>
> C/C++能用上下文无关法描述。但是*一旦*使用上下文无关法，很多原属于语法层面的特性被迫*推迟*到语法分析之后（语义分析阶段）实现。
>
> 例如 a * b，上下文无关法分析器只能将它标记为“带歧义的星号语句”，交给之后的程序处理。这会导致语法规则混乱，以及使语义分析器承担语法分析器的部分职责，所以少有项目用这样的方法实现。



**编译器模型**

> gcc 架构：https://zhuanlan.zhihu.com/p/372526494
> 部分见 *C++ - 基础 - gcc 与 g++*。

前端：语法分析、静态检查（如类型检查、语法检查（如break在for/while/switch内））、中间代码生成。即对源程序进行分析，产生中间表示。
后端：根据中间表示，生成目标代码（生成汇编代码、二进制码）。
在前后端之间可以称为中端，编译器还要进行大量优化。

通过一个适当定义的中间表示形式，可以把针对源语言 i 的前端和针对目标机器 j 的后端组合，得到源语言 i 在机器 j 上的一个编译器。

一个编译器可能构造一系列中间表示：源程序->高层中间表示形式->...->低层中间表示形式->目标代码。

可作为中间语言(intermediate language)的：语法树（一个高层的表示），DAG（合并了语法树中的公共子表达式），三地址代码（通过选择不同的运算符，可以是高层的，也可以是低层的），后缀表达式，java，C语言（用C编译器作为后端。C是比较底层的语言）。

各种中间表示（如AST）和 语法制导翻译SDT 都可以解释程序，或者生成其它的中间表示。
生成中间表示的过程都是语义分析。
语义分析其实是非常广的概念，编译器只能做静态的语义分析，作用非常有限，常见的功能就只是类型检查。运行时也会进行语义分析。

**三地址代码(three-address code)**

形如：`x = y op z`或`op x,y,z`，一条指令的右侧最多只有一个运算符。

不同描述方式：

- 四元式(quadruples)：用`op arg1 arg2 result`4个字段表示。
- 三元式(triple)：只用`op arg1 arg2`3个字段表示。每条指令有一个编号，通过使用`(编号)`作为参数，表示某指令的计算，而不保存临时结果`result`。如`t1=-c; t2=a*t1; t3=t2+b;`表示为`0:minus,c,_ ; 1:*,a,(0); 2:+,(1),b`。
    对于表达式，DAG和三元式等价，而DAG表示方便优化。
    但是，当指令位置/顺序改变时，所有引用改变指令的指令参数都要修改。
- 间接三元式(indirect triple)：实际执行的指令序列，是一个指向指令的编号序列（如 012034），而不是直接用指令序列本身表示。改变执行顺序时，改变编号序列，而不会改变指令编号，所以不需要修改参数。
    ![image-20230113203913416](https://cdn.jsdelivr.net/gh/Apricity-qvq/ImageHostForTypora@main/img/image-20230113203913416.png)
- 静态单赋值(SSA)：左式中（被赋值变量）每个变量名最多只出现一次。
    如果在不同控制流对同一变量$x$赋值，由于SSA会在不同位置对$x$使用不同的变量名$x_1,x_2,...$赋值，所以需要一个$\varphi$函数决定最终使用哪个$x$的赋值$x=\varphi(x_1,x_2,...)$。如`if`为真时$\varphi(x_1,x_2)$返回$x_1$，否则返回$x_2$。

**Java的编译**

- 动态编译（dynamic compilation）指“在运行时进行编译”；与之相对的是事前编译（ahead-of-time compilation, AOT），也叫静态编译（static compilation）（比如C++在运行前就编译）。
- JIT编译（just-in-time compilation）狭义来说是当某段代码即将第一次被执行时进行编译，也叫“即时编译”。JIT编译是动态编译的一种特例，能提供优化空间，提供效率，但难实现。
    JIT编译一词后来被泛化，与动态编译等价。
- 自适应动态编译（adaptive dynamic compilation）也是一种动态编译，但它通常执行的时机比JIT编译迟，先让程序“以某种式”先运行起来，收集一些信息之后再做动态编译。这样的编译可以更加优化。

Javac把Java源码编译生成Java字节码（class文件），JVM把class文件加载、链接、执行得到结果。由JVM解释执行字节码文件，可以做到平台无关。
Java程序一般通过解释器(Interpreter)(JVM)解释java字节码执行：直接执行字节码，可以不将字节码翻译成机器码。
一些代码只会执行一两次，直接生成字节码然后解释，比编译再执行快很多。
当虚拟机发现某个方法或代码块运行的特别频繁时，会把这些代码认定为“热点代码”(Hot Spot Code)。为了提高热点代码的执行效率，在运行时，虚拟机会把这些热点代码编译成与本地代码相关的机器码，优化然后缓存下来。这个过程是即时编译，完成这个任务的编译器称为即时编译器 (Just In Time Compiler)。
即时编译器并不是虚拟机必须的部分，但确实影响性能。

上面的编译分类只是一种方式，并不一定绑定某种语言，语言可以是 AOT 也可能用 JIT（而且分界不是很明显？）。

**解释性语言和编译型语言**

编译型语言在编译时直接生成机器代码，解释性语言编译时生成字节码，在运行时解释字节码。
编译过程，可以生成机器码，执行时直接执行；也可以生成中间表示（如字节码），执行时解释，或再编译成机器码执行；也可生成更高层表示（如抽象语法分析树AST），再解释中间表示；还可以不进行编译，直接解释源码。
区别在于，编译和执行过程做的工作量不同。编译一般至少生成一个中间表示，提高执行的速度，重复运行时不用重复编译；使用中间表示，可以方便地跨平台。这是编译时间和执行效率的权衡。
编译时的优化也类似，是编译时间和更高效的机器码之间的权衡。

以主流的分类来看，编译时直接生成机器码（或者较底层的中间表示）的叫编译型语言，否则是解释型语言。但很多语言其实两者都可，只是主流实现不同。
如果执行时是字节码，则可以叫字节码解释器？执行时是AST，则叫AST解释器。

**为什么不直接解释AST，而是转成字节码**

AST是语法树的精简和规约版？去掉了括号、分号之类的token。

> > https://pages.cs.wisc.edu/~fischer/cs536.s06/course.hold/html/NOTES/4.SYNTAX-DIRECTED-TRANSLATION.html
>
> First, let's consider how an abstract-syntax tree (AST) differs from a parse tree. An AST can be thought of as a condensed form of the parse tree:
>
> - Operators appear at internal nodes instead of at leaves.
> - "Chains" of single productions are collapsed.
> - Lists are "flattened".
> - Syntactic details (e.g., parentheses, commas, semi-colons) are omitted.
>
> In general, the AST is a better structure for later stages of the compiler because it omits details having to do with the source language, and just contains information about the essential structure of the program.
> The parentheses are not needed in the AST because the structure of the AST defines how the subexpressions are grouped.
> ![image-20230116192945525](https://cdn.jsdelivr.net/gh/Apricity-qvq/ImageHostForTypora@main/img/image-20230116192945525.png)

AST的缺点：

- AST还是有些没有执行语义的节点（如语句块的大括号），但解释时还是要访问它们。
- AST通常会有一些与解释执行无关的额外信息，占用内存。
- 通常使用链式结构，遍历子节点时有很多间接引用。
- 一般通过递归实现，需要很多参数传递和过程调用。

上述问题可以缓解，但字节码不仅没有这些问题，还有其它优势：

- 字节码指令可以紧凑、准确的描述其执行语义。
- 可以大大减少没有实际执行语义的指令数。
- 很容易通过循环而非递归实现。

这也是常见的原则：进一步划分任务，便于优化和实现。
如果不追求效率，想实现简单，AST解释器也可以。

## 编译器

传统的编译器通常分为三个部分：前端 (frontEnd)，优化器 (Optimizer) 和后端 (backEnd)。
在编译过程中，前端主要负责词法和语法分析，将源代码转化为抽象语法树；优化器主要负责对中间表示（IR，比如语法树）的优化，使代码更加高效；后端则是将已经优化的中间代码转化为针对不同平台的机器代码，并针对机器架构、指令集进一步优化。



**GCC 背景**

GCC (GNU Compiler Collection) 是 GNU 编译器集合项目，包含许多编译器程序，支持多种语言。比如：gcc、g++。
GCC 最早是 GNU C Compiler，专用于 C 的编译器，但后来集成了不同编译器，gcc 本身也不再是编译器，变成了 driver。



**LLVM 背景**

LLVM 提供了与编译器相关的支持，能够进行程序语言的编译期优化、链接优化、在线编译优化、代码生成，可以作为多种编译器的优化器和后端来使用（支持多种前端）。

最初苹果使用 gcc 作为编译器，但由于 1. 苹果最初主要使用 Objective-C 并需要许多新特性，但 gcc 不给实现；2. 苹果希望有更好的功能，在 Chris Lattner 加入苹果后创建了 LLVM。
之后苹果创建了 LLVM 的前端 Clang，以便完全替代 gcc。

LLVM 项目主要包括（具体见 llvm.org）：

- LLVM 核心库，提供了源码独立和目标独立的优化器。
- Clang 是 C/C++/Objective-C 编译器，提供了更快的编译速度、可读性更强更有用的警告报错和强大的静态检查，且模块化。
- LLDB 是一个强大的调试器，比 GDB 更快更省内存。
- libc++ 提供高效的标准库实现。

> LLVM 是项目名称，并非 Low Level Virtual Machine 的缩写，与虚拟机没关系。
>

> LLVM 统一了 LLVM IR（中端），如果需要开发新的处理器架构后端，那么只需要开发后端即可。除了后端模块化，连前端也实现了模块化，开发者可以自定义前端规则，开发一门自己的编程语言。该编译器框架被Apple、FaceBook、Google广泛应用，甚至Apple为该编译器框架开发了Clang。

**GCC 与 LLVM 的区别**

- Clang 使用 BSD 许可证，GCC 使用 GPL 许可证。
- GCC 使用 ld 作为链接器，Clang 使用 lld。
- GCC 使用 gdb 作为调试器，Clang 使用 lldb。



**clang**

Clang 是基于 LLVM 开发的、主要支持 C/C++、Objective-C 语言的编译器前端，具有快速的编译速度和低内存占用。

## 编译优化

**向量化** / **SIMD**

向量化 (vectorization) 指利用 CPU 里的 vector registers，它们可以同时存储同种类型的多个变量，并通过相关指令一次性操作所有数据。

向量化通常优化循环，有一定条件：

- 类型简单：基础或较小的平凡类型？
- 是线性遍历数组。增量最好为1，以便内存读取（memory wall：内存速度与 CPU 运算速度相差越来越大）。
- 可确定循环边界。
- 循环之间没有依赖关系。某次循环的运算不会依赖其它循环运算的结果。
- 没有别名。如：`int *p = A - 1;`，无法确定指针指向，可能存在依赖关系。
    restrict 可以声明非别名。

SIMD 可以用批量比较来处理循环中的条件判断。

**loop invariant computation**

将循环内的相同求值表达式移到循环外部，减少计算。
要保证表达式结果不会在循环中被更改。

**loop unswitching**

将循环内的条件判断移到循环外部，减少判断。
要保证条件语句的结果不会在循环中被更改。

**loop interchange**

交换多重循环的顺序，来优化对多维数组的访问、更好的利用 cache。

LLVM 可通过`-enable-loopinterchange`开启。

**loop distribution**

将单个循环的过长循环体分割成多个循环，使编译器更容易进行向量化，使数据更可能被放到寄存器和缓存中。

LLVM 可通过`-enable-loop-distribute`开启。
Intel 有一个指令`#pragma distribute_point`用来指定切割位置。

**loop fusion**

合并两个循环。

**常量传播 (Constant Propagation)**

> http://compileroptimizations.com/

如果某个地方使用的某个变量的值在编译期可知，直接使用该值来替代它。
（这里的常量不是指 C++ 的常量，只是进行优化的概念）

```cpp
x = 1;
f(x); // 可替换为 f(1)。如果不再使用 x，也不需要为 x 分配空间
```

**常量折叠 (Constant Folding)**

如果某个表达式包含多个编译期已知值的操作数，即它的结果在编译期可知，直接使用该结果来替代它。

```cpp
f(1 + 2 + 3); // 可替换为 f(6)

x = 1;
f(x + 2); // 可配合常量传播

s = "a" + "b"; // 可替换为 s = "ab"
```

**复写传播**

用某个变量代替某个表达式中出现的其它变量，可以简化代码或减少变量读取。

```cpp
x = 1;
y = x + 2;
z = y + 3;
f(z); // 可替换为 f(x + 5) 或 f(6)
```

**公共子表达式消除 (CSE Elimination)**

如果两个表达式完全相同，那么可以只计算第一个表达式的值，避免重复计算第二个表达式。
如果涉及函数调用、系统调用或两个表达式之间有副作用，就不能随意消除了。用 *属性说明符 - pure* 可以标记一个函数为纯函数、没有副作用，以便消除。

**死代码消除 (Dead Code Elimination)**

删除没有意义或不会被执行/不可达的代码。

**尾递归优化 (tail call optimization)**

*尾调用 (tail call)* 指一个函数的最后一步是返回一个函数的调用结果。
*尾递归 (tail recursion)* 指函数在尾调用时调用的函数是自身。

由于尾调用是函数的最后一步，因此函数原有的栈帧不需要保留，只需要一层栈帧，这种优化称为尾调用消除。此时的递归效率约等于循环实现。
常规情况每次递归需要 push 一个新栈帧，可能导致函数栈帧过多、占用大量内存、效率降低（尤其是尾递归的情况，此时就叫尾递归优化）。

注意`return f(x)`是尾递归，但`return f(x) + 1`不是，这种情况下需要将函数改写成`g(x) {... return g(x)}`的形式以便优化。

> 对于：`return f(x)`在分支中的情况；`return xx||f(x)`即尾调用在逻辑表达式或三目运算符中的情况；编译器也能进行优化（需要 O2）。
> 尾调用优化会让调用栈或错误信息更不直观。
>
> 对于 C++，`return f(x)`可能不是尾递归：如果栈对象或参数的析构函数是非平凡的，还要调用对象的析构函数（比如 string_view 可以，string 不行）。此时只有 RVO 优化。如果想要效率还是要循环实现替代函数调用，不要依赖该优化！

## 静态分析

静态分析指分析源代码以自动发现 bug 的一系列算法和技术。它和编译器警告类似，但是可以借助运行时的调试技术（比如测试）来发现更多 bug；还可以尝试推理代码的语义更深入的发现。

缺点：

- 静态分析可能要比编译慢非常多。某些 bug 的检测可能要指数级算法（不过也会限制在合理时间内）。
- 可能会误判。
- 只能发现分析器算法能找到的那部分 bug。



> clang-tidy 是一个基于 clang 的 C++ linter，主要是发现和修复一些典型的编程错误，比如：代码质量差可能存在隐藏错误，违反编程风格，接口误用，部分可通过静态分析发现的错误。
