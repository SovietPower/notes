**cookie**

> https://zhuanlan.zhihu.com/p/596978442
> https://zhuanlan.zhihu.com/p/164696755（有点乱）

HTTP 是无状态的协议（对于事务处理没有记忆能力，每次会话完成时，服务端不会保存任何会话信息）：每个请求都是完全独立的，服务端无法确认当前访问者的身份信息，无法分辨上一次的请求发送者和这一次的发送者是不是同一个人。所以服务器与浏览器为了进行会话跟踪（知道是谁在访问我），就必须主动的去维护一个状态，这个状态用于告知服务端前后两个请求是否来自同一浏览器。而这个状态需要通过 cookie 或者 session 去实现。

cookie 是服务器发送到用户浏览器（客户端）并保存在客户端的一小块数据，它会在浏览器下次向同一服务器再发起请求时被携带。既能本地存储一定数据，也能身份认证。
cookie 是不可跨域的，每个 cookie 都会绑定单一的域名，无法在别的域名下获取使用。一级域名和二级域名之间可以共享使用（只要domain匹配）。
cookie 的主要属性：键值对，domain（该cookie所属域名），path（限制在哪个路由下生效，和domain一起构成url，默认为`/`，如为`/abc`则只有`/abc/xxx`可访问该cookie），maxAge（失效时间，一个倒计时，负值则为临时cookie，关闭浏览器即会话结束就失效，默认-1），expires（过期时间），secure（是否仅能使用安全协议传输，如HTTPS, SSL，默认false），httpOnly（是否能通过JS脚本读取，由服务端设置）。

当客户端发起请求时，浏览器先检查该域名是否有cookie，如果有则自动添加到`request header`中的cookie字段。所以每次请求cookie都会被发送。适合的数据一般只有身份认证信息。

缺点：

1. 每个特定域名下的cookie数量有限，比如50个；不能跨域。
2. 存储量太小，只有4KB，且只能存储字符串。
3. 每次HTTP请求都会发送到服务端，影响获取资源的效率。
4. 存储在本地，不太安全，不能存敏感数据。

**浏览器本地存储**

js提供了两种存储方式：sessionStorage 和 globalStorage。在H5中，用 localStorage 取代了 globalStorage。
sessionStorage 用于本地存储一个会话中的数据，这些数据只有在同一个会话中的页面才能访问。当会话结束后，数据就随之销毁。所以 sessionStorage 仅是会话级别的存储，而不是持久化的本地存储。
localStorage 是持久化的本地存储，除非通过js删除，或清除浏览器缓存，否则数据永远不会过期。
本地存储默认不会发送，要手动加需要的信息添加到header。

**session**

cookie太长了，而且每次都要携带，影响效率。
session将需要的信息（如登录的用户是谁）保存到服务端：服务端为用户生成 session，为该 session 分配唯一的 sessionID，将 sessionID 返回给客户端作为 cookie。
之后客户端的请求都要在 cookie 中带 sessionID，服务端通过 ID 找到对应用户和认证信息，然后执行其它操作。

当然 session 不一定要通过 cookie 实现，还可以放在 url 参数中等。

优点：

1. 存储在服务端，安全。
2. 可存储任意大小、任意数据类型。

缺点：

1. 存储数据会占用服务器资源。因为通常频繁使用，所以要保存在内存。
2. 一般服务端会有多台服务器工作，但 session 只登录时存在了一台服务器上，需要同步。

缺点2的解决方式：

1. session 复制：服务器生成 session 后同步给其它机器。但同样 session 要存多份，造成数据冗余；当节点很多时，复制会造成很大开销。
2. session 粘连：让每个客户端请求只发送给固定的一台机器。可以哈希 ip 来选择机器。但相当于没有分布式，这个机器挂了对应客户端就无法访问。
3. session 共享：最常用的方案，将 session 保存在 redis，memcached 等中间件中。请求到来时，去这些中间件读取 session。但每次请求都要读取中间件，有一些性能损失；同时要保证中间件集群的高可用性（但对于大公司，因为有其它需求所以本就要有集群）。

**token**

对于普通业务，session 共享要求的 redis 集群不太值得。
token 只需要保存在客户端。
用户登录时，服务端根据用户信息生成 token，返回给客户端。客户端之后发送请求时，都带着该 token。
一般的 token 会保存在数据库中，以验证请求的 token 的合法性。
JWT 这种 token 则不需要将 token 保存。服务端在收到 token 时，会根据密钥和相应算法，对 header 和 payload 生成签名，如果该值与 token 中的一致，则 token 正确。

>  JWT (JSON web token) 规定了一种 token，包含三部分：header, payload, signature，用`.`分割。传输时会用 base64 编码。
>  header 包含加密算法、token类型，payload 包含需要的键值对，signature 为签名，保证token不被修改。
>  例：`{"alg":"HS256", "typ":"JWT"}{"name":"abc", "ID":"123", "exp":"60"}{avadqvqbq}`。

优点：

1. 只要服务端的密钥不泄露，token 就是安全的（不会被篡改）。
2. 数据不需要保存在服务端。

缺点：

1. 数据没有加密，所以不适合存隐私数据。
2. token 一般要比 sessionID 长很多，每次请求都带会有开销，并且服务端需要解析。
3. 不太安全，一般放在浏览器的 local storage 中，JS可直接读取（也可在 cookie，但可能放不下）。
4. token 一旦由服务端生成，它在过期前就是有效的，无法主动让 token 失效。除非在 server 为 token 设立一个黑名单，但这又要在服务端保存数据，和 session 区别不大。
   如果要让 token 失效，一般是直接在本地删除 token，下次再生成。
   所以 token 更适合一次性的命令认证，设置一个比较短的有效期。

多数情况下还是用 session，一次性的命令认证可以用 token。

> 项目中使用jwt的方式：
> 后端：
> 在用户登录时，除用户信息外，还返回一个token，通过github上的jwt库加密基本的用户信息生成，包含信息本身、过期时间和密文。
> 对于需要登录信息的网页，通过`auth := v1.Group("/")`、`auth.Use(middleware.JWT())`创建路由组，然后路由。通过JWT中间件检验请求头部的Authorization内容：如果token解析正确且没超时，则继续（`c.Next()`），否则`c.Abort()`。

**token 与 cookie 的区别**

1. token 一般放在 header 的 Authorization 自定义头里，而不是 cookie，所以可以跨域。
   如 cookie 不能处理多应用（多系统）的单点登录(SSO)，用户只需要登录一次就可以访问所有相互信任的应用系统。
2. 移动平台可能会禁用 cookie，也就不能用 session，但对 token 没有影响。
3. 使用 cookie 有被 CSRF 攻击的风险（见[这里最后](https://zhuanlan.zhihu.com/p/596978442)）， token 不会。但 token也不是很安全。

**token 与 session的区别**

- session：在服务端记录会话信息，使服务端有状态化。
  token：用户的状态不会存储在服务端的内存中，所以是无状态的。

**为什么要前后端分离**

减少耦合，将不需要关心的细节抽象出来，可以提高工作效率、减少bug产生。我们只需要遵守接口的定义即可。
另一个也是前后端差别太大了？面向的人不一样，展示的东西也不一样。后端可以直接展示给程序员错误日志等专业但具体的信息，前端要展示给用户的，则必须是美观简洁的信息。

**时间轮算法**

时间轮是一个循环数组，每个位置指向一个任务链表，表示要在对应的时间段开始执行的任务列表。
时间轮与一个钟表类似，有一个指针指向当前时间对应的位置。假设时间精度为 1s，那指针就相当于一个每秒动一次的秒针，每次移动，就读取该时间需要执行的任务并执行。

**系统可用性**

从时间维度计算：可用性 = 正常运行时间 / (正常运行时间 + 故障时间)
从请求维度计算：可用性 = 成功请求数量 / 请求总数量

时间维度的可用性，一般用 X 个 9 来描述（见[这里](https://zhuanlan.zhihu.com/p/356668452)）：
3 个 9：可用性为 0.999，即一年中最多有 0.1% 的时间（约 500min）业务不可用。
4 个 9：可用性为 0.9999，即一年中最多有 0.01% 的时间（约 50min）业务不可用。
5 个 9：可用性为 0.99999，即一年中最多有 0.001% 的时间（约 5min）业务不可用。

1 和 2 个 9 称不上可靠。6 个 9 则要求太强，成本会很高。

## 分布式/高并发

分层将系统在横向维度上切分成几个部分，每个部分负责一部分相对简单并比较单一的职责，然后通过上层对下层的依赖和调度组成一个完整的系统。
在网站的分层架构中，常见的为3层，即应用层、服务层、数据层。应用层具体负责业务和视图的展示；服务层为应用层提供服务支持；数据库提供数据存储访问服务，如数据库、缓存、文件、搜索引擎等。
分隔是在纵向方面对软件进行切分。网站越大，功能越复杂，服务和数据处理的种类也越多，将这些不同的功能和服务分隔开来，包装成高内聚低耦合的模块单元，不仅有助于软件的开发维护也便于不同模块的分布式部署，提高网站的并发处理能力和功能扩展能力。
对于大型网站，分层和分隔的一个主要目的是为了切分后的模块便于分布式部署，即将不同模块部署在不同的服务器上，通过远程调用协同工作。分布式意味着可以使用更多的计算机完同样的工作，计算机越多，CPU、内存、存储资源就越多，能过处理的并发访问和数据量就越大，进而能够为更多的用户提供服务。

在网站应用中，常用的分布式方案有：
分布式应用和服务：将分层和分隔后的应用和服务模块分布式部署，可以改善网站性能和并发性、加快开发和发布速度、减少数据库连接资源消耗。
分布式静态资源：网站的静态资源如JS、CSS、Logo图片等资源对立分布式部署，并采用独立的域名，即人们常说的动静分离。静态资源分布式部署可以减轻应用服务器的负载压力；通过使用独立域名加快浏览器并发加载的速度。
分布式数据和存储：大型网站需要处理以P为单位的海量数据，单台计算机无法提供如此大的存储空间，这些数据库需要分布式存储。
分布式计算：目前网站普遍使用Hadoop和MapReduce分布式计算框架进行此类批处理计算，其特点是移动计算而不是移动数据，将计算程序分发到数据所在的位置以加速计算和分布式计算。

**QPS TPS**

QPS/TPS 是每秒能完成的请求/事务数。
并发量 是系统能同时处理的请求/事务数。
QPS = 并发量 / 平均响应时间。

一个系统的吞吐量由最大 QPS 和 最大并发数两者决定。任何一个达到峰值，吞吐量都没法再提升。
系统的最大 QPS 受物理资源限制，而并发数取决于用户量，主要靠前者衡量系统性能。

**集群**

对于用户访问集中的模块需要将独立部署的服务器集群化，即多台服务器部署相同的应用构成一个集群，通过负载均衡设备共同对外提供服务。
服务器集群能够为相同的服务提供更多的并发支持，因此当有更多的用户访问时，只需要向集群中加入新的机器即可；另外可以实现当其中的某台服务器发生故障时，可以通过负载均衡的失效转移机制将请求转移至集群中其他的服务器上，因此可以提高系统的可用性。

**计算向数据移动**

分布式系统的一个特点是 计算向数据移动，而非数据向计算移动。

以统计大文本中某个单词数为例：
数据向计算移动：将数据分批发给进行计算的服务器。因为数据量太大，效率低。
计算向数据移动：在拥有数据的多个服务器上 运行计算程序。每个服务器可以得到非常小的结果，可最后在一台服务器上完成。大大减少了数据的传输，并行也提高了计算性能。

**负载均衡算法有哪些**

将请求或任务分摊到多个节点执行。

1. 轮询：将请求轮流发到不同服务器上。平等的对待所有服务器，不管服务器的具体情况。
2. 随机：将请求随机发到一个服务器上。期望情况下和轮询一样。
3. 加权轮询：与轮询类似，轮流分，但更高权重的服务器可以分到更多请求。权重可以由服务器当前负载、抗压能力决定。加权随机同理，机器获得请求的概率与权重有关。
4. 源地址哈希：根据请求 IP 的哈希值，决定将请求交给哪个服务器。这样同一个客户端的请求总会由相同服务器处理。
   类似的，可以根据请求的其它信息的哈希值决定，比如 URL。
   实现简单，但这其实是单机处理请求，某个节点挂掉就会导致一部分服务不可用。
5. 最小连接数：动态选择积压连接数最少的服务器处理。
   类似的，可以选相应时间最短的服务器。

**一致性哈希**

> https://blog.csdn.net/a745233700/article/details/120814088

将资源分配到不同服务器上，可以减少单一服务器的访问压力。
决定一个资源 x 具体分配到哪个服务器上，最简单的方式就是 $hash(x)\%n$，其中 n 是服务器总数量。
但是使用这种方法，一旦服务器的数量发生改变（比如我加一台服务器），会导致大量资源需要被迁移到新的服务器上，因为$k\%n$的值和$k\%(n+1)$的值往往是不同的（也不一定不同，比如7%2, 7%3）。
为了减少资源的迁移，我们希望每次只有少量资源所属的服务器会发生改变。

一致性哈希中，首先设一个比较大的数，比如$2^{32}$。所有节点的 ID 取值在$0\sim 2^{32}-1$之间。
对于资源 x，先计算 $hash(x)\%2^{32}$的值，然后取第一个比这个值大的节点作为它所在的服务器。
整个模型就是一个环，节点随机分布在环的各个位置，一个资源计算哈希值得到位置后，顺时针走第一个遇到的节点就是它所在的服务器。
这样，当我们删除环上的第 n 个节点时，只需要把 n 的资源，也就是 n-1 到 n 这个区间上的资源，分配给 n+1 就可以了。
插入同理，只会涉及环上的一部分。

但这样会有数据倾斜的问题。如果环上的节点分布不均匀，大量的资源和请求会出现在少数节点上，这就和单机一样了。（算法的实际表现，取决于环上的节点分布是不是均匀）
而我们能用的节点并不算多，要在节点集合变化的情况下，总是维持分布均匀，是比较难的。

但如果环上的节点足够多，即使节点随机取值，也能保证节点在环上是均匀分布的。
所以我们可以让一个物理节点，去映射到环上的多个节点。这样一个节点会处理环上的几部分数据，但尽可能保证了资源分布均匀。

**数据倾斜**

在分布式系统中，数据一般会分布在不同节点上。如果大量节点集中在少数节点上，就是数据倾斜，会导致性能下降。
原因是负载均衡策略做的不好。

一般是通过数据的 key 的哈希值决定它放在哪个节点。如果 key 的分布不均匀（在 key 后加随机值），或哈希函数的选择差（多个哈希，不过会影响效率），就会导致这个问题。
也可能是分配标准的问题，比如按时间或地点决定数据分配。
某些 SQL 操作的结果会不均匀，比如`group by`。

**如何实现限流**

1. 最简单的就是一个计数器。比如每分钟最多接收 100 个请求，每次来一个请求时，如果计数器<100，就接收并+1；否则就忽略。然后每隔 1 分钟清零一下计数器。
   但是它不能把请求均匀地分到各时间点，还是可能在短时间内处理大量请求，之后一段时间没事干。
   比如有可能在第 0s 收到 100 个请求，后续的一分钟都没有请求；然后在第 60s 又收到 100 个请求，后续的一分钟又没有请求。这样的话，每分钟接收的请求数就不能设置得太高，效率不会很好。
   可以减少清零计数器的间隔？那就和漏桶类似了。
2. 滑动窗口，或者就是一个循环队列。队列大小有限，每次移动 l,r 指针来处理请求，根据 l,r 是否重合判断是不是还能接收请求。
   可以保证同一时刻，最多只会处理 n 个请求。
   但与计数器类似，不能把请求均匀地分到各时间点，可能在短时间收到大量请求。
3. 漏桶，去模拟一个漏水的桶。每次取出少量的请求执行。后续的请求会在桶里等待，过多的请求溢出的话就直接扔掉。
   可以均匀地处理请求，但不能处理突发的大量请求。
4. 令牌桶。如果桶没满，就定期放入 n 个令牌进去。对于每个请求，取出一个令牌处理，如果没有就丢弃，或者进缓冲区等待。
   不仅可以均匀地处理请求，也能处理突发的大量请求。

**Serverless**

软件开发里很重要的概念是抽象，抽象使我们不需要关注其它组件的实现细节，更专注于关键的任务上。API, os, GUI, 接口等都是抽象。
容器和 serverless 是程序运行环境之类的抽象。使用容器，开发者就不需要关心操作系统，更专注于业务，只要配置好容器就可以。serverless 是进一步抽象，连容器或服务器的配置都不需要关心。

此外，serverless 可以更充分、合理地利用服务器资源。
serverless 可以是按使用时间计费的，对于一些小规模应用，实际的服务器使用时间可能很短，内存需求也不大，如果也购买服务器就会浪费很多资源。部署到 serverless 则可以大大减少成本。原版每个人都要一台服务器，现在许多用户可以共享一台服务器，还不会影响体验。
使用 serverless 的用户也不需要担心稳定性和性能，这些都是云服务商负责的。

**分布式锁**

在单机、多线程的环境下，处理共享资源的读写冲突，是靠的进程的锁。
同样，在多节点的情况下，需要一个外部系统提供锁，来保证能正确访问共享资源。
常用的提供分布式锁的系统有 redis 和 zookeeper。

利用 MySQL 的唯一索引也能实现加锁（看 insert 是否成功），但涉及数据库会很慢；锁不会自己失效，也要自己续期。哪方面都不如 redis。

**redis 实现分布式锁**

redis 提供了 setnx (set if not exists)，通过检查`setnx lockName value`是否成功，可以决定是否加锁成功。
解锁时，直接使用`del lockName`删除该键值对。

这样的问题是：如果 A 获取了锁，然后因为某些原因没有释放锁（比如挂掉了，或者逻辑出问题了没处理锁），这个锁就永远不会释放。
为此，可以在获取锁时设置一个超时时间，这个时间后锁自动被释放（自动 delete）。
但要是先`setnx`获取锁，再`expire lock seconds`设置时间，如果机器在获取锁之后就挂掉，超时时间也不能成功设置。要保证加锁、设置超时时间是原子的，必须同时完成。
所以要用一条指令，在 setnx 时就同时指定超时时间：`SET lockName value NX EX seconds`。

但过期时间很难确定，太长浪费资源，太短不能保证机器用完资源，锁会被错误释放。
所以，需要一个线程定期轮询加锁的服务（每过 1/3 锁的周期检查一次），如果服务正常、还在使用资源，就设置新的过期时间。服务挂掉后，定时线程也会挂掉，锁之后就会自动过期。

> lua 也能做到类似的事。

> 理论上还有一个问题：如果 A 的锁过期了，被 B 获取，但 A 不知道；A 之后会进行释放锁，导致把 B 的、而不是自己的锁释放了。
> 为此可以将锁的 value 设为自己的标识符，在释放时检查 value 是不是自己设的。如果成功才删。
> 这个检查和删除也要是原子的，可以通过 lua 脚本实现，因为 redis 执行 lua 时会阻塞，保证它们一起完成。
>
> 但其实不该有这种情况，本质也是过期时间设置的不合适，应该确保 A 不用后才能释放锁（要么 A 主动放，要么 A 挂掉不会再用等超时），否则 A, B 都有共享资源的使用权。
> 当 A 发现锁被其它线程获取后，应该回滚自己的操作，返回失败。还需要人工验证数据是否正确、超出问题的原因。

使用 redis 时，常常使用 主从集群 + 哨兵 的架构。因为集群不是强一致的，在主节点上加锁（即写一个键值对），主节点崩溃后可能导致该锁丢失（没成功同步到其它节点），另一个线程就可以重复加锁、操作资源。

官方的解决方法是 redlock (红锁)，有些像 Raft，要在超过一半的主节点上成功加锁，才认为加了锁。
另一个方法是 fencing token，具体就简单看了下，没接触过具体场景也很难评判。

**redlock**

> http://redis.cn/topics/distlock.html
>
> redisson 中提供了`RedissonRedLock`，传入多个 redis 实例后，可以直接调用`tryLock()`与`unlock()`。
> 代码可参考[这里](https://www.cnblogs.com/huangwentian/p/14622441.html)。
>
> redisson 加锁时，如果不传入过期时间，默认每隔30s检查一次锁、进行续期（还是锁是30s、10s续期？）。通过[lockWatchdogTimeout](https://github.com/redisson/redisson/wiki/2.-配置方法#lockwatchdogtimeout)配置。

redlock 需要 n 个独立的 redis 实例进行加锁（实例自己可以加额外集群）。
客户端获取锁需要几步：

1. 获取当前时间，记为开始获取锁的时间。
2. 依次（或并发）尝试从这 n 个实例中，用相同的 key + 一个唯一的 value 获取锁（即 set + nx ex）。
   如果某个实例在超时时间内没有响应，跳过尝试下一个实例。超时时间要小于锁的过期时间。
3. 当且仅当在大多数 (>= n/2 + 1) 实例上成功加锁，且 当前时间 减去 开始获取锁的时间 小于锁的过期时间，客户端加锁成功。
   锁的实际有效时间，是 锁的过期时间 减去 获取锁所花的时间（无论如何，锁都是在 开始获取时间 + 过期时间 的时刻过期）。
4. 如果没有在多数实例上成功加锁，或 获取锁需要的时间 超过了 锁的过期时间，都是加锁失败，客户端需要在所有 n 个实例上进行解锁，然后等待一段随机时间后重试（避免多个客户端总是同时加锁。等待时间需要大于获取锁需要的时间，避免所有人都总是获取不到锁）。
   即使没有在某个实例上成功加锁，也要进行解锁，因为失败的请求可能会在后续执行成功。

释放锁很简单，在所有 n 个实例执行解锁即可（无论是否获取到锁都是这样）。
n 可以为 5，足够可靠。

如果已经拿到锁，要延迟锁的时间，可以再次向所有 redis 实例发送 lua（如果之前的 key 还存在并且 value 没变，即锁属于自己，设置新的过期时间）。
同加锁，只有在多数实例上完成更新，才认为成功延长了锁。

实际不常用，因为：

1. 成本高？需要多个 redis 实例（多个集群）。
2. 依赖系统时钟。多个实例间的时间可以不一样，但时钟频率需要一样，并且时钟不能发生变化，至少相对于失效时间来说差别可以忽略不计。
   （每个实例上的实际过期时间，是 实例设置 key 时的时间 + 过期时间，设置 key 时的时间肯定是不同的）

安全性：
在锁的有效时间内，每个实例上只会有一个客户端成功加锁；所以能在至少超过一半的实例上成功加锁的客户端，只有一个。

崩溃恢复：
redis 宕机重启后，可能丢失数据，即丢失某个已经加锁的 key，导致第二个客户端加锁成功。
如果要求 AOF 每次操作都写入磁盘，可以避免，但会非常影响性能。
但只要一个重启的 redis 节点，不参与到当前任何已有的锁，也就是不允许任何客户端从该节点上获取已存在的锁，就可以避免锁被获取两次的情况。
所以可以让故障的 redis 节点等待一段时间才变为可用。等待时间要大于当前所有锁的过期时间，在该时间后，节点之前持有的锁都会被自动释放。
延迟重启可以在不使用持久化策略的情况下保证安全，但可能会导致系统短时间内完全不可用（多数节点故障重启）。

**zookeeper 实现分布式锁**

> 代码参考：https://www.cnblogs.com/whtt/p/11951640.html
>
> 与 redis 实现分布式锁相比，该锁可以自动释放，不需要考虑节点挂掉、锁无法释放的问题，所以不需要续锁，非常简单；zk 的强一致性也很可靠（不需要红锁）。但效率可能偏低，需要一个 zk 集群。

**写锁（排他锁）**：每个资源的锁对应一个唯一节点。
加锁时，客户端尝试创建对应锁的临时节点，创建成功则获取到锁；创建失败，则监听该节点。
客户端完成业务或宕机，都会删除临时节点、即释放锁。其它监听的客户端会收到通知，重新尝试获取锁。

但这会导致惊群问题：一个客户端释放锁后，会通知所有等待的客户端，但实际上这次通知只对1个客户端有影响，其它客户端总是要继续等待，造成性能和带宽的浪费。
解决方法：类似读写锁，将获取锁的过程分为2步：

1. 客户端首先在该锁对应的节点下（比如 /mutex1）创建一个临时顺序节点。
   节点名称可以记为 `/mutex1/00000001`，即 通用前缀 + 顺序序号。
2. 客户端获取 /mutex1 节点下的所有子节点信息。如果自己是序号最小的子节点，加锁成功，否则等待，监听序号比自己小的 最后一个节点，被唤醒时重复该步骤（也要获取节点、判断序号。触发可能只是因为它的链接断开，并非前面的所有锁都释放）。

本质上就是将请求放入队列串行化。类似 os 中锁唤醒进程的方式。

> 如何实现分布式可重入锁？
> 感觉 zk 这种利用顺序序号 加锁解锁的方式，实现可重入锁比较困难。需要在客户端去做检测是否获取到锁、释放几次锁的逻辑。
> 如果用 zk 的话，我更倾向于避免使用可重入锁，只是会略微增加代码量（如：要将一个函数分成 假设已加锁 f() 和 假设没有加锁 f_need_lock() 两个版本，后者会加锁然后调用 f()）。
> 不过 go 也是不推荐用可重入锁的。见 *go - 可重入锁*。

**读写锁（共享锁）**：每个资源的锁对应一个节点下的若干锁。
获取锁的过程有几步：

1. 客户端首先在该锁对应的节点下（比如 /shared\_lock）创建一个临时顺序节点。
   节点名称可以记为 `/shared_lock/hostName-R/W-00000001`，即 通用前缀 + host name + 读写标记 + 顺序序号。
2. 客户端获取 /shared\_lock 下的所有子节点信息，并监听子节点的变动信息。
3. 如果要加读锁，判断：如果自己是序号最小的子节点，或比自己序号还小的子节点加的都是读锁，则成功获取到锁，否则等待。如果要加写锁，判断：如果自己是序号最小的子节点，则成功加锁，否则等待。
4. 如果没获得锁，当子节点发生变化时，重复步骤 2~4。即重新获取子节点信息，判断能否加锁。

释放锁同上，如果客户端完成业务或宕机，都会删除临时节点、释放锁。

该算法中，每个客户端会监听所有子节点，也会导致惊群问题。
解决方法：客户端只监听影响自己的节点，而不是整个子节点列表。
如果要加读锁，则监听序号比自己小的 最后一个写节点；如果要加写锁，则监听序号比自己小的 最后一个节点。
（同写锁，监听触发后，仍要获取子节点列表、判断是否可加锁）

**zookeeper**

ZooKeeper 允许分布式进程通过共享的命名空间相互协调。这个命名空间的组织方式与标准文件系统类似。
命名空间由数据节点 znode 组成。znode 类似文件系统中的文件，拥有一个唯一路径标识（由斜杠 / 分割的字符串序列），每个 znode 既可以读写数据，也可以关联若干子 znode，相当于文件和目录的结合（但临时节点不能有子节点）。
zk 的用处是多节点协调，所以节点存储的数据通常很少。所以与专为存储而设计的典型文件系统不同，zk 的数据保存在内存中，非常高效。

zk 可以创建集群，所以很可靠、有高可用性。集群使用 paxos 通信。
zk 可以实现严格有序的访问，可用于在分布式系统中的多个客户端之间实现同步。

znode 有两类，一类是持久节点 (persistent)，一类是临时节点 (ephemeral)。客户端可以创建也可以删除两类节点。
客户端和 zk 断开连接后，zk 会自动把它创建的临时节点删除。
作为目录节点的 znode 可以按是否顺序分为 4 类：PERSISTENT、PERSISTENT\_SEQUENTIAL、EPHEMERAL、PHEMERAL\_SEQUENTIAL（持久/临时目录节点、持久/临时顺序目录节点）。
创建顺序节点时，znode 名称后会附加一个单调递增的顺序编号。顺序编号会由父节点维护。

客户端可以在某个 znode 上设置 watcher 来监听节点（类似一个注册在 znode 上的触发器），当 znode 的状态发生改变时（如 创建了新的子节点、被删除、数据修改），zk 会异步通知客户端。

zk 提供的 API 非常简单，见 https://blog.csdn.net/ximenjianxue/article/details/100537672。

**zookeeper 的应用**

> https://www.cnblogs.com/leesf456/p/6036548.html

zk 可作为**配置中心**：
每个应用在启动时去 zk 获取配置信息，并注册一个 watcher。配置信息发生变化时，zk 会通知应用。

**生成全局唯一 id：**
客户端调用 zk 创建顺序节点的 api，就可以获得唯一 id（也可以再拼接一个前缀）。

**集群管理：**
监控系统监听某个节点 x。在集群中添加机器时，在 x 节点下创建一个临时节点；当集群中某个机器断开连接时，zk 会删除它创建的临时节点。两种消息都能被监控系统监听到。

**zookeeper 的 watch 机制**

zk 客户端有两个线程，一个 send 线程负责通信，另一个 event 线程负责监听。
send 线程可以在 zk 的服务端注册 watcher，然后将该 watcher 保存在本地的 watch 管理器中。
当监听的节点变化时，zk 会通过 event 线程通知客户端，然后执行 watch 管理器中的相关回调函数。

watch 的特点：

- 需要先注册才会触发。
- 注册后之后触发一次，如果数据之后再次变化，不会再次通知。
- watcher 事件会异步发送到客户端的监听线程；客户端会串行执行这些事件。
- 事件对象非常轻量，只包含通知状态、事件类型和节点路径，具体的变化需要客户端自行获取。

**BASE**

与 ACID 类似，BASE 是以下三个特点的缩写：

- Basically Available (基本可用)：不保证事务的参与方一定都可用。
- Soft state (柔性状态)：允许系统的状态更新有一定延迟，即存在一个不影响系统可用性的中间状态。
- Eventually consistent (最终一致性)：能够保证系统的最终一致性。

满足 BASE（但不满足 ACID）的事务称为柔性事务。
BASE 是 AP 的延伸，通过放宽对一致性的要求，提高系统的吞吐量。

> 遵守2PL算法的并发调度一定是可串行化的
>
> 事实上分布式事务用的很少，基本都是失败时补偿来实现最终一致性。
> 数据对不上时，可以人工修正，或用脚本处理。
>
> 补偿：一是应用层无需增加额外补偿逻辑；二是，失败回滚是少数情况，整体性能高于两阶段提交，只需要一次交互就可以了，大大节省了单机资源。

分布式事务指事务的参与者位于分布式系统中的不同节点上，要保证不同节点上的操作依旧满足事务的 ACID 特性。
比如创建订单服务，就涉及订单系统和库存系统两者的分布式事务。

难点在于：

1. 网络通信不可靠，所以事务的执行需要用多次通信来保证。
2. 存储节点的增加，放大了单个存储节点在事务过程中出现故障的风险。
3. 如果用锁实现事务隔离性，在某节点故障或网络不稳定的情况下，会严重影响性能。

> 关系数据库的本地事务满足 ACID，但它无法在多个数据库之间提供事务支持。
>
> 分布式事务中，如果某个服务或资源挂掉，也会很影响其它服务（比如导致每次执行事务的响应时间延长）。因此还需要额外备份。如果要一个状态、数据完全一致、即强一致性的备份，还要用 raft、paxos 等共识协议实现一个强一致集群。但正确实现强一致的可用系统的成本比较高，也不能保证很安全，可能并不值得。
> 总的来说，分布式事务的使用成本很高，小公司、普通业务可能不常用，直接用一致性比较低、但简单的算法。

**尽最大努力通知**

实现最终一致性的最简单方法：
本地事务执行完成后，发送到 mq；消费放会取出消息进行执行，如果失败就多次重试。
mq 有持久化机制。最终总会成功，即认为服务最终总会可用。

**本地消息表**

本地消息表可以简单实现最终一致性：

1. 当服务需要操作多个资源时，可以在本地数据库上开启事务，该事务会执行在这个数据库上的业务，并在消息表中插入一条记录，它记录了对其它资源的操作。两者作为一个事务一起提交或失败。
2. 服务会用一个脚本，定期轮询消息表中的消息，将它发到 mq 中。如果失败就重试，直到消息被指定资源所处理。

本质就是利用数据库持久化一系列操作，然后异步、多次发送幂等请求，直到成功。

**可靠消息服务**

也可以利用 mq 的持久化机制做到上面的事（但要保证消息成功发送）（不知道和上面有什么区别）：

1. 服务首先将 prepare 消息发送给 mq，检查是否能发送成功。
2. 如果发送成功，执行本地事务，这时才将操作资源的信息发送给 mq。如果发送失败，发送补偿消息并回滚自己的事务。
3. mq 的消费方定期检查并消费消息。如果执行成功，发送 ack；如果执行失败，重试；如果业务逻辑不能执行，向原服务发送回滚信息？

**XA**

> TODO：
> Flink 如何对外部数据库一起做2PC：https://www.ververica.com/blog/end-to-end-exactly-once-processing-apache-flink-apache-kafka
>
> https://zh.wikipedia.org/zh-sg/X/Open_XA

> XA 是最早的分布式事务模型，是由 X/Open 国际联盟提出的 X/Open Distributed Transaction Processing (DTP) 模型，全称 X/Open XA 协议。
> DTP 模型中包含一个全局事务管理器 (TM, Transaction Manager) 和多个资源管理器 (RM, Resource Manager)。全局事务管理器负责管理全局事务状态与参与的资源，协调资源一起提交或回滚；资源管理器则负责具体资源的本地事务操作。各个资源自己的事务是全局事务的一个分支 (branch transaction)。
> TM 和 RM 之间通讯的接口规范叫 XA。资源通常就是指关系型数据库，还有消息队列等，需要支持 XA。
>
> 几乎所有主流数据库都支持 XA 规范（比如 MySQL 的 InnoDB，Oracle），可以理解为提供了一个两阶段提交接口。
> XA 会保证执行完成的事务都被持久化、不会丢失，且可回滚。
>
> 2PC 流程不算复杂，更难的是各个环节的故障处理和容错（比如节点直接 crush）。

XA 模式就是基于数据库的 XA 规范来实现的 2PC 协议（两阶段提交协议），对业务没有侵入性。
分为两步：

1. prepare：事务管理器向所有资源管理器发起请求，询问能否执行指定命令；如果能，资源就执行，并锁定相关数据，但不提交。
2. 根据情况分为两种：
   commit：如果所有资源都成功完成第一步，提交。提交必须成功，如果失败会重试。
   rollback：如果有资源没有完成第一步，回滚，释放锁。

如果一个全局事务只涉及一个资源，可以变成 1PC。

> 如果 XA 事务的第一阶段中，事务管理器要设置超时机制，在某个资源故障或无法联络时及时回滚，并保证故障资源中的锁最终会释放。这个框架会帮我们实现。

事务的隔离性要求，状态转换之间的中间状态，也就是脏数据，不应该被其它任何事务所观察到。
本地事务中，可重复读通过 MVCC 来保证不会读到其它事务的中间状态。但在分布式事务中涉及多个节点，可重复读就不足以避免读到中间状态（如[MySQL 文档](https://dev.mysql.com/doc/refman/8.0/en/xa.html)中所说）。
XA 协议没有定义数据库该如何实现分布式事务，但要保证事务的隔离性、保证不出现脏读，需要使用可串行化隔离级别（对于本地事务，只需要 MVCC）。
可串行化一般就是 MVCC + next key lock，或 S2PL，都需要锁来保证后续请求在读写的时候都会阻塞、不会读脏数据。

> 可以证明，可串行化隔离级别下，XA 能避免脏读（读未提交），因为写要加写锁，读也要加读锁。
> 虽然第二阶段的提交不能同时提交，但可以保证如果第二阶段开始（各数据库真正开始提交），其它事务要么能读到所有新数据，要么读到的都是旧数据即可。
>
> 设一个 XA 事务 T 修改了 A 节点上的 a -> a'，修改了 B 节点上的 b -> b'，然后进入提交阶段。
> 如果另一个事务 T2 读到了新的 a'，说明 T 的一部分已提交，如果 T2 想读 b，那么要么直接读到新的 b'，要么由于 T 锁住了 b 导致 T2 阻塞，等待后读到新的 b'。
> 如果另一个事务 T3 读到了旧的 a，想读 b；由于 T3 锁住了 a，T 无法进入提交阶段，所以 T3 只会读到旧的 b。
> 所以另一个事务要么读到 a, b，要么读到 a', b'，满足原子性和隔离性，没有脏读。
>
> 如果各资源都是可串行化，那么实现的 XA 也是可串行化的。
> 但串行化隔离级别的性能较差，所以很多分布式数据库都自己实现了分布式 MVCC 机制来提供全局的读已提交。一个基本思路是用一个集中式或者逻辑上单调递增的东西，生成和分配全局 Snapshot，每个事务或者每条 SQL 执行时都去获取一次，从而实现不同隔离级别下的一致性。
> 比如 Google 的 Spanner 就是用 TrueTime 来控制访问全局 Snapshot；percolator 用 timestamp oracle 生成 snapshot；pg-xc 用 gtm 生成 snapshot，cockroach 使用 hlc 来生成时间戳。

**AT**

> 文档：https://seata.io/zh-cn/docs/dev/mode/at-mode.html

AT 模式也分为两阶段：

1. 第一阶段：每个资源执行 SQL 并提交，并在 undo\_log 表中写一条数据用于回滚。两者通过一个本地事务完成。
2. 第二阶段：如果有资源第一阶段没有成功，通过之前记录的日志全部回滚。

只在执行事务时需要锁，锁粒度比 XA 小，但效率也一般（相比本地事务）。
AT 也可以由框架实现，对业务几乎没有侵入性（seata 中只需要修改数据源代理，即可实现 XA 与 AT 之间的切换）。

结束后，可以异步删除 undo\_log 中的数据。
使用 seata 中的 AT 模式，需要创建 undo\_log 表。回滚记录会记录更新前后的数据镜像（包含各个字段）。

因为第一阶段内锁就会被释放，没有实现隔离性，其它本地事务是可以读写脏数据的（脏写）。如果该事务回滚，且它在第一阶段修改的数据已经被其它事务修改，回滚会失败（否则另一个事务的修改会丢失）。
为了避免这种情况，**在一个分布式事务提交前，需要**在事务协调者 TC 那里给相关数据**加全局锁**，避免另一个分布式事务写脏数据并提交；**在二阶段完成后，才会释放全局锁**。
如果一个事务提交前没有拿到全局锁，会回滚并释放本地锁，然后重试一定次数。

但是，资源的本地事务不经过 TC，所以依旧会出现写脏数据的情况。
因为 undo\_log 中会记录数据修改前后的值，所以全局事务在回滚时，能发现是否有其它事务做出过修改。如果有，回滚会失败，并不断重试。
解决方法有三个：

1. 用`@GlobalTransaction`注解将本地事务改为全局事务（分布式事务），但显然会影响效率。
2. 用`@GlobalLock`注解标识本地事务，也尝试获取 TC 那里的全局锁，避免与某个全局事务冲突。如果获取不到，可以重试（默认30次，可在注解中配置）。具体见[这里](https://yunyanchengyu.blog.csdn.net/article/details/123131592)。
3. 人工处理，修改脏数据。

如果数据库的隔离级别设为读已提交，则 seata AT 模式实现的分布式事务隔离级别为读未提交（允许脏读，但不能脏写）。
如果要求全局事务的隔离级别也是读已提交，需要给读也加全局锁，也就是用`select ... for update`而非`select`。
出于性能的考虑，seata 只对`select ... for update`进行了代理（全局处理）。

**TCC**

TCC 也是 2PC，但是是由业务自己实现所有逻辑，不需要资源支持 XA 协议。
TCC 要求业务提供服务时，必须提供某些不确定状态，来掩盖中间的不一致状态，对业务的侵入性比较高。

TCC 首先冻结资源，等待各服务反馈后提交或回滚，本质上也是一个在业务层面上实现的 2PC（比如优惠券从 冻结 变为 已使用）。
与 2PC 一致，分为两步：

1. try：各个服务检查是否能执行请求，并冻结相关资源，但不真正执行。
2. confirm：如果所有服务都成功完成第一步，利用第一步中冻结的资源执行业务（不需要再检查）。业务的执行必须成功，如果失败需要重试。
   cancel：如果有服务没有完成第一步，取消执行，释放冻结的资源。

对每个分布式事务，需要实现 try, confirm, cancel 接口，开发成本高。
因为涉及失败重试，confirm 与 cancel 的业务逻辑必须是幂等的（操作恰好只生效一次）。

TCC 很常见，比如用户下单时，库存服务会在 try 阶段检查库存是否有余量，然后冻结指定数量的库存。如果用户长时间没有支付，订单会被释放，库存服务进入 cancel 阶段，释放冻结的库存。如果用户进行支付，且支付服务冻结用户的部分余额成功，两个服务就可以进入 confirm 阶段，告诉用户成功，并确保冻结的库存与余额最终会扣减。

TCC 也只是实现最终一致性，中间会读到脏数据（系统可能处于不一致状态），但会通过一个额外的状态掩盖，比如数据会被标记为“冻结”或“结算中”。业务可以识别这些脏数据，保证不会修改这些脏数据，所以没什么影响。
但需要业务之间有规定，并不像真正的事务之间能完全隔离一样（隔离性）、非常通用。

**saga**

> The saga pattern provides transaction management using a sequence of *local transactions*. A local transaction is the atomic work effort performed by a saga participant. Each local transaction updates the database and publishes a message or event to trigger the next local transaction in the saga. If a local transaction fails, the saga executes a series of *compensating transactions* that undo the changes that were made by the preceding local transactions.
>
> https://learn.microsoft.com/en-us/azure/architecture/reference-architectures/saga/saga
>
> 文档：https://seata.io/zh-cn/docs/user/saga.html

与 AT 类似，第一阶段各资源执行本地事务并提交；如果有资源没有成功，进入第二阶段，全部利用补偿进行回滚。
只在本地事务执行时需要锁，效率高。

这就是最简单的最终一致性实现方案，先执行所有事务，如果失败了再调补偿接口。
可以与外部无法修改的系统一起使用，只需要对方提供补偿或反向操作的接口，不需要像 TCC 一样提供三种接口，或提供对 XA 的支持。
补偿接口通常容易实现，但存在无法补偿的情况（如下）。

与 AT 一样，saga 没有保证隔离性，极端情况下，会出现事务修改未提交数据，导致原事务可能没有办法进行补偿或回滚。
比如有一种进阶红包，每个用户同一时刻只能拥有一张，消费后就能领下一张优惠更大的红包。如果一个分布式事务 A 先使用红包，用户在另一个事务 B 成功领取了更大的红包，然后 A 失败，就会导致无法回滚或补偿，可以告诉用户因为订单失败，领的红包被收回；但如果用户已经把红包使用了，就会很麻烦。

在 seata 中使用 saga 时，需要根据服务流程创建一个状态机，具体见官网描述。

使用时要注意几点：

- **允许空补偿**（原服务未执行，但补偿服务执行了）。
  出现原因：服务执行超时，服务没收到请求、但收到补偿请求，全局事务回滚。
  解决：可以在没有找到要补偿的业务主键时，返回补偿成功，并将原业务主键记录下来。
- **防止悬挂控制**（补偿服务比原服务先执行）。
  出现原因：服务执行超时，服务的请求比补偿请求晚到达，全局事务回滚。
  解决：每次执行请求时，要检查当前业务主键是否已经在空补偿记录中存在，如果存在要拒绝服务的执行。
- 保持幂等，原服务与补偿服务都需要保证幂等性。
- 业务流程设计时遵循“宁可长款，不可短款”的原则，长款意思是客户可以少了钱，但机构要多了钱，还可以以机构信誉可以给客户退款；反之则是短款，少的钱可能追不回来了。所以在业务流程设计上一定是先扣款。
- 有些业务场景可以允许让业务最终成功，在回滚不了的情况下继续重试完成后面的流程。所以状态机引擎除了提供“回滚”能力还需要提供“向前”恢复上下文继续执行的能力，让业务最终执行成功，实现最终一致性。

seata 的特点：

1. 应用层基于SQL解析实现了自动补偿，从而最大程度的降低业务侵入性；
2. 将分布式事务中TC（事务协调者）独立部署，负责事务的注册、回滚；
3. 通过全局锁实现了写隔离与读隔离。

在 seata 的架构中有三个角色：

1. TC (Transaction Coordinator)：事务协调者，维护全局和分支事务的状态，驱动全局事务提交或回滚。
2. TM (Transaction Manager)：事务管理器，定义全局事务的范围：开始全局事务、提交或回滚全局事务。
3. RM (Resource Manager)：资源管理器，管理分支事务处理的资源，与TC交谈以注册分支事务和报告分支事务的状态，并驱动分支事务提交或回滚。

TC 为单独部署的 Server 服务端，TM 和 RM 为嵌入到应用中的 Client 客户端。

**seata 各模式的对比**

**XA 与 TCC/saga**

前者对业务没有侵入性，也就是不需要设计特别的业务需求、大改业务来实现；能真正实现事务的 ACID 特性。
但是会影响性能，而互联网业务通常更重视响应时间，没有很强的一致性要求，所以 XA 并不常用。

TCC 的工作量很大，涉及分布式的工作都需要自己写提交和回滚逻辑；如果业务修改，也要进一步更新。
只是保证最终一致性，所以用户会看到中间状态，即脏数据。

**XA 与 TCC**

- XA 是基于 XA 协议实现的分布式事务，作用于资源层，对业务的侵入性很小。只要资源支持，就适合于各种场景。程序员可以像使用本地事务一样，使用基于 XA 协议的分布式事务。
  XA 会利用数据库的支持，对数据进行锁定、提交或回滚，程序员只需要写 SQL 实现业务（AT 也是）。
  
  而 TCC 是在业务层面上冻结资源（比如我有100块余额，消费了50但没有确认，就会有50块的冻结余额），不是直接锁数据，所以依旧允许其它请求读写相关数据（读写余额、冻结数额），效率高，但与业务强相关（引入冻结状态与数值），需要程序员自己提供 try, confirm 与 cancel 的实现（saga 也是）。
  
- XA 协议能够保证事务的 ACID 特性，但需要资源使用可串行化隔离级别，即各资源要给数据加锁。所以 XA 可能导致后续请求阻塞；如果事务管理器或某个资源长时间不响应，阻塞的时间就更长。
  所以更适合执行时间短的短事务，很影响系统的并发性能（对于热点数据，相当于串行执行）。

- XA 需要资源有对 XA 协议的支持。不过一般的商业数据库和队列都支持 XA。

**TCC 与 saga**

TCC 是先冻结、再提交或取消，而 saga 是直接执行、如果出问题再回滚。

大多数情况下系统都是正常的，而 saga 只需要一次执行，效率要更高。
但是 saga 没有隔离性，会直接读到脏数据，比如创建订单中优惠券扣减服务失败，订单需要回滚，但在一小段时间内，用户会先看到优惠券被使用，然后又看到优惠券没有失败。如果还有其它服务，需要对优惠券使用信息进行统计，还需要额外的状态回滚处理。
TCC 虽然也会读到脏数据，但会通过一个状态对它进行掩盖，比如给优惠券加一个使用中的状态，也就是第一步冻结资源；在扣减成功之后，修改优惠券为已使用，也就是第三步 commit。服务之间还是有延迟，但用户会觉得很合理、体验更好。

saga 比较简单，只需要服务提供一个补偿接口，所以如果有外部、难以修改的服务，依旧能用 saga。

**XA 与 AT**

类似 TCC 与 saga，XA 是 2PC，性能低些但隔离性强，而 AT 是直接提交，如果出现问题再回滚，可能读到脏数据。

**雪花算法**

> 数据库中的数据最好按索引顺序插入（一般就是主键索引的顺序？因为主键索引最大），可以减少更新索引的代价：顺序插入会使 B+ 树节点分裂和合并的次数最少；因为数据是有序的，所以顺序插入也可以减少数据页分裂的次数，减少 IO。
>
> 数据整体上有序，也方便作为 KV 的 key。

雪花算法用于在分布式系统中生成唯一的递增 ID。ID 有64位（十进制19位）：

- 第1位为符号位，始终为0，无意义。
- 41位时间戳，单位为毫秒。41位可以用69年。
- 10位机器标识，标识号可以是机器 ID，也可以是机房 ID +机器 ID，可以标识 1024 台机器。
- 12位递增序列号，允许同一机器在1ms内最多生成4096个不同 ID。
    只要保证单机1ms内生成 ID 数小于4096，就可以一直 seq = (seq+1)%4096，不需要重置。

用的时候按照这个规则生成即可。
时间戳使用时可以减去一个起始值（如2020年），从而延长有效时间。
由于机器标识号不同，所以集群生成的 id 不是严格递增的，只是整体上是递增的。也可以把序列号放到前面位，然后用 redis 自增去取，保证全局递增。

因为需要时间戳，所以有个时钟回拨问题：如果机器上的时间被修改，就可能生成过去已有的时间戳。
解决方法：1. 如果发现当前时间早于之前生成时间戳的用过的时间，不生成，等待时间恢复（但只适合回退较小的情况）。2. 用几个额外位，每回退一次就给这些位+1，避免与其它 id 冲突（但要占用其它位，且显然并不能用很多次）。

## 组件

**kafka**

> TODO：
> https://www.cnblogs.com/leesf456/p/6063694.html

kafka 是一个实现了发布/订阅模型的消息队列，其中包括 3 种角色：

- producer：生产者，将客户端生产的消息发送给 kafka。可以异步和批量发送。
- consumer：消费者，接收 kafka 的消息进行处理。
- broker：代理/中间人，接收生产者的消息，转发给指定消费者。kafka 集群中的每个服务器都是一个 broker。

kafka 中的消息以 topic 进行分类，不同 topic 可以看做不同的消息队列。每一条消息都需要指定 topic，然后转发给订阅了该 topic 的消费者。
为了提高消息处理的效率，topic 还被分为若干 partition，保存在不同 broker 中，使得消费者能够同时消费某个 topic 中的多条消息。
消息可以根据不同策略，选择发送到哪个 partition。比如：如果用哈希决定，则相同的消息总会被发给同一 partition；可以负载均衡；可以根据消费者数量。
对于同一个 partition 内的消息，kafka 会保证消息按序进行处理；但不同 partition 之间在消费时不能保证顺序。

多个 consumer 可以组成一个消费者组，分别消费指定 topic 中不同 partition 的消息。同一消费者组中的各个消费者，不会消费同样的消息。

为了提高可靠性，每个 partition 都会有多个副本，保存在不同 broker 中。
partition 的多副本机制与 raft 类似（通过 zookeeper 实现的？），每个 partition 包含一个 leader 和多个 follower，leader 会处理该 partition 的所有读写请求，并与其它节点进行同步；follower 只用于备份，不用于读写。
当 leader 故障时，会选举新的 follower 作为 leader。见下 kafka 中的选举。
副本数量通过 副本因子 设置（设为 3 表示 1 个 leader、2 个 follower）。

**kafka 中的选举**

主要的选举有两个：

**Controller 选举：**
kafka 集群中有多个 broker，其中的一个 broker 会成为 kafka controller，管理整个集群中所有 topic、分区和副本的状态。
当某个 partition 的 leader 挂掉后，它负责选举新的分区 leader。
controller 的选举通过 zookeeper 完成：成功在 zk 中创建`/kafka/controller`临时节点的 broker，会成为 controller。其它 broker 监听该节点，在节点因为 controller 挂掉而删除时重新尝试。

**分区 leader 选举：**
每个 partition 包含多个副本，由一个 leader 管理。
当分区 leader 挂掉，或新的分区创建（包括创建新的 topic）后，选择 follower 集合中的第一个，成为 leader。
该 follower 要与 leader 保持一定程度的同步（属于 ISR 集合），以减少数据丢失，但可以通过配置关闭（`unclean.leader.election.enable = false`，允许 OSR 集合中的节点选举）。

还有两类选举：
**GroupCoordinator 的选举：**

> GroupCoordinator是负责执行消费者的分区分配和再均衡操作，在初始阶段，当消费者未保存与消费组对应的GroupCoordinator节点信息时，需要通过向集群中负载最小的节点发送请求来寻找，Kafka通过消费组的groupId的哈希值计算__consumer_offsets中的分区编号，找到分区后，再寻找分区leader副本所在的broker节点，该节点就是对应的GroupCoordinator，消费者最终的分区分配方案以及组内消费者所提交的消费位移信息都会发送给次分区leader副本所在的broker节点。

**消费者组 leader 选举：**
GroupCoordinator 需要为消费组内的消费者选举出一个消费组的 leader。
选举方式很简单，第一个加入消费组的消费者为 leader。如果原 leader 退出，在消费者记录中选第一个作为 leader（就是随机选）。

**kafka 分区中的主从同步**

客户端只能读写 leader，follower 会与 leader 进行同步。

有三个概念：ISR (in-sync replica) 是能够成为 leader 的节点集合，OSR (out-sync replica) 是默认不能成为 leader 的节点集合，AR (all replica) = ISR + OSR。
记 LEO (log end offset) 为下一条消息的 offset（即最新的 offset + 1），HW (high watermark, 高水位线) 为 ISR 中最小的 LEO。消费者最多只能消费到 HW 之前的消息，因为后面的消息还没有在 ISR 中的所有 follower 上完成同步（不够可靠）。

follower 会向 leader 发起 fetch 请求，获取最新消息，更新 LEO，从而更新 HW，允许消费者读更多消息。
符合条件的 follower 可被加入 ISR。

故障恢复：
follower 宕机重启后，会将自己 HW 之后的消息删除，然后再和 leader 同步。
leader 宕机后，在 ISR 中选择一个 follower。为了保证数据一致，其它 follower 需要把自己 HW 之后的消息删除，重新同步。

可见，上述机制只能保证数据一致，不能保证 HW 之后的消息不丢失。
想要保证消息不丢失，需要做其它配置，见下。

**kafka 配置**

> 文档：https://kafka.apachecn.org/documentation.html#configuration

4 个与数据一致相关的配置：
`request.required.acks = all`：生产者端的设置，要求消息被写入到所有副本（还是 ISR？）才算是成功。
`min.insync.replicas`：最小副本数量，数量需要大于 1，来确保 ISR 中至少有一个 follower，能够在 leader 挂掉后替代 leader。
`unclean.leader.election.enable`：选举时是否能在 OSR 中选举 leader。默认为 false，否则可能丢数据。
`enable.auto.commit`：消费端的设置，设置为 false，可以确保消息在消费完后才手动提交。
