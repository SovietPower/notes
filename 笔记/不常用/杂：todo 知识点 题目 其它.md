**布隆过滤器 Bloom Filter**

用于高效判断某个数据是否在某个集合中。使用一个大位图和k个哈希函数。
添加一个值时，将位图中 该数据的k个哈希值的位置设为1。
判断一个值时，如果位图中 该数据的k个哈希值的位置均为1，认为它存在。

```cpp
// add(x)
for i in k
    B[hash(i, x)] = 1
// check(x)
for i in k
    if !B[hash(i, x)]
        return false
return true
```

优点：与哈希表相比，使用k个哈希函数，提高了准确度；不记录原数据，空间占用少。
缺点：可能误判（但只会将原本不存在的认为存在，存在的不会认为不存在）；不能删除里面的元素；不能准确知道里面的元素。

数据量、哈希函数个数、位图大小，有数学关系以保证一定正确率，见[这里](https://pages.cs.wisc.edu/~cao/papers/summary-cache/node8.html)或[这里](https://en.wikipedia.org/wiki/Bloom_filter)。
有时需要建立白名单，来避免误判情况。

应用：快速判断一个邮件是不是垃圾邮件；爬虫时对 URL 进行去重，避免爬到相同的；缓解缓存击穿问题。

**Counting Bloom Filter**

允许删除元素。应该是用几位数字取代一位，实现计数。

**如何压缩整数序列**

**1. 变长编码**

因为很多常用数字的前面有很多没用的1，所以类似 utf-8，可以设计一个变长编码。
首先只存放数字从最高位的1往后面的位。编码后的数最后一个字节的第一位为1，代表这个数的结束，前面所有字节的第一位为0。然后这些字节的剩下7位依次存放 原数字从最高位1往后面的那些位。

适合于较小数字出现频次高的情况，比如 utf-8。
负数因为是补码表示，所以前面会有很多1 不利于压缩。所以可以保存数的绝对值，然后在最开始加一个符号位。

**2. 缩小值域使用不同编码**

适用于有序序列，见 *数据库 - Elastic Search*。
也可使用位图+数组，即 roaring bitmap。

**3. 前缀压缩**

类似 RocksDB data block 存储连续 key 的方式，每个 entry 存储 shared_bytes 记录与前一个 key 公共的前缀长度，然后只存 unshared_bytes 长度的不同后缀。

**一根木棍分成三段，能组成三角形的概率是多少**
组成三角形的条件是，对任意一边 x，另外两边之和大于 x，即$x<1-x$，即每条边的长度要小于$\frac 12$。

有两种情况，首先是两个分界点随机选择的情况（两个点是不同的）：
考虑两个分界点的位置，可以积分（令 x 为靠左的点到左端点的距离，注意因为两个点是不同的、分界点的位置对称，所以结果要乘2）：$2\int_{0}^{0.5}\int_{0.5}^{x+0.5}dxdy=2\int_0^{0.5}dx\cdot(x+0.5-0.5)=2\cdot\frac 12\cdot 0.5^2=\frac14$。
积分是概率，因为是均匀分布 U(0, 1)，所以取任一位置的概率是$\frac1L=1$。
另一种方法是：首先要保证，两个分界点在中点的两边，这个概率是$\frac 12$（固定一个，考虑另一个；或考虑所有4种情况。一个分界点恰好取在中点的概率为0？）。
然后要保证，两边的两个分界点，距离 $l$ 不能超过$\frac12$。可以发现 $l$ 的取值在 $(0,1)$ 上的分布是对称的，对称轴是$\frac 12$，所以 $l<\frac12$的概率就是$\frac 12$。两者要依次满足，所以概率为相乘$=\frac14$。

另一种情况是，先切一次，再在剩下的部分（它的右侧）切一次：
首先考虑第一个点的位置为 $x\in(0, 1)$，第二个点在剩余木棍上的范围为 $y\in(0, 1-x)$（与第一种情况不同，不是对称的）。
第二个点想要在原木棍上的右侧，必须从$0.5-x$的位置开始取（因为该线段的起点位置是$x$，在$0.5-x$位置开始取可保证$y$在右侧。或保证了$x+y_{min}>0.5$）。想要与$x$距离不超过 0.5，就只能取到 0.5（起点是$x$！）。
可行的概率为：$\int_0^{0.5}\int_{0.5-x}^{0.5}\frac{1}{1}\cdot\frac{1}{1-x}dxdy=\ln 2-\frac 12$。
注意 x 是 U(0, 1)，任一点的概率是 1；y 是 U(0, 1-x)，任一点的概率是 $\frac{1}{1-x}$。

**抛硬币**
A有 n+1 个硬币，B有 n 个硬币，A比B得到更多的正面朝上的概率是多少?
我们先假设甲乙都只抛了 n 次，则 甲 > 乙 和 乙 > 甲 的概率相等，设为$p$，则$1−2p$为相等的概率，那么甲的第$n+1$次正反面的概率为都$0.5$，所以 甲>乙 的概率为：$p+0.5∗(1−2p)=0.5$

**实现 memcpy**
因为一条指令一次最多处理8B数据，所以分成多个 long long 去分别传。

```cpp
void *memcpy1 (void *destination, const void *source, size_t num)
{
    int i = 0;
    char *dst = reinterpret_cast<char*>(destination);
    char *src = reinterpret_cast<char*>(const_cast<void*>(source));
    while (i < num)
        *(dst + i) = *(src + i), ++i;
    return destination;
}
// num/8
void *memcpy (void *destination, const void *source, size_t num)
{
    int div = num>>3, mod = num&7;
    long long *dst = reinterpret_cast<long long*>(destination);
    long long *src = reinterpret_cast<long long*>(const_cast<void*>(source));
    int p = 0;
    while (div--)
        *(dst + p) = *(src + p), p += 8;
    return memcpy1(destination + p, source + p, mod);
}
```

**挑一个最有挑战的项目说一下，不用说技术细节**

就说那个 6.824 吧，里面涉及到多线程或多协程。多个协程之间是共享很多资源的，如果不加限制，它们可以同时读写一个变量或数据，这非常容易出错。（需要举个例子吗？比如银行里我卡里有100块钱，想全转出来，我同时在两台机器上进行操作，机器1发现我确实有100块，进入下一步；机器2也发现我确实有100块，也进入下一步。下一步它们同时完成转账，我就转出去了200块钱）
想避免这种情况，就要保证任意时刻只能有一个协程拥有这个资源，就是给这个资源加锁。其它协程要等待锁被释放后，才能去加锁，操作资源。
这涉及一个锁粒度大小的问题，如果机器一开机就锁住，关机才释放锁，那肯定很安全，但其它机器就会长时间获取不到资源，可能无法正常执行，程序效率会很差。
想要提高效率，就要能准确判断 这个资源什么时候才被使用、什么时候真的不再被使用，在这两个位置加锁和释放锁。但后者是很难判定的，搞错了就会出现多线程相关的问题，这种问题是很难调试和复现的，因为取决于程序运行的状况。

**团队中遇到问题怎么去说服别人，或者你是怎么被别人说服的**

我觉得最重要的有几点，一是：要表达清楚自己的观点和逻辑，并且要倾听对方的合理观点，不能像网上很多人一样只会情绪输出，不讲任何有意义的论点。即使不认同，也要理解别人的观点，才能更有意义地反驳。
二是：除非确实有逻辑严谨的理论证明，不然不能假设自己就一定是对的、别人就一定是错的。
三是：讨论的氛围要平等、开放，鼓励所有人表达自己真正的想法。
四是：求同存异吧，尤其是主观性问题，有时候没必要一定要把对方说服。要寻求共同利益和适当妥协吧。

再具体些，问题或者分歧有两种，
一种是客观的、或理论性的问题，这种我觉得只适合用反例或理论分析去说服其它人，只凭“我认为”是远远不够的。
另一种是主观性、或者难以用理论验证的问题，这种想说服人是最难的，可以利用已有的事实和数据统计去进行反驳。

**怎么处理工作与生活的冲突**

1. 明确工作和生活上的优先事项，然后再进行时间和资源的分配。
2. 不管是工作还是生活里的某些事，制定一个合理的计划和截止日期、管理好时间、避免拖延都很重要。
3. 划清工作与生活之间的边界，工作时就专注于工作，平时生活要充分休息。

**如何看待加班**

如果项目紧急，或没有如期完成任务，或刚来对业务、架构完全不了解，加班还是有必要的。
如果能学一些东西、积累更多经验，或做一些有意义的事，加班有时候也是值得的。

当然最重要的还是高效、安全地完成任务，不能指望平时随意些、然后临时多努力点就把一切做好。
以及，也要看个人状况，很精神、没什么事的时候当然可以加；但如果很累、硬加班，并不会带来工作效率的提升。
