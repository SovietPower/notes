### 项目介绍

通过词法分析、语法分析和语法制导翻译，实现对包含分支、循环的C语言程序的解析。

>SDD（语法制导定义）是上下文无关文法和属性与规则的结合，应该是语义分析的一种实现（不确定，找不到）。
> SDT（语法制导翻译(方案)）是完全利用语法分析实现语义分析和翻译的方式，是SDD的一种实现方式：将语义动作嵌入到产生体。
> 很多情况下，翻译可以直接扫描分析完成，不需要生成中间代码，不需要显式构造语法分析树。
> 我觉得优缺点与Java选择解释字节码而非AST类似：
> 优点：对比较简单的文法，实现简单高效，不需要翻译另一种形式，然后再解释。
> 缺点：不生成中间表示形式，每次运行都要一遍完整的词法、语法分析过程；整个解释一次完成，对复杂的文法难优化。
> 
>常见语言都会生成中间表示，或直接生成机器语言，不需要再词法、语法分析。
> Java生成字节码，运行时可以直接解释字节码，或者转成机器码，可以实现平台无关。

### 项目详细内容

**简单介绍下用到的算法：**

词法分析比较简单，在字典树上插入各个关键字和运算符，然后在树上匹配每个符号，判断它是哪个token。

语法分析是构造的SLR语法分析表，因为文法不是SLR的，所以手动处理了分析表中的两个冲突。

语义分析没有构造语法分析树，是构造了一个L属性的SDT，利用LR语法分析，在产生式规约时计算综合属性；添加一些空产生式，在空产生式规约时计算继承属性。

> 综合属性：计算只涉及自己和子节点的属性。比如产生式 A->BCD，在语法分析树中A是BCD的父节点，如果 A.x 属性的计算只涉及A和BCD，则 A.x 是综合属性。
>
> 继承属性：计算会涉及到父节点和兄弟节点。对产生式 A->BCD，继承属性 C.x 的计算可能涉及ABCD 4个节点的属性值。
>
> 通常会要求继承属性的计算只能涉及父节点和自己左边的兄弟节点（C只能使用ABC的属性，不能使用D的），来避免属性间的依赖关系形成环，无法计算。

**具体说下算法的原理：**

**词法分析**

首先是词法分析，是把程序的每个小部分识别成一类预定义的token（词法单元）。比如`INT`这种关键字，机器不能一眼看出来这是`int`，就需要词法分析把它识别成一个`int`token。再比如`int a=1; int b=3;`，我们需要把a和b识别成一个标识符token，只不过分别有一个值a和b，1和3识别成常数token，分别拥有值1和3。所以两句都会变成`int 标识符=常数;`，在语法上都是完全一样的，因为都是声明变量语句。

词法分析的输入是源程序，输出是一个token序列，会作为语法分析的输入。实现用的是字典树，在树上插入各个关键字和运算符，然后在树上匹配每个单词。

**语法分析**

语法分析用来确定连续的几个词有什么意义。比如遇到`int 标识符=常数;`这几个token，我们可以知道这是一个声明语句，就可以在符号表中添加一个对应名字和对应值的变量。再比如遇到`if 表达式 then 语句 else 语句`，我们可以知道根据表达式的值选择执行哪个语句。

实现这个分支语句的方式有很多，最简单的，就是读到if之后，我们再读一个表达式，用这个中缀表达式求出值（用栈保存元素，遇到优先级最高的运算符就弹出来算，再放进去，最后剩下的优先级一致。可以解决两层优先级，多次执行），根据值决定是执行then后面的语句，还是一直读到对应的else，执行else后面的语句。

这种写法没什么算法思想，就是要很多的`if else`和递归。这其实是在写一个自动机：我们遇到if后进入一个读表达式的状态，再根据表达式的值，跳到执行then的状态，或者执行else的状态。这种用自动机表示语言的方法叫3型文法，或者正则文法，因为它就是能用正则表达式表示的所有语言。

可以预想，这种写法把规则完全嵌到了代码里，会有大量if嵌套和递归，不是很方便扩展，表达复杂的语言也很麻烦。JSONParser就是这样写的，因为JSON语法少，自动机就几个状态，所以容易实现。

另一种方法是上下文无关文法，也叫2型文法，相对容易扩展和修改语法规则，最重要的是，它能表达的语言比3型文法，或者正则表达式更多。

2型文法中，分支语句可以这样表示：`ifStmt -> if (expr) then stmt else stmt`，当我们识别到右边的所有元素，就可以将它们规约成一个`ifStmt`，表示它们整体是一个分支语句。然后利用`stmt -> ifStmt | assignStmt`，我们把这个分支语句规约成语句`stmt`；最后利用`program -> stmt`，我们可以知道这条语句可以构成一个程序。

写出来就是：`program -> stmt -> ifStmt -> if (expr) then stmt else stmt`，就可以知道这个程序包含一条语句，这个语句是if语句，它的形式是`if then else`。

这样把token序列的各个部分，规约成一个整体，再把这个整体和其它部分规约成另一个整体，最后规约到只剩一个符号，这个过程叫语法分析。

语法分析分为LL和LR。LR更常用，因为LR能表示的文法比LL更多（几乎所有上下文无关文法都可用LR表示），也很高效。

>  第一个L表示它们都是从左到右扫描输入，完成分析；第二个L表示产生最左推导，R表示产生最右推导。k表示每一步还需要知道当前符号的后k个符号，才能决定动作。
>
> LL的原理：LL能识别的文法要满足：对任意两个产生式，它们的`first`不相交，也就是它们所有可能的开始符号不同，所以根据开始符号，我们每次都能唯一确定要展开哪个产生式。
>
> 所以LL在扫描到一个字符时，就能确定要应用哪个产生式，然后我们去判断后面的部分是不是正好是这个产生式。所以LL也叫自顶向下的语法分析（已知用到的产生式，判断各个符号）。
>
> LR的原理：扫描token，根据语法分析表决定，是把token暂时存到栈里，还是栈顶的部分token和当前token构成了一个产生式，把它们规约成一个token。也就是根据当前各个符号，确定它们是否构成一个产生式，所以LR也叫自底向上的语法分析。

LR的语法分析表决定了每一时刻做什么动作，不同LR算法可以生成不同的表，它们的表示能力各不相同。

最简单的是SLR(simple LR)，可以看做LR(0)（在分析表中有一点区别），也就是只需要知道当前符号是什么，不需要知道这个符号后面有什么，就可以做出决策。所以它需要保存的状态少，效率高，但是表达的语言有限。项目中使用的就是SLR。

> 规范LR方法：也叫LR(1)，需要同时知道当前符号和下一个符号，才能决定如何处理当前符号。在所有状态中都需要加入一个向前看符号，所以生成的状态数很多，分析表可能非常大。
>
> LALR：向前看LR，对LR(1)的优化，合并了LR(1)中的很多状态，它的状态数和SLR一样少，但表达能力更强。一般是最合适的。
>
> 但通过手动处理语法分析表中的冲突，SLR也可表示它本来不能表示的语言（当然有局限性）。

**语义分析**

语法分析只能保证程序符合预定义的文法，并没有实际执行程序。比如我们在识别到一个表达式时，要对其求值；识别到赋值语句时，要执行相应赋值动作。这是通过 给每个符号一些属性、给每个产生式一个语义规则 实现的。

语法分析也不能保证程序含义正确（语义分析是上下文有关的）。比如赋值语句`a = 1.2`在语法上没有问题，但假如`a`没有定义，或者`a`是整数不能赋值浮点，语法分析是检测不出错误的（而语义动作中可以检查符号表中有没有a、检查a的类型）。还有很多常见的依赖上下文的问题：数组下标的表达式结果不是整数，break语句出现在循环外，变量重复声明。

有些应该可以通过细化文法，或者用上下文有关文法表示 来解决，但把这些规则放在语义分析中更好处理，也简化了语法的设计。

比如一个无类型中缀表达式求值的SDD：

| 产生式        | 语义规则               |
| ------------- | ---------------------- |
| expr -> A     | expr.val = A.val       |
| A -> A1 + M   | A.val = A1.val + M.val |
| A -> M        | A.val = M.val          |
| M -> M1 * N   | M.val = M1.val * N.val |
| M -> N        | M.val = N.val          |
| N -> ( A )    | N.val = A.val          |
| N -> constant | N.val = constant.val   |

M是一个乘法表达式，A是一个加法表达式（分离 M, A 保证运算顺序正确），N是一个整体。

在规约时执行相应语义规则，最后`expr.val`即为表达式的值。



SDD有多种实现方式：建立注释语法分析树（或抽象语法树，可能是最常见的），使用一个递归下降的语法分析器，还有是构造一个SDT（语法制导的翻译方案）：在产生式中直接放入程序片段。

>  在两种情况下，SDT可以在语法分析过程中实现：
>
>  - 文法是LR的，且SDD为S属性：可以用LR。
>  - 文法是LL的，且SDD为L属性：可以用LL，也可用LR。

项目中构造的是L属性定义的SDT。将产生式内部的程序片段替换为一个空产生式`M->E`，将所有程序片段放到产生式尾部；在LR语法分析规约一个产生式时，执行它的语义规则。

> 理论上对加入的空产生式属性的计算是不合法的，但由于LR语法分析栈的特殊性，我们总可以合法的读到需要的属性。



**词法分析和语法分析分离的原因：**简化编译器设计，把复杂的任务拆分；提高编译器效率，分别设计专用于词法分析和语法分析的优化算法；提高可移植性，只需要在词法分析中考虑输入设备的特殊性。



**文法的例子：**

```
program -> compoundstmt
stmt -> ifstmt | whilestmt | assgstmt | compoundstmt
compoundstmt -> { stmts }
stmts -> stmt stmts | E
ifstmt -> if ( boolexpr ) then stmt else stmt
whilestmt -> while ( boolexpr ) stmt
assgstmt -> ID = arithexpr ;
boolexpr -> arithexpr boolop arithexpr
boolop -> < | > | <= | >= | == | !=
arithexpr -> multexpr arithexprprime
arithexprprime -> + multexpr arithexprprime | - multexpr arithexprprime | E
multexpr -> simpleexpr multexprprime
multexprprime -> * simpleexpr multexprprime | / simpleexpr multexprprime | E
simpleexpr -> ID | NUM | ( arithexpr )
```

这个是一个用大括号括起来的程序，包含if、while、赋值和语句块。

arithexpr是算术表达式的主体，arithexprprime是算术表达式的各部分。

## MapReduce

**MapReduce**

是一个分布式执行任务的通用模型，将所有的计算拆分成基础的 map 和 reduce 逻辑，降低分布式实现的复杂度。程序员只需要提供 map 和 reduce 的实现就可以了。
但也因为模型太简单，map, reduce 的具体实现可能非常难。任务在 map 时需要能分成多个独立的块，在 shuffle 时能按某个 key 分类，在 reduce 时能对分类的结果统一处理。本质还是自己写了一个并行程序，但 map reduce 帮我们实现了中间的同步机制。
一个任务还可能要进行多次 map reduce 来完成，而且性能也不是很好。
后来的框架都是基于 DAG 去做分布式计算，效率会更高，如 Spark。

一个Map或Reduce任务，称作task；所有task，称作job。
一个worker通常执行多个task（运行多个进程）。
Reduce Worker通过RPC处理Map Worker输出的数据，然后发送最终结果。总是在本地运行，所以通信代价不高。

**Map**
map 将所有数据转换为键值对，并根据 key 将数据按一定规则进行切分，在之后分配给不同的 reduce worker（这部分可以是 shuffle 的任务。map 也可以将输出分别存到 r 个mr-X-Y文件中。X为map处理者，Y为reduce处理者）。

**shuffle**
对于某一个 key，包含该 key 的所有键值对要被分配给同一个 reduce worker。但它们可能分散在不同 map worker 上。
shuffle 过程将同一个 key 在不同机器上的数据汇集到一起，转移到特定的 reduce worker。

可以在 map 中将键值对按 key 排序，提高此处的效率。

**Reduce**
reduce worker 每次处理一个 key 和它的 value 集合，得到某个 key 的计算结果。

> 概括来说，map 将自己的数据依次放入指定函数（如将数据分配到指定组），返回结果。
> reduce 作用到一个序列，首先将两个元素进行运算，然后将生成的元素和下一个元素进行运算，直到只剩一个元素，返回。

**优化**

很重要的一点是，数据是如何切分、分给不同 worker 的。如果切分函数设计的不好，会导致整个系统被少量没有完成的 worker 拖慢进程。
除了切分不均匀外，worker 长时间执行也可能是因为机器除了问题，所以需要定期检查 worker，如果一个 worker 很久没完成任务或响应，可将任务分给其它 worker。

有时候需要一个 master worker 来协调整个过程、分配 worker。

**应用**

word count，建立倒排索引，求和，排序

### 过程概括

1. 实现 master 与 worker 的通信。使用的是项目提供的 RPC 函数。
2. master 给 worker 分发 map 任务。通过 channel 分发。
3. map worker 获取并执行任务：将数据按照 key 划分到不同文件。
4. master 监视任务：当所有任务完成时，进入 reduce 阶段，分发 reduce 任务；当检测到任务超时时，将其重新分配。
5. reduce worker 获取并执行任务：将数据按 key 分组，然后每次处理一个key和它的value集合。

注意：

1. `-race`检测可能的冲突。

### 一些问题

**锁**

根据锁的种类和常用地方的不同，使用了多个锁，而不是只用一个`rf.mu`：

```go
{
	state   State
	stateMu deadlock.Mutex // 保护对状态的访问。正常情况下节点就是 follower 或 leader（更新日志），不需要访问 state。但是用处也不大，用到 state 的时候也要用其他字段

	// persistent state on all servers
	currentTerm int
	votedFor    int
	logs        []LogEntry
	perStateMu  deadlock.Mutex // 保护节点的持久化信息：persistent state(currentTerm, votedFor and logs)
	// Be careful that stateMu MUST lock before perStateMu and unlock after perStateMu if used to avoid deadlock.

	// volatile state on leaders
	nextIndex  []int          // index of the next log entry to send to tha server (initialized to leader last log index+1)
	matchIndex []int          // index of highest log entry known to be replicated on server (initialized to 0, increases monotonically)
	leaderMu   deadlock.Mutex // 只用于 leader 保存其他节点的状态
	
    voteMu deadlock.Mutex // 发起投票 tryElection() 与 给出投票 RequestVote() 不能同时执行，否则会出现既给自己投票、自己成为 candidate，又给别人投了票的情况。需要用锁包住两个函数。
	// voteMu MUST lock before mu, perStateMu and unlock after mu, perStateMu if used to avoid deadlock.
}
```

这样会非常麻烦，但应该性能更好？（一般都是靠`rf.mu`锁整个结构体）

为了避免死锁，规定了这四种锁 加锁和解锁的顺序，比如避免发起投票与给出投票同时执行的锁，必须在所有锁之前加。
但还是会有死锁，没调出来，虽然有死锁检测，但还是很难查，所以还是放弃了，用了两个锁（可以是 perStateMu 与 leaderMu）。
