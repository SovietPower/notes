**工厂模式**

> https://zhuanlan.zhihu.com/p/83535678
> https://zhuanlan.zhihu.com/p/83537599

简单来说，就是通过工厂类获取新对象（类似对象池），而不是自行构造并处理其初始化。

- 简单工厂模式：每次添加新类型，需要修改工厂类，违背了开闭法则。
- 工厂方式模式、抽象工厂模式：每次添加新类型，需要增加一个对应的具体工厂类，会增大代码量。
- 模板工厂模式：直接使用模板，避免每次增加新的工厂类（但如果实现不同还是不可避免？）。
- 注册模板类+单例工厂模板类：将工厂模式中产品的注册和获取抽象成两个类；使用单例模式的工厂类也更便于使用。扩展性高、封装度高。

**单例**

> 单例模式实质上就是创建全局对象，它们都是静态存储期对象。如同使用全局变量，在绝大多数场景下这并不是一个好做法，应当先检查一下设计是否合理（可见 *规则 - 尽量避免使用单例*）。
> （C++11及以后）使用单例模式时，最佳的实现只有一个：magic static。双重检查锁定是过时、复杂且错误的？原因参见 Scott Meyers 与 Andrei Alexandrescu 的 "C++ and the Perils of Double-Checked Locking"。
>
> 如果多个线程试图同时初始化同一静态局部变量，那么[初始化严格发生一次](https://zh.cppreference.com/w/cpp/language/storage_duration)。
>
> TODO：https://zhuanlan.zhihu.com/p/651173499
> 见维基 double-checked locking 有正确的写法。
>
> 单例的构造不会引入数据竞争。
> 如果需要控制/同步 static 实现的单例对象的析构顺序，只能使用`static auto p = new X{};`这种形式存储指针，然后在需要析构时`delete p`。但注意这很可能导致重复析构或没有析构。
> 见 *guidelines - I.3*。

单例模式分为懒汉式和饥饿式。
懒汉式：延后初始化，只有在第一次访问单例时，才创建单例对象。在对象不存在时，可能会被多个线程同时访问，需要考虑线程安全。
饥饿式：程序开始运行时，就创建并初始化单例。访问单例不需要考虑线程安全（只会读）。
懒汉式只在必要时创建，可能节省内存空间。

单例类的特点：

- 无参构造和析构函数为 private，构造、赋值（共4个）均为 private 或 delete，避免外部操作。
    （其实只需要禁用拷贝构造，禁用拷贝也会禁用移动？）
- 类中有一个获取静态实例的静态方法：包含成员`static Singleton* instance;`，包含方法`static Singleton* getInstance()`。

```cpp
class Singleton
{
private:
	static Singleton* instance;
	Singleton() = default;
	~Singleton() = default;
	Singleton(const Singleton&) = delete;
	Singleton& operator=(const Singleton&) = delete;
public:
	static Singleton& getInstance();
};
```

**单例是唯一的吗** / **单例/全局对象在跨 dll 中使用可能出现问题**

单例是一个静态存储期对象，与普通的全局对象没什么不同，所以只需考虑一个全局对象是否唯一。
因为不在头文件中定义的单例不能在编译单元间共用，所以只考虑在头文件中的情况。

定义一个静态存储期对象时，编译器会在当前翻译单元的符号表中创建该对象的符号，并在静态存储区为该对象预留空间，并进行静态初始化。
通过同一头文件使用的全局 static 对象，在每个翻译单元都拥有不同的地址；但对于 inline 的全局对象，在链接时编译器只会选择一个符号作为最终可执行文件使用的符号，该符号是唯一的。

在头文件中定义变量时，为了避免 ODR，需要用 inline 修饰变量。在 C++17 前不能将变量标为 inline，可以将变量作为 inline 函数的静态局部对象来实现同样的效果。
因为在类内定义的函数是 inline 的，所以单例对象会有 inline 修饰，理论上是唯一的。

但是，这只在 ELF 文件中满足：ELF 不区分符号来自哪个 .so，遇到多个 inline 对象时，先加载哪个 so 就用哪个，相当于将所有符号合并成了一个。编译器默认会导出 .so 的所有符号。

在 PE 文件中不满足：编译构建 dll 时，需要给符号显式声明`__declspec(dllexport)`，否则动态库不会导出该符号。
PE 文件的符号表中记录了每个符号会从哪些 dll 中导出(?)，如果某个 dll 导出了符号 a，那么在该 dll 内一定会使用自己的符号 a，不会使用其它 dll 的 a。因此如果多个 dll 同时导出符号 a，那么它们会各自拥有一个 a，链接器不会合并该符号。

解决方式：需要确保有且仅有一个 dll 导出了该符号，才能供其它编译单元使用并且保证唯一。
通常不太好做，但可以用一个 dll 来管理所有单例对象，仅由它负责导出，其它编译单元都使用 dllimport。
可以见[这里](https://zhuanlan.zhihu.com/p/696878184)和[这里](https://www.zhihu.com/question/425920019)。

> 可以用 CRTP 继承单例类来实现单例。直接用另一个无关的模板单例类可能不直观。

**单例不唯一的影响**

[type_info](https://zh.cppreference.com/w/cpp/types/type_info) 就是一个单例，它的构造函数和 operator = 都被弃值，只能通过 typeid 获取。
因为 dll 的问题，它在程序中也可能不是唯一的，所以标准库对 operator == 的实现通常为：

```pp
#if _PLATFORM_SUPPORTS_UNIQUE_TYPEINFO
    bool operator ==(const type_info& __rhs) const {
      return __mangled_name == __rhs.__mangled_name;
    }
#else
    bool operator ==(const type_info& __rhs) const {
      return __mangled_name == __rhs.__mangled_name ||
             strcmp(__mangled_name, __rhs.__mangled_name) == 0;
    }
#endif
```

`__mangled_name`是指向类型名称字符串的指针，当平台可以保证单例 type_info 唯一时，只需要比较两个指针的地址；否则需要比较内容。

二次检查+锁+写屏障。（但最简单的还是函数内静态变量）

由于指定重排和缓存的不一致性，创建单例后需要通过写屏障保证：1. 单例的创建一定在后面的其他语句（准确来说是写）执行之前完成。2. 写屏障之前的写入，即单例的创建，一定会提交到共享缓存或内存中，对其它 CPU 立即可见。

```cpp
inline void MemoryWriteBarrier() {
	__asm__ __volatile__("sfence":::"memory");
}

template<typename T>
class Singleton {
public:
	static T& GetInstance()
	{
		if (destroyed) {
			puts("Singleton has been destroyed.");
			abort();
		}
		if (instance == nullptr) {
			std::lock_guard<std::mutex> lock(mu);
			if (instance == nullptr) {
				T* p = new T();
				MemoryWriteBarrier();
				instance = p;
				atexit(Destroy); // in stdlib.h
			}
		}
		return *instance;
	}
	// 可用于带参初始化
	static T& GetInstance(const T& t)
	{
		if (destroyed)
		{
			puts("Singleton has been destroyed.");
			abort();
		}
		if (instance == nullptr)
		{
			std::lock_guard<std::mutex> lock(mu);
			if (instance == nullptr)
			{
				T* p = new T(t);
				MemoryWriteBarrier();
				instance = p;
				atexit(Destroy); // in stdlib.h
			}
		}
		return *instance;
	}

	static bool IsAlive() {
		return instance != nullptr;
	}

private:
	Singleton() {};
	~Singleton() {};
	// 禁用拷贝
	Singleton(const Singleton&);
	Singleton& operator =(const Singleton&);

	static void Destroy() {
		destroyed = true;
		if (instance) {
			delete instance;
			instance = nullptr;
		}
	}

	static std::mutex mu;
	static T* volatile instance; // 注意 volatile 要写在后面！
	static bool volatile destroyed;
};

template<typename T>
std::mutex Singleton<T>::mu;
template<typename T>
T* volatile Singleton<T>::instance = nullptr;
template<typename T>
bool volatile Singleton<T>::destroyed = false;
```

**单例模式 - 懒汉式**

如果不需要考虑线程安全，3行就可以实现：

```cpp
static Singleton* instance;
static Singleton* getInstance()
{
    if(instance == nullptr)
        instance = new Singleton();
    return instance;
}
```

注意在类外部初始化：`Singleton* Singleton::instance = nullptr;`。

但是会出现内存泄露的问题？（但后面两者都是在程序结束时才释放单例对象，不加也没有任何区别）
可以将 instance 改为`unique_ptr<Singleton> instance;`？
或在内部，定义一个专门用于析构 instance 的内部类：

```cpp
private:
	class Deleter {
	public:
		~Deleter() {
			if(Singleton::instance != nullptr)
				delete Singleton::instance;
		}
	};
	static Deleter deleter;
```

在程序结束、deleter 被释放时，单例对象会被释放。

为了保证线程安全，需要在第一次创建时加锁：

```cpp
static Singleton* getInstance()
{
    if(instance == nullptr) // R
    {
        std::lock_guard<std::mutex> lock(mu);
        if (instance == nullptr)
            instance = new Singleton(); // W
    }
    return instance;
}
```

事实上，加锁的代码仍然不够安全，因为 new 语句分为三步：申请空间，初始化空间，返回指针。由于指令重排，返回指针可能发生在初始化之前，导致 instance 在非 null 的时候，它指向的对象依旧可能没有构造完成。
要保证加锁线程的所有写，都要立即对其它线程生效？可以通过内存屏障完成，比如将`instance`改为`atomic<Singleton*>`类型的原子变量，使用 seq\_cst 顺序同步代码中的 R 与 W，可以保证读与写正确的同步。

> 因为原子读有性能损失，可以用临时变量减少一次内存读。注意访问 instance 时修改它。
>
> 锁对范围内的资源有 acquire/release 语义，即第一次加锁期间写入的对象，会对第二次加锁的线程可见。但对不加锁的线程不生效。

C++11 中，能保证函数的局部静态对象，只会在第一次访问函数时被初始化，并且是线程安全的。
所以直接将 instance 改为函数的静态对象即可（注意此时返回的是引用）：

```cpp
static Singleton& getInstance()
{
    static Singleton instance;
    return instance;
}
```

> static 对象初始化的时间：
> 对于非局部静态对象 (non-local static)，初始化发生在 main 函数执行前，可以认为是程序执行之前，所以肯定是安全的（此时只有1个主线程）。但不同非局部静态对象之间的初始化顺序，是随机的。
> 对于局部静态对象 (local static)，初始化发生在第一次进入该函数、且第一次遇到对象定义语句时。C++11 保证局部静态对象的创建是线程安全的，其它线程在初始化位置可能等待。
> 初始化语句只执行一次。

**Actor 模型**

Actor 模型是一个通用的并发编程模型。

Actor 模式本质就是将不同的任务分配给不同的 actor 执行，它们不涉及共享资源访问，可以避免锁等同步机制。
Redis 使用多个不同实例代替多线程实现并行，就是这种模式。
