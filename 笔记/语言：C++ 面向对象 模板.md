## 面向对象

**概念：各种函数**

成员函数就是某一类内部的函数，也叫方法，不能独立存在。

方法重载（overload）：
重载函数的调用根据参数的个数、序列来确定（返回值任意），而虚函数依据对象确定。
如果是类中的成员函数，则只能重载自己类的。

方法重写（overwrite）：
发生在父类和子类之间，子类会屏蔽父类的同名函数。
当函数参数不同时，virtual 有无任意；当函数参数与父类相同时，基类函数必须没有 virtual，否则就是覆盖。

方法覆盖（override）：
针对虚函数，发生在父类和子类之间，参数要相同，且返回值类型、抛出异常类型要是父类同名函数的子集，该方法权限要大于等于父类同名函数的权限，该方法是否为 const 要与父类一致。

虚函数（virtual）：
虚函数是c++实现多态的手段，具体指的就是如果派生类中的方法为虚时，基类指针可以根据其指向的具体对象的类型而绝对调用哪个函数。如果方法不为虚，则用指针调用方法和直接用对象调用是没有区别的，基类指针必然调用基类方法，而派生类指针必然调用派生类方法。
一个基类类型的指针，可以通过引用指向一个派生类，如`class B: public A ; B b; A *a = &b;`，A调用虚函数时，会调用B的；否则调用的就是A的。
静态方法不能是虚函数，因为虚函数需要通过一个实例的虚函数表访问，但静态方法属于类不属于实例本身。

友元函数（friend）：
定义在类外部的普通函数或类，但它需要在类体内进行说明。
**友元不是成员函数**，但是它可以访问类中的私有成员。
在类之间，友元关系是单向的，且不能传递。

析构函数：
格式：`~类名() {}` 没有参数列表、返回值类型及返回值。

**析构函数的调用时间**

对于局部变量，在函数返回时析构。
对于静态变量和全局变量，谁先构造的，谁就后析构。全局变量的初始化在主函数运行前，静态变量的初始化在第一次遇到该静态变量的声明时。
对于`new`创建的变量，在程序结束后系统也不会回收，需要手动delete析构。

**继承**
子类构造时，也会调用父类的默认构造函数。
父类析构时，不会调用子类的析构函数。如：B 继承 A，`A = new B()`，`~A()`不会调用`~B()`。

1. 子类中所有的构造函数都会默认访问父类中的空参数的构造函数，因为每一个子类构造内第一行都有默认的语句super()；
2. 若父类中没有空参数的构造函数，那么子类的构造函数内，必须通过super语句指定要访问的父类中的构造函数；
3. 若子类构造函数中用this来指定调用子类自己的构造函数，那么被调用的构造函数也一样会访问父类中的构造函数。

如果 B 继承 A，某函数形参类型为 A，则传入 B 会隐式转换为 A 类型。

**调用基类的方法或成员**

```cpp
struct B {
    B(): x{0} {}
    void f() {}
    int x;
};
struct D: B {
    D(): B() {}

    // using B::B; // 能直接调继承的方法或成员，但也可声明
    void f() {
        B();

        x = 1;
        B::x = 2;
        B::f();
    }
    int x; // 可直接定义与基类同名的成员。此时访问基类需要限定符
};
```

**虚函数**

> 见 *虚函数的效率*。
>
> 当使用基类的指针或引用处理派生类时，对被覆盖的虚函数的调用，将会调用在派生类中定义的行为。这种函数调用被称为虚函数调用或虚调用。
> 虚调用在使用有限定名字查找（即`...::function`）时被抑制（此时不是虚调用）。
>
> 注意，只有基类声明函数为 virtual，才在使用该基类时有多态效果。
> 如果子类的函数 f 与基类的同名虚函数 f 的签名不同，那么子类的 f 会隐藏基类的 f，而非覆盖 (override)，它们之间不是虚函数关系，没有多态效果（使用基类就会调基类的 f，使用子类才会调子类的 f）。
>
> 当子类调用继承自基类的某个函数 f 时（f 没有在子类中定义，可以非虚），如果 f 调用了虚函数 g，也有多态效果：如果子类覆盖了 g 则调用子类的实现，没有则调用父类的实现。毕竟只要有子类的虚表就能多态。
>
> 在构造函数或析构函数中调用虚函数时，不会考虑它的派生类中覆盖的虚函数，即使这个构造或析构是通过派生类间接调用的。毕竟在调用基类的构造时，派生类还没有被构造；在调用基类的析构时，派生类已经被析构了。

[虚函数](https://zh.cppreference.com/w/cpp/language/virtual)用来实现多态，基类指针和引用可以指向一个子类对象，并调用该子类实现的虚函数，而不是基类实现的函数。
父类中定义的`virtual`函数，子类可以继承使用（不声明或不提供定义），也可进行覆盖 (override)。
使用基类指针指向一个子类对象，调用的非虚函数为指针类的函数，调用的虚函数为实际指向对象类的函数（如果该类覆盖了该虚函数）。
非虚函数取决于指针类型，虚函数取决于对象类型。所以使用的虚函数是在运行时确定的。

如果一个类有虚函数，就会有相应的虚函数表。继承的子类会有一个新的虚函数表，其中重写的虚函数指向新的地址，没重写的虚函数指向父类中函数的地址。
当一个类有虚函数时，该类的对象中就会多存一个指针 vptr，指向它的虚函数表 vtbl，虚函数表中保存了每个虚函数所在的地址。
比如有基类 A 和子类 B，A 定义了虚函数 f1, f2，B 覆盖了 f1，则 B 的虚函数表中，f1 会指向 B 实现的 f1，f2 会指向 A 实现的 f2。

所以对于有虚函数的类的对象，其起始位置会多一个指针，多占用 4 或 8 字节。
此外调用的虚函数需要在运行时 检查虚函数表确定，而不能在编译时就确定，效率要低一些。
所以如果没有必要 或不作为基类，不要声明虚函数。

用于多态的基类，其析构函数应定义为虚函数（并在子类中实现对应的析构函数）。
因为使用基类指针指向子类时，如果析构函数不是虚函数，就会调用基类的析构函数，导致子类的对象可能无法正确释放，**产生内存泄露**。

> **构造函数可以是虚函数吗？**
> 构造函数不能也不需要是虚函数。在执行构造函数前，对象的内存都没初始化，自然也没有虚表。
> 而且 X 的构造函数就是要实际生成一个 X 对象，用虚函数没有意义。
>
> **模板函数可以是虚函数吗？**
> 不能。通常实现中，虚表的每个条目指向一个虚函数，同一基类的不同子类，其虚表条目的数量、每个虚函数的偏移量（在哪里）都是相同的。
> 但当存在多个编译单元时，由于某些实例化是在链接时发生的，因此编译器没法在编译时知道它们会实例化多少虚函数、产生什么样布局的虚表，只能按照自己编译单元内部的使用情况构造虚表。因此同一基类的子类，在不同编译单元中产生使用的虚表，可能是不同的，没法正确调用某个虚函数。
> （但理论上可以在链接时合并不同翻译单元的虚表，让它们有一致的虚表）
>
> 设 A 是 B 的基类，`A* x = new B()`，则称 x 的静态类型 (static type) 为其直接指向的基类 A，动态类型 (dynamic type) 为其实际类型 B。
> 注意，**如果虚函数的参数有默认值，则默认值取决于 x 的静态类型 A 中的定义**，即使虚函数的实现确实取决于 B。**因为默认实参需要在编译期确定，而动态类型只能在运行时确定。**
> 如：父类 A 有虚函数`virtual void f(int a = 1)`，子类 B 覆盖虚函数`virtual void f(int a = 2)`，则`x->f()`会调用`B::f()`，但参数的默认值来自`A::f()`，是 1。
>
> 调用虚函数时，访问权限取决于调用虚函数的对象的静态类型，而非动态类型。即对于`A* p`，只要 A 中的虚函数 f 声明是 public，就可调用`p->f()`，即使子类用 private f 覆盖。
>
> 多态要通过指针或引用实现。`Base x`只会被视作 Base 类型。

**虚函数表存放的位置 / 虚函数的原理**

每个有虚函数的类会有一个虚函数表。因为类决定了虚函数表，所以同一个类的多个对象可以共享虚函数表，只需要用 vptr 指向（没有指针，就不好找该类的虚函数表位置，除非在全局维护）。
虚函数表位于内存中的只读数据段 (.rodata)，也就是常量所在的位置。条目指向代码段。
每个有虚函数的对象会保存一个虚表指针，指向该位置。

在派生类中，对于没有重写的虚函数，在虚函数表中的值与基类一致，指向基类的虚函数；重写的虚函数，在虚函数表中的值指向派生类的虚函数。
如果派生类定义了新的虚函数，也在虚函数表的后面加入它的地址。
对于基类的任意一个虚函数，它在虚函数表中的位置，和在派生类的虚函数表中的位置是相同的。所以运行时，只需要读取 vptr 指向的虚函数表中的对应位置，然后调用那个函数，就能实现动态绑定。

多重继承中，对每个（有虚函数的）基类，都会有一个虚指针。虚函数表的意义一样。
假设类 C 依次继承了 A,B，则 C 对象的内存布局依次为：A 的虚函数表指针，A 的数据成员，B 的虚函数表指针，B 的数据成员，C 的数据成员。
当创建一个 B 类型指针指向 C 时，指针直接指向的位置，正是 B 的虚指针所在的位置。
对于派生类定义的新虚函数，放在第一个虚指针指向的虚函数表后面。（第一个继承的类也称为主基类？）

> 我们说的虚函数表，其实只是虚表的一部分。虚指针是指向虚表的。
> 虚表依次包含：offset to top (到对象起始地址的偏移值)，RTTI information (指向 RTTI 信息)，virtual function pointers (虚函数表，保存虚函数指针)。如果有虚继承，则前面还要加上：virtual call offsets, virtual base offsets。
>
> offset to top 一般为0，只有在多继承时会不为0。比如：C 继承了 A,B，则在 D 的第二个的虚表中（对应B），offset 为 -sizeof(A)。
>
> RTTI (Run-Time Type Identification) 是运行时类型识别，能够在运行时获得对象的类型信息。
> 使得基类的指针或引用，能够知道它指向的对象的实际派生类型，从而能实现 typeid 和 dynamic_cast 两个操作。
> `typeid(x).name()`会返回 x 的类型 (string)，但该类型信息非常简单。
>
> 虽然我们可能能直接访问虚表，但这是 UB：
>
> ```cpp
> struct A {
> 	virtual void f(int x) { printf("A::f(%d)\n", x); }
> };
>
> A* a = new A{};
> int*** vptr = reinterpret_cast<int***>(a);
> void(*p)(int) = reinterpret_cast<void(*)(int)>(vptr[0][0]);
> p(3); // 可能输出 A::f(0)
> ```

一个类型的指针，也可以强制转换成另一种没有继承关系的类型，并调用新类型的虚函数。因为它们的 vptr 位置相同，运行时只是去指向 vptr 指向的表的某个位置指向的代码，只是可能不正确。

通过选项`-fdump-lang-class`可以查看类的布局和虚表内容。

**多态实现**

> 侵入式多态：通过在对象中嵌入虚表指针或其它元数据，接口处只需使用普通对象指针或引用达成的多态。C++ 的虚表就是这样。
> 非侵入式多态：对象内不包含元数据，但接口处要使用胖指针（对象指针 和 虚表指针或元数据 构成的 pair）实现的多态。go 的 interface 就是这样？

**虚函数的效率**

具体调用哪个虚函数，通常是在运行时才能确定的。因此，与编译时就能确定地址的普通函数相比：

- 多两次间接寻址/解引用（一次访问虚指针得到虚表地址，一次访问虚表得到函数指针）。
- 更重要的是，虚函数通常无法预知实现，导致：
    - icache 无法预取导致 cache miss（对较差的 CPU 影响更大）。
    - 硬件很晚才能知道虚函数跳转的位置，可能导致分支预测失效（不能将后面的指令提前放入流水线）。
    - 外层函数无法利用调用函数的信息进行优化（这可能能大量减少汇编数量）。
    - 无法内联，多一点函数调用开销，无法执行某些优化（如常量传播）。而且内联是可以内联很多层函数的。

通常情况下影响不大。但当函数体本身开销较小、虚函数调用开销占比较大、频繁调用时，就会影响性能。比如大量的虚 getter/setter。

优化方向：

- 如果能编译期多态（有些子类实际在编译期就能确定类型），可以用 CRTP。
- 小函数的调用开销更高，尝试写成普通函数。
- 如果逻辑需要在类外部多次调用虚函数，考虑封装成一个虚函数，只调用一次。
- 依次调用多态数组的虚函数时，先将数组对象按类型排好序，利于 cache。
- 使用函数指针或 *函数 - function_ref* 替代类型已知的虚函数频繁调用。
- 利用 Profile-Guided Optimization (PGO) 和 JIT 也有机会将虚函数内联。

> clang 编译选项`-fstrict-vtable-pointers`允许做更激进的优化：Enable optimizations based on the strict rules for overwriting polymorphic C++ objects.
> 它向编译器保证，在对象的生命周期内，虚指针是固定不变的。这样对同一对象的多次虚函数调用，编译器就不需要多次对虚指针解引用了（每次调用少一次解引用）。
> 标准没有提供不经由销毁旧对象再重建新对象，就更改虚指针的方式。因此正确的代码是不会有问题的。但如果代码没有正确管理对象的生存期，就可能导致出错。

**虚函数的覆盖与重写**

PS：好像并不区分这两个概念。

覆盖发生在父类和子类之间，参数要相同，且返回值类型、抛出异常类型要是被重写父类函数的子集，该方法权限要大于等于被重写父类函数的权限，该方法是否为 const 要与父类一致。
重写也发生在父类和子类之间，但子类会屏蔽父类的同名函数。
当同名函数不满足覆盖条件时，就发生重写，会创建一个新的函数，而不是覆盖原函数。
例：

```cpp
class Base {
public:
    virtual void Show(int x); // 虚函数
};

class Derived : public Base {
public:
    virtual void Show(double x); // 参数列表不一样，新的虚函数
    virtual void Show(int x) const; // const 属性不一样，新的虚函数
};
```

有时我们想覆盖，但意外地进行了重写。为避免该错误，C++11 引入了两个关键字：

- [override](https://zh.cppreference.com/w/cpp/language/override)：确保该函数为虚函数，并覆盖某个基类中的虚函数。
  子类覆盖的虚函数也是虚函数，只要注明 override，写不写 virtual 就都行。
- [final](https://zh.cppreference.com/w/cpp/language/final)：确保该函数为虚函数，并且不会被派生类覆盖。
    阻止类的进一步派生 和 虚函数的进一步重写。

应该尽可能使用这两个关键字来减少错误。
子类不应该去覆盖父类的普通成员函数。如果需要，将父类的普通成员函数修改为虚函数。

**抽象类**

**定义了纯虚函数的类为[抽象类 (abstract class)](https://zh.cppreference.com/w/cpp/language/abstract_class)**，不能创建抽象类的实例，只能创建它的派生类的实例（如果也不是抽象类）。
抽象类定义了子类的各个接口，可以为子类创建规范。

**纯虚函数**

虚函数可以提供实现，也可以不实现（但这样也没法调用，没意义）。
纯虚函数表明基类没有给出该函数的具体实现，需要继承该类的子类实现，是一个接口或规范。
析构函数可以是虚函数或纯虚函数（并且应该是虚函数），但构造函数不能加 virtual，因为虚函数要通过对象中的 vtable 找到函数起始地址，但在构造函数执行前对象都不存在，也就没有 vtable，没法调用虚的构造函数。

在虚函数声明的后面加纯说明符`= 0`即可定义纯虚函数：`virtual void func() = 0`。

可以在类外给出纯虚函数的定义，只要纯说明符`= 0`和函数定义不同时出现在声明中即可。
即使给出了定义，类也还是抽象类、不能创建实例，但是抽象类和派生类的成员函数可以通过有限定的函数标识 调用抽象基类的纯虚函数（这样是非虚调用，如：`Base::f()`）。
在抽象类的构造/析构函数中虚调用纯虚函数（不加限定的直接调用）是 UB。

**继承与组合**

（公共）继承表示 is-a 关系，组合（与非公共继承）表示 has-a 关系。继承能实现多态、代码复用，但这些功能都能通过 组合+接口+委托实现。
如果不是 is-a 的关系（不想让 A 对象能作为 B 对象使用），或继承层次过深、过复杂时，会影响到代码的可读性和可维护性，就应使用组合而非继承。

一般来说，总可以将继承替换为组合：如果继承关系设计的不好，有些子类会继承并不应该实现的某些方法，就需要重写这些方法，让它们抛出异常（但很可能误调用）；可能会需要拆分功能，继承多个抽象类，导致逻辑很复杂。
例子见[这里](https://blog.csdn.net/A_art_xiang/article/details/127565300)。
继承没有组合灵活，组合可以使对象更自由的组合多个模块，而继承中（如果不想很麻烦）每个对象要么都有某个模块，要么都没有。

不过如果继承关系不复杂，用继承也没关系；继承还有两个特点：

- 能够重写父类的方法。
- （在 C++ 中）能使用空基类优化。

**继承方式**

B 在继承 A 时，可以在 A 前面加修饰符表示继承方式。class 默认 private，struct 默认 public。

- private/私有继承：`class Derive: (private) Base`。
  基类的 protected 和 public 成员会在派生类中变为 private，可以在派生类内部使用，但不能在类外部或派生类的子类中使用。
- public/公共继承：`class Derive: public Base`。
  基类的成员的访问权限不变。
  **只有 public 继承表示 is-a 关系**，表示子类是父类的子集，子类对象一定是父类对象，能够在父类对象上做的行为也能在子类上做。而其它继承方式只是为了继承属性或方法。
- protected/保护继承：`class Derive: protected Base`。
  基类的 protected 和 public 成员会在派生类中变为 protected，可以在派生类和子类中使用，但不能在类外部使用。

基类的 private 成员，不管怎样派生类都无法访问。
（但注意，一个类的 private 成员，在该类内部就可以调用，即使不是该对象！**C++ 的访问修饰符以类为单位，而不是以对象为单位。**同一个类内自然算内部代码，不设权限也不会破坏封装、导致错误。比如拷贝构造函数中，可以访问参数的私有变量）

公共继承最常用，但对于 C++，不是所有继承都是 is-a 的关系，继承可以只是继承属性或方法。
这其实与面向对象的设计原则不符（子类与父类就是 is-a 关系），Java, C# 中都只有公共继承，但 C++ 并不是严格的面向对象，允许其它范式、更自由或更看重效率？
只有公共继承的派生类，才可以使用基类指针指向该派生类（否则会`A is an accessible base of B`）。
非公共继承不允许派生类被当做基类使用，能够隐藏基类的实现细节。

私有/保护继承实现的功能（继承），与在类中定义私有/保护对象类似（组合）（私有继承类 A，与定义私有类 A 对象类似）。
但是，继承能够使用空基类优化，私有继承空的类不会占大小，而组合使用私有变量至少还要 1 字节。

非公共继承在业务中很不常用，但在模板库这类通用、安全的代码中有用（可能是因为空基类在业务中也不常用）。

> https://isocpp.org/wiki/faq/private-inheritance
> 私有继承其实就是组合的另一种形式，通常都使用组合，如果需要继承则使用私有继承（比如使用虚函数）。

**赋值兼容原则/类型兼容原则**

（仅限公共继承）
派生类的对象或指针可以赋值给基类对象或指针；派生类可以初始化基类的引用。反之不可。
即派生类对象可以作为基类对象使用，但此时只能访问基类中的成员（但可以用自己的虚函数）（切片现象）。
所以任何基类出现的地方，都可以用派生类去代替，反之不可。

但有些场景下，将子类当作父类用会不符合预期。
例：见 *面向对象 - 构造/析构函数*，即`base = std::move(derived)`。
使用 protected/private 继承使基类成为不可访问的基类（派生类不能再被作为基类使用），就可能在不当使用时，产生编译错误，更早发现问题。

**多重继承**

一个类可以继承多个基类。这种类可能有多个虚函数表（即多个虚指针）。
设 D 继承的类为 B1, B2, ..., Bn，则 D 的对象的内存布局，依次为：B1 的虚函数指针与对象、B2 的虚函数指针与对象、...、Bn 的虚函数指针与对象、D 的对象。

多继承中，不同基类的起始地址，并不都是派生类的起始地址，所以对于一个派生类 D，要想安全的进行基类指针 \*B1, \*B2 之间的转换，必须通过 dynamic_cast。
例：

```cpp
class BaseA
{
public:
	int x;
    virtual void funcA() { cout << "BaseA::funcA()" << endl;
};

class BaseB
{
public:
    virtual void funcB() { cout << "BaseB::funcB()" << endl; }
};

class Derived : public BaseA, public BaseB {};

int main()
{
    Derived d;
    BaseA* pa = &d;
    BaseB* pb = &d;
    BaseB* pbe = (BaseB*)pa;    // error！只是简单进行了类型转换，但pa的地址不应是pb的地址
    BaseB* pbc = dynamic_cast<BaseB*>(pa);

    cout << "sizeof(d) = " << sizeof(d) << endl;

    cout << "Using pa to call funcA()..." << endl;
    pa->funcA(); // correct

    cout << "Using pb to call funcB()..." << endl;
    pb->funcB(); // correct

    cout << "Using pbe to call funcB()..." << endl;
    pbe->funcB(); // error! 输出 BaseA::funcA()

    cout << "Using pbc to call funcB()..." << endl;
    pbc->funcB(); // correct

    cout << "pa = " << pa << endl; // BaseA 的地址：n
    cout << "pb = " << pb << endl; // BaseB 的地址：n+16 (虚函数+int)
    cout << "pbe = " << pbe << endl; // error！仍为 BaseA 的地址：n
    cout << "pbc = " << pbc << endl; // BaseB 的地址：n+16
}
```

继承和多继承的构造函数，都可以在派生类调用构造函数时，在初始化列表中调用：`D(形参列表): A(实参列表), B(实参列表), C(实参列表)`。派生类的将在最后调用。
注意，与初始化列表中的变量一致，多继承中构造函数被调用的顺序，取决于继承时的顺序，与列表中的顺序无关。
析构函数会自动调用，与构造顺序相反。

**多重继承中的二义性**

继承的多个基类中的同名变量可能产生二义性。
对此，需要用作用域运算符`::`来限定所访问的成员是属于哪一个基类的：`ClassName::VariableName`。也因此任一基类只能被直接继承一次，否则无法处理冲突。
作用域运算符不能嵌套（连续使用），如：不能`A::B::x`，如果需要只能在 A 中加一个函数`getBx()`。

派生类中的成员可以与基类中的成员同名。如果没有指定作用域，则默认使用当前类中的。

**虚基类**

在多继承中，如果一个类 D 继承的两个（多个同理）基类 B 和 C 继承了相同的基类 A，则默认情况下 D 会继承 A 两次，拥有 A 的两份数据，使用时要通过 B 和 C + 作用域运算符进行区分。
如果想让公共的基类只被继承一次（菱形继承），则在继承它时，需要使用 virtual 指定它为虚基类（B 和 C 必须都声明虚基类）。

```cpp
/*
默认情况：
A<-B<-D->C->A
D 继承 A 两次
*/
/*
虚继承的情况：
D->(B,C)->A
D 只继承 A 一次
*/
class A {};
class B: virtual public A {};
class C: virtual public A {};
class D: public B, public C {};
```

指定虚基类的格式：`class Derived: virtual (public/private/protected) Base`。

虚基类的构造函数，会优先于非虚基类的构造函数执行。析构函数执行会更晚。

**虚继承**

当派生类继承了一个**或多个**虚基类时，或说虚继承了一个或多个类时，会创建**一个**虚基类指针 vbptr，虚基类指针指向虚基类表，虚基类表中保存了 虚基类的数据对象，距离 vbptr 地址的偏移量。
虚基表指针可以视为一个数据成员，会在继承时也继承下来。
例（图见[这里](https://blog.csdn.net/yi_chengyu/article/details/120856687)）：

```cpp
class A {int a;};
class B {int b;};
class C: virtual public A, virtual public B {int c;};
则 C 的内存布局为：
00: vbptr (C本身)
08: c
16: a (虚基类A)
24: b (虚基类B)
虚基类表为：
0: 0
1: 16 (= A的起始地址 - vbptr)
2: 24 (= B的起始地址 - vbptr)

class A {int a;};
class B: virtual public A {int b;};
class C: virtual public A {int c;};
class D: public B, public C {int d;};
则 D 的内存布局为：
00: vbptr1 (D正常的继承B，拥有B的数据成员)
08: b
16: vbptr2 (D正常的继承C，拥有C的数据成员)
24: c
32: d (D本身)
40: a (虚基类A)
虚基类表1为：
0: 0
1: 40 (= A的起始地址 - vbptr1)
虚基类表2为：
0: 0
1: 24 (= A的起始地址 - vbptr2)
```

如果有虚继承，虚基类指针会在该类的起始地址位置，然后才是虚函数表指针。但是前者会被当做普通对象继承，而后者不是？

**非静态成员初始化**

两种方法，具体见[这里](https://zh.cppreference.com/w/cpp/language/data_members)。

1. 在构造函数的成员初始化器列表中初始化。
2. 通过*默认成员初始化器*：在成员声明包含花括号或等号初始化器。如果在成员初始化器列表中初始化该成员，该初始化会被忽略。

```cpp
// 1.
struct S {
    int n;
    std::string s;
    S() : n(7) {} // 直接初始化 n，默认初始化 s
};
// 2.
struct S {
    int n = 7;
    std::string s{'a', 'b', 'c'};
    S() {} // 默认成员初始化器将复制初始化 n，列表初始化 s
};
```

**静态成员初始化**

[static](https://zh.cppreference.com/w/cpp/language/static) 只用于类静态数据成员的声明，不用于它的定义；可以将成员声明为不完整类型。变量的定义需要在类外部进行。
但 constexpr 或 inline 静态数据成员除外，因为它们在类内的声明就是定义。

> inline 和 constexpr 的静态数据成员，会在类内声明时同时被定义。
> **const 的静态数据成员可以在类内进行初始化，但这仍是声明，该成员不会定义**（特殊规则）；仅读取它的值仍然是正确的（有常量初始化器的 const 对象可用于常量表达式，会发生左值到右值转换），但如果要 odr 使用它，仍然需要在外部定义（此时的定义不需要加初始化器）。
>
> 需要在类外部定义的原因：
>
> - 因为 ODR。如果在头文件定义 static 数据成员，编译器会在编译期将它记录为一个强符号。当这个类在头文件中被引入到多个翻译单元后，就会产生多个名字相同的强符号，导致链接时重复定义。（**这与 static 变量或普通函数是一致的**）
>     因此像在头文件中声明函数、在源文件中定义函数一样，静态成员变量的声明与定义也要分开，除非像函数一样添加 inline（不过 C++17 才引入 inline 变量，可以用 inline 函数）或 constexpr（蕴函 inline）。
>
> const 静态成员特殊的原因：
>
> - 加上 const 后，如果文件里没有对它进行取地址、引用它等 odr use 行为，编译器可以根本不给它分配内存和符号，而是在汇编中直接用立即数代替这个 static const 变量。所以只要不 odr 使用它，就不会有错误（读它属于读编译期常量，所以不是 odr 使用）。
>     当对其进行取地址时，它一样会变成强符号；如果多个翻译单元都对其进行取地址，也会发生重复定义。
>
> 这和普通函数一样，普通函数是强符号，如果在头文件中定义也会重复定义。
> 在类内定义的成员函数是 inline 的，所以头文件中定义它们不会出现重复定义。
>
> 非 static 数据成员不会有问题，因为它们根本不会在编译期产生符号。

**友元 / 友元函数 / friend**

[友元声明](https://zh.cppreference.com/w/cpp/language/friend)出现在类内部，但不是成员函数，会向声明的函数或类授予当前类 private 和 protected 成员的访问权。
可以声明其它类的成员函数是友元（包括构造析构）。
友元声明的格式为：`friend 函数声明`、`friend 函数定义`（可以在声明一个函数是友元时同时给出它的定义）、`friend 类型`。

类和非成员函数的声明，不需要出现在它们的友元声明之前：如果一个名字的第一次出现是在友元声明中，会假定该名字在当前作用域是可见的。但成员函数不同，友元声明前，必须先知道该成员函数的声明。
友元声明不是常规的声明（只是影响权限），如果友元声明的类和非成员函数 没有在前面进行声明，则直到它们在类外进行声明前，它们都是不可直接使用的，即使它可能在类内进行定义（除非通过 ADL，见下 *hidden friend*）。

```cpp
struct X {
	friend void f() {
        // 可以在类内部定义，不影响结果
    }
	X() { f(); } // error: f 未声明
	void g();
};
// error
void X::g() { f(); }
void h() { f(); }

void f();
// ok
void X::g() { f(); }
```

友元不是类的成员，因此不受其区域访问控制级别 (public, ...) 的约束。

友元关系是单向的，且不能传递。

> 最常见的就是用于流输出私有成员：`friend ostream& operator<< (std::ostream &out, const T&)`。
>
> 如果定义友元运算符`operator++(int)`，则会被解释为`operator++(obj, int)`，而非`obj.operator++(int)`。
>
> 成员函数与友元函数有一点区别：成员函数的第一个参数必须是当前对象，不能通过隐式转换得到对象再调用。
> 比如：设类 A 有构造`A(int)`，则定义`A operator +(const A&)`后，可以使用`a1 + a2`、`a + 1`调用它得到 A，但不能使用`1 + 2`。而定义`friend A operator +(const A&, const A&)`后，可以使用`1 + 2`获得一个 A，1 与 2 都会隐式转换。
> 通常我们二元运算是对称的（比如 +、==），因此用 friend 声明成员的二元运算符更好。

**hidden friend**

> https://www.justsoftwaresolutions.co.uk/cplusplus/hidden-friends.html

友元函数的定义有两种方式：

- 在类内进行友元声明，同时在类外声明，然后在类内或类外定义：此时友元与正常函数一样，可以直接调用。
- 在类内进行友元声明，但不在类外声明，并直接在类内进行定义：此时是一个 hidden friend，它会被作为当前类命名空间内的成员，但不能被普通查找（不管是有作用域还是无作用域）找到，只能通过 ADL 在参数为当前类时找到。

隐藏与否就是看是否在类外有声明，与在哪定义无关。

```cpp
// #1 类外声明，类内友元声明并定义
struct A;
void f(A);
struct A {
    A(int) {} // 转换构造函数
	friend void f(A) {}
};

// ok，与普通函数类似
f(A{});
f(1);

// #2 类外声明并定义，类内仅友元声明
struct A {
    A(int) {}
	friend void f(A);
};
void f(A) {}

// ok，与普通函数类似
f(A{});
f(1);

// #3 没有类外声明，类内友元声明并定义
struct A {
    A(int) {}
	friend void f(A) {}
};

f(A{}); // ok，在 A 作用域内找到了 f
f(1); // error，当前作用域不存在 f，也无法通过实参 int 找到 f
```

另一个例子：

```cpp
struct A {
	int x;
    A(int x): x(x) {} // 转换构造函数
	friend bool operator ==(const A& l, const A& r) {
		return l.x == r.x;
	}
    // friend auto operator <=> (const A& l, const A& r) = default;
};
struct B {
	operator A() {
		return A{0};
	}
};

// ok
cout << (A{1} == A{2}) << '\n';
cout << (A{1} == B{}) << '\n';
// error：operator == 仅在 A 作用域内存在，除非实参包含 A，否则无法找到它
cout << (B{} == B{}) << '\n';

// ok：声明后不再隐藏
bool operator ==(const A& l, const A& r);
cout << (B{} == B{}) << '\n';
```

hidden friend 可以避免一些隐式转换带来的问题，减少错误；可以减少重载候选函数的数量，提高编译速度。
在编写 *定制点对象* 时很有用。

**空基类优化** / **(ebo)**

允许空的基类子对象大小为0（见[这里](https://zh.cppreference.com/w/cpp/language/ebo)）。

但是，如果**首个非静态数据成员**的类型，与继承的某个空基类的类型相同，或是继承的某个空基类的派生类，则无法使用空基类优化，因为两个同类型的基类子对象必须有不同的地址。
C++ 20 引入了`[[no_unique_address]]`，可能违反地址不同的约束，但能实现与空基类类似的优化。

当多重继承时，空基类的优化取决于编译器，对于 GCC，所有空基类均应用空基类优化，不分配任何空间。

**隐式生成的构造函数**

如果用户声明了某个函数，那么称这个函数是*用户声明*的 (user-declared)（与编译器的隐式声明相对）（注意 = delete 也是一个用户声明的函数）。
如果用户声明了某个函数，并且没有在首次声明时标记为 default 或 delete，那么它就是一个*用户提供*的函数 (user-provided)。

```cpp
struct A {
    // 两个都不是用户提供的，但都是用户声明的
	A() = default; // = default/delete 就是函数定义
    A(const A&) = delete;
};
struct B {
    // 两个都是用户提供的
    B(); // 后续可能会给出用户定义，也可能 = default，但编译器看不到
    B(const B&) {...}
};
B::B() = default;
```

当存在**用户声明**的复制构造、移动构造、复制赋值、移动赋值、析构函数中的任何一个时，编译器不会再为类生成隐式定义的移动构造函数（类似的表述是：声明这些函数会抑制移动构造的生成；声明这些函数会使默认的移动构造函数被隐式弃置）。

**默认构造函数**

创建类时，如果不自行实现以下函数，类会提供默认实现：构造函数、析构函数、拷贝构造函数、移动构造函数、拷贝赋值函数（operator =）、移动赋值函数。
`Node a = b`会调用拷贝构造，`a = b`会调用拷贝赋值 =。
子类的默认构造函数中，会调用基类的构造函数和各个成员的构造函数。

默认构造是无形参或所有形参都有默认实参的构造函数，用于进行*默认初始化*。只要类有默认构造函数，就是[可默认构造](https://zh.cppreference.com/w/cpp/types/is_default_constructible)的（默认初始化只需要类有默认构造）。
如果定义了任意构造函数，编译器就不会自动生成默认构造函数（但其它的还在，比如拷贝构造，见 *三五零法则*）。
但是一旦我们给出了默认构造函数，该类或结构体就不再是平凡的 (trivial)，就不是一个 POD 类型，会失去一些优点，比如 POD 可以直接使用 memset 完成初始化。
如果我们既想定义有参构造函数，又想使用编译器生成的（称为预置的）默认构造函数，来满足 POD 条件，可以使用 = default 标注构造函数，让编译器给出默认实现。

如果我们不允许类使用某个函数，即不想编译器默认实现某个函数，可以用 = delete 标注函数。
如：`A& operator=(const A&) = delete;`禁用拷贝构造函数；`A(double) = delete;`禁用以 double 为实参的构造函数。

**默认的无参/拷贝/移动构造函数就是调用每个成员的无参/拷贝/移动构造函数。**
注意，默认的拷贝/移动构造会考拷贝/移动所有成员，而空实现`{}`或实现后未在初始化列表中拷贝/移动，则不会。
部分内容见 *C++ - 左值和右值*。

> 注意，默认的拷贝/移动 构造/赋值非常简单，如果实现了自定义的拷贝/移动构造，基本就需要实现对应的拷贝/移动赋值（三五零法则）。
>
> 注意，基本类型（包括指针）move 后的值不变，所以如果一个类有指针成员 p，且使用默认生成的移动构造、在析构函数中释放 p，则会在 a = move(b) 中 double free p：1. 见上，类声明析构函数会使默认的移动构造函数被隐式弃置，所以这实际调用的是复制构造复制指针；2. 就算假设有默认的移动构造，p 移动后并不会被自动置为 nullptr。需要自己实现移动构造，或者用 unique_ptr，或者直接禁用拷贝/移动。
> 这也是三五零法则：如果需要析构释放资源，就应该定义拷贝/移动函数，因为默认生成的里面不会释放资源或避免被释放两次。

> [一个不符合该命名的规则](https://zhuanlan.zhihu.com/p/606088156)是：函数被显式和隐式弃置是有区别的：显式弃置的函数仍可参与重载决议，且优先级不受影响，可能会优于其它函数被选择（此时程序非良构）。（其实隐式弃置是指编译器不会再生成该函数，而非有了再弃；而显式弃置则是定义并禁用了，`= delete`就是函数体）
> 显式弃置就是使用`= delete`，隐式弃置包括很多，如：用户定义了析构函数导致移动构造与赋值弃置（见 *面向对象 - 三五零法则*），父类的移动构造被弃置导致子类的移动构造弃置。
>
> 如下面的代码中移动构造被**显式弃置**，但因为比复制构造更合适，所以会被选择，导致使用 deleted function（其实这样更合理，显式弃置就是不想使用该重载）。

>  A() = default;

> A f() {

**构造/析构函数**

构造函数和析构函数不会被继承，但是会在子类进行构造/析构时**自动调用**（自定义的构造函数也是）。
所以基类的构造与析构函数，需要声明为 public 或 protected 供子类调用。

用于多态的基类，其析构函数应定义为虚函数（并在子类中实现对应的析构函数）。
因为使用基类指针指向子类时，如果析构函数不是虚函数，就会调用基类的析构函数，导致子类的对象可能无法正确释放，**产生内存泄露**。

因为子类可以当做基类使用，在涉及析构、移动构造（即资源释放）时非常危险。析构函数可以定义为虚函数，但移动构造定义为虚函数也没用，比如：
B 继承自 A，两者有各自的移动构造函数。定义`A a; B b;`，执行`a = std::move(b);`，将总会调用 A 的移动构造函数（将 b 看做 A），可能导致 b 的成员无法正常或全部移动。
虽然 move 后的 b 不会（不应）再使用，但如果 B 的析构函数没考虑到这个问题，可能会出问题。

> 析构基类子对象后，不能保证不属于基类的派生类对象成员仍然存活。访问它们可能是 UB。
> 可能还要看：https://www.zhihu.com/question/628978219/answer/3309858519

**构造函数调用顺序 / 构造顺序**

> 当给出了构造函数的实现时，则是先执行基类和成员的默认构造函数（如果有默认成员初始化器，即声明时带初始化器，则按初始化器进行；如果有初始化列表，则按列表进行），然后执行构造函数体。
> 析构与此相反：当给出了析构函数的实现时，先执行析构函数体，再执行成员和基类的析构函数。
>
> 注意，在自定义的构造函数中（不管是默认构造还是自定义的拷贝/移动构造），如果没有在初始化列表中调用基类或非静态成员的构造函数（`D(): Base(), x() { ... }`），且它们没有默认成员初始化器，那么就会默认初始化它们（对类为调用默认构造，对 POD 就是不初始化）。之后再在构造函数体内赋值成员时，属于赋值 operator = 而非初始化，不会调用构造。不过默认定义的拷贝/移动构造会进行逐成员拷贝/移动。
> 拷贝或移动构造函数也是如此。因此如果想要在自定义的拷贝或移动构造函数中，拷贝或移动构造某个数据成员或基类，需要自己在初始化列表中写出，否则都会被默认初始化。比如：`struct B: A { B(B&& b): A(std::move(b)) {} };`。
>
> 注意，POD 类型的默认初始化是什么都不做，因此没有给 POD 成员（非静态和线程局部对象）值进行值初始化时，会产生不确定值，需要在构造函数内初始化。
> 编译器隐式生成的默认构造函数也不会进行任何初始化，它与拥有空函数体和空初始化器列表的用户定义的构造函数有严格相同的效果，其每个成员都会进行默认初始化。

首先调用基类各成员的构造函数，然后调用基类的构造函数，然后是派生类中各成员的构造函数，最后才是派生类自己的构造函数（可以认为类的构造函数中，会先执行自己成员的构造函数）。
类成员的构造函数的调用顺序，与它们的声明顺序一致。
基类和非静态成员默认进行默认初始化。如果在初始化列表中写明，则不会默认初始化，而是使用初始化列表中的构造方式。
初始化列表会先于构造函数体执行。

```cpp
class A {
public:
  A(int a) { std::cout << '1'; }
  virtual ~A() { std::cout << '4'; }
};
class B : public A {
public:
  B(): A(3) { std::cout << '2'; } // 如果不在初始化列表中初始化A，则会调用A()初始化父类，但A()不存在，会CE
  virtual ~B() { std::cout << '3'; }
};
```

> [具体初始化顺序](https://zh.cppreference.com/w/cpp/language/constructor)：
>
> 1. 如果有虚基类，按基类声明的从左到右、深度优先顺序，初始化各虚基类（所有虚基类子对象都在非虚基类子对象之前初始化）。
> 2. 按基类声明的从左到右顺序，初始化各直接基类。
> 3. 以类定义中的声明顺序，初始化各个非静态成员。
> 4. 执行构造函数体。
>
> 初始化列表中的初始化器的顺序不会影响初始化顺序。

析构函数的调用顺序总是相反：先是派生类，再是类成员，再是基类（从右到左）。

对于多继承的多个基类函数，按照从左到右的顺序调用构造。
如果多继承中有虚基类，则先依次执行虚基类的构造函数，再依次执行非虚基类的构造函数。

> 最好不要在析构函数中执行虚函数。
> 析构时，派生类的析构会先执行，然后才是基类的析构。
> 如果基类的析构函数中使用了虚函数，由于此时对象的派生类已释放（使用派生类的数据成员就是未定义的），虚函数表会调整指向，执行基类的虚函数。这其实与虚函数的语义不符（为了避免 UB 选择这样）。

**初始化列表**

构造函数后`:`的部分即为初始化列表，可用于：初始化基类、初始化成员变量、初始化 const 的成员变量。
const 的成员变量无法在构造函数内部初始化，只能在定义时或使用初始化列表赋值。
没有在初始化列表中定义的成员，会在调用构造函数前调用默认构造，否则会按照初始化列表中的方式调用构造。
再在函数体内设置成员，属于赋值而非构造。

变量初始化的顺序是其声明的顺序，跟初始化列表中的顺序无关。初始化列表中的参数会根据声明顺序依次赋值，然后执行函数体。

> 注意：拷贝/移动构造也可以有初始化列表。**如果定义了拷贝构造而不在初始化列表中使用基类和成员的拷贝构造函数，则基类和成员会调用默认构造，而不是对应的拷贝构造。**只是默认定义 (= default) 的拷贝构造会逐成员/逐基类进行复制。移动同理。

**析构函数为什么不默认是虚的**

因为虚函数需要虚表，导致类对象多保存一个虚表指针，可能会浪费空间。比如空类只有 1B，但如果有虚析构函数则为 4/8B。

**转换函数**

[转换函数](https://timsong-cpp.github.io/cppwp/n4659/class.conv.fct)(conversion function)（[ref](https://zh.cppreference.com/w/cpp/language/cast_operator)）允许一个类隐式转换或显式转换到另一种类型。

**转换构造函数**

不带 explicit 的构造函数被称为[转换构造函数](https://zh.cppreference.com/w/cpp/language/converting_constructor)。
转换构造函数 定义了一个从它的各实参类型到它的类类型的隐式转换。
非 explicit 转换函数（operator T）定义了一个从它的类类型到另一个类型的隐式转换。
复制初始化会将它们作为用户定义的转换序列中的一部分，但不会考虑 explicit 的构造和转换函数。

**explicit**

explicit 用于修饰构造函数和类型转换函数 operator T()，表示它们（该构造函数或类型转换）必须被显式调用，不能通过隐式类型转换调用。
这些函数默认都是 implicit 的，可能导致问题或不想要的情况。

如：`class A`有一个构造函数`A(double a)`以及拷贝构造函数`A(const A&)`，如果不加 explicit，则`A a = 1;`将通过，1 会被隐式转为 double 然后构造成 A，从而能调用`A(const A&)`。**隐式转换代表一个 double（或是能隐式转换成 double 的值）都可被看做一个 A 类型对象、出现在 A 类型的位置上。**
比如：函数`f(const A &x)`能够通过`f(1.0)`调用。
如果用`explicit A(double a)`，则`A a = xx;`右式必须是一个 A 类型值，不能通过一个 double 隐式构造出 A，一个 double 也不能代表 A、出现在它的位置上，除非经过显式类型转换或使用构造函数。

原理是：复制初始化选择构造函数时，不会考虑 explicit 函数。**即它不会用于隐式转换和复制初始化**。
具体见：https://zh.cppreference.com/w/cpp/language/explicit

> `A(const A&)`、`A(A&&)`这种从自身类的构造函数，就没必要 explicit 了。
>
> 如果只是不想隐式转换到特定类型，直接弃置它们就可，比如：`operator long long()`、`operator int() = delete`。

例：

```cpp
class A {
public;
	A(int a) { x = a; y = 0; }
	A(int a, int b) { x = a; y = b; }
    int x, y;
}
class B {
public;
	explict B(int a) { x = a; }
    int x;
}
int main() {
    A a1(1); // ok
    A a2 = 1; // ok，将通过复制初始化调用A::A(int)
    A a3 = A(1); // ok，显式的类型转换
    A a4 = {1, 2}; // ok，复制列表初始化，可认为 {1, 2} 被隐式转为 A

    B b1(1); // ok
    B b2 = 1; // error: 复制初始化不会考虑B::B(int)
    B b3 = B(1); // ok，显式的类型转换
}
```

> C++20 起允许`explicit (expr)`，其中 expr 是 bool 类型的常量表达式，可用于模板。
> 当且仅当 expr 求值为 true 时函数才是 explicit 的（与`noexcept (expr)`类似）。

**自定义类型转换**

自定义的类型转换运算符`operator T()`默认是隐式的。
当类对象进行某个没有定义的类型转换，或一个或多个类对象使用某个没有定义的 operator 运算时，编译器会尝试各种对象本身的隐式类型转换，如果找到能够通过编译的转换，则隐式转换，否则才编译失败。
也就是说，**如果 Node 定义了隐式`operator bool()`，那么任何时候，它将都可以作为 bool 使用**：如果某个地方不能放 Node 但能放 bool，那么就将其转为 bool。
因此，显式类型转换`explicit operator bool()`是必要的。

```cpp
struct Node {
	int *p;
	operator bool() const {
		return p != nullptr;
	}
};

Node a{new int{3}};
cout << std::boolalpha;
cout << a << '\n'; // true，因为 bool 重载了 <<
cout << "int(a): " << static_cast<int>(a) << '\n'; // 1

Node b, c; // 即使 a,b 是不同类型，也可通过转换为 bool 进行运算
cout << "a == b: " << (a == b) << '\n'; // false

cout << "a + b: " << (a + b + c) << '\n'; // 1+0+0 = 1，因为 bool 可以隐式转换为 int
```

当类型不支持某个运算符（比如 <<）而编译出错时，编译器实际上已经尝试了各种各样的隐式转换（比如 int、bool、const char*...，因为它们都支持 <<），才最终确定无法编译。也因此会给出大量的编译报错，告诉你无法匹配任何候选类型。

**静态成员**

静态成员不与类的对象关联，而只与类有关联。
静态成员的定义（初始化）必须在类外部进行（定义该类的文件内），除非静态成员是 const 或 constexpr（即静态常量，只读）。它也不能在构造函数中初始化。
类外部初始化的例子：`int ClassName::a = 1;`、`Singleton* Singleton::instance = nullptr;`。
注意只能初始化一次，否则会重复定义。

如果不在类外定义，则只有`A::v`的声明，不存在定义（但可在链接时找到）。

> 如果未限定作用域，对于在静态数据成员的定义中所使用的名字，它的查找规则与查找成员函数中使用的名字相同，即使它可能在类外部。

**类对象不存在的情况 / 一个类对象包括什么？**

可以对某类声明一个空指针：`A a = nullptr`，a不包含实际对象，但是依旧可以调用A类中的非虚函数和非涉及具体对象的变量。
事实上，一个实际对象只包含虚函数表和它的成员变量，空对象只是不包含这些。其它函数和静态变量与类绑定，而不是必须与对象绑定。
如果一个对象什么都没有，则占用1字节作为占位。

> 1. 类中一旦有 virtual 修饰的成员函数（不论数量），即虚函数，编译器会构建虚函数表，在该类的对象中会存放一个指向虚函数表的指针，占用4字节（或8字节，即指针大小）。
>    如果有，虚函数表指针会放在类内存结构的最开始。
>
> 2. static 静态变量是对类而言的，为所有对象共有，不计算对象的占用空间。
>
> 3. 什么都没有的类，或仅有非虚成员函数的类，或仅有类型声明的类，都为空类，编译器会为其构造占用1字节空间的变量（通常为char型）；若该类为基类，则继承自空基类的子类不计算这1字节的空间，仅计算子类所占空间（**编译器的空基类优化：允许空的基类子对象大小为0**）。
>
>    但是，设一个类 B 中包含了某个空类 A，如果 B 继承自 A，且 B 的第一个非静态成员是一个 A 类型的成员 a，则这个空类 a 仍然会占用1字节，因为 C++ 要求相同类型的对象必须地址不同，如果不分配1字节，则两个 A 类型对象 (B 的实例与成员 a) 会拥有相同的地址。
>    如果 B 不继承自 A，或 B 的第一个成员不是 A 类型，则空类不会占用空间。

例：如下代码中，若按4字节对齐，则`sizeof`为12B。若为1字节对齐，则为9B。若为64位机器、按8字节对齐，则为16B。

```cpp
class C
{
    public:
        virtual void mem_fun(){} // 4B或8B
    private:
        char m_chData; // 1B
        int m_nData; // 4B
        static char s_chData;
};
char C::s_chData=’\0’;
```

**类的大小**

类只是一种类型定义，本身没有实际大小，我们说的类的大小都是指类的对象的大小。sizeof 一个类名会返回该类对象的大小。

类(对象)的大小只与非静态数据成员有关，与函数和静态数据无关。静态数据保存在共同区域，由所有对象共有，所以不会计入某个对象的大小。
虚函数对类的大小有影响，因为会引入虚指针。

> 声明移动构造或移动赋值，会弃置拷贝构造、拷贝赋值和移动赋值/移动构造。

- 三之法则：**如果某个类需要自定义的析构函数、自定义的复制构造函数或自定义的复制赋值运算符，那么它几乎肯定需要全部三者。**
    所以如果实现了三者之一，则要注意是否需要实现另外两者，使用默认的隐式定义很可能出问题（比如析构时需要释放资源，那么在拷贝赋值时肯定需要类似的资源释放，不能用默认行为）。
- 五之法则：**任何想要使用移动语义的类，必须声明全部五个特殊成员函数**（析构函数、拷贝构造、移动构造、拷贝赋值、移动赋值）。**由于声明拷贝、移动或析构函数之一，就会导致移动构造和移动赋值被隐式弃置**，因此即使不实现也应该显式声明 default。

```cpp
class base_of_five_defaults
{
public:
    base_of_five_defaults(const base_of_five_defaults&) = default;
    base_of_five_defaults(base_of_five_defaults&&) = default;
    base_of_five_defaults& operator=(const base_of_five_defaults&) = default;
    base_of_five_defaults& operator=(base_of_five_defaults&&) = default;
    virtual ~base_of_five_defaults() = default;
};
```

使用默认移动构造和移动赋值、不给出实现通常不是错误，但会失去优化机会。

- 零之法则：有自定义析构函数、复制/移动构造函数或复制/移动赋值运算符的类，应该专门处理所有权。**其它类（无所有权语义的类）都不应该拥有自定义的析构函数、复制/移动构造函数或复制/移动赋值运算符**。
    当某个基类要用于多态时，可能需要将它的析构函数声明为 public 虚函数。由于这会阻止隐式移动的生成，因此必须将各特殊成员函数显式声明为 default（即上面的代码）。但是，由于 *规则 - C.67 多态类应该抑制复制操作*，基类的拷贝构造和赋值可能应设为 delete（如果需要就 protected）。

三五零与 *规则 - C.21 如果自定义或显式弃置了拷贝、移动或析构函数，那么应该自定义或显式弃置它们所有* 类似。

**mutable**

[mutable](https://zh.cppreference.com/w/cpp/language/cv) 说明符允许 const 类对象修改相应类成员。可以修饰非引用、非常量类型的非静态数据成员。

mutable 常与 mutex 一起使用（M&M 规则），允许 const 函数给类内的 mutex 加锁，以保证线程安全。

> const 修饰函数，表示这个函数不会修改对象的内部状态。
> mutable 修饰成员，则表示这个成员不算对象的内部状态（或者说外部可观察状态），因此可允许 const 函数修改它。比如：对 mutex 加锁，更新访问次数 counter。
>
> https://timsong-cpp.github.io/cppwp/n4659/dcl.stc#9
> 设 A 类对象 a 拥有 mutable 的 B 类成员 b，则不管 a（或 *this）有没有 const 修饰，a.b 都不会有 const 修饰符，因此可以被修改。
> 因此将 a.b 传入函数时，会优先匹配`B&`而非`const B&`。
> 例：
>
> ```cpp
> struct X {
> X() { std::cout << "1"; }
> X(X&) { std::cout << "2"; }
> X(const X&) { std::cout << "3"; }
> X(X&&) { std::cout << "4"; }
> ~X() { std::cout << "5"; }
> };
> struct Y {
> mutable X x;
> Y() = default;
> Y(const Y&) = default;
> };
> Y y1;
> Y y2 = std::move(y1);
> ```
>
> 由于 Y 的移动构造被弃置，因此 y2 初始化也会匹配 Y 的默认拷贝构造`Y(const Y&)`，会调用各成员的拷贝构造。
> 虽然 y1 有 const 修饰，但 mutable 成员 x 会去除 const，因此在拷贝 x 时会优先匹配`X(X&)`而非`X(const X&)`。
>
> ```cpp
> void g(const X& x) {
> 	puts("gx const");
> }
> void g(X& x) {
> 	puts("gx non-const");
> }
> void f(const Y& y) {
> 	g(y.x); // 如果x是mutable，匹配 g(X& x)
> }
> f(y1); // 输出 gx non-const
> ```

**成员函数限定符**

> https://zh.cppreference.com/w/cpp/language/member_functions

成员函数可以添加多种限定符，来修饰或区分成员函数是被什么样的实例（即 *this）调用。（这与普通函数不同）

- cv 限定符：const、volatile

限定只有 const/volatile 的类对象（*this 有同样的 cv 限定）才可调用该成员函数。
注意，const 函数依然可以修改类内的引用和指针指向的值，因为修改的不是它们本身，而是其指向的位置。const 指的是对象本身的不变性。

cv 修饰符不同的函数具有不同类型，可以重载以实现最佳匹配。

- 引用限定符：(const) &/&&

限定只有 (const) 左值/右值类对象才可调用该成员函数（注意 const & 可以同时匹配左值右值）。
这允许我们在不同的情况下，返回不同的类型。左值对象可以返回左值引用，右值对象可以返回移动后的成员函数，来避免拷贝构造：

```cpp
class Widget {
public:
    using DataType = std::vector<double>;
    DataType& data() &	//对于左值Widgets，返回左值
    { return values; }
    DataType data() &&	//对于右值Widgets，返回右值
    { return std::move(values); }
private:
    DataType values;
};
auto vals1 = w.data();				//调用左值重载版本的Widget::data，拷贝构造vals1
auto vals2 = makeWidget().data();	//调用右值重载版本的Widget::data，移动构造vals2
Widget{}.data(); //调用右值版本
```

这允许我们区别对待左值对象和右值对象，避免不必要的拷贝。在工厂函数中很有用。
与 cv 限定符不同，引用限定符不改变 this 指针的性质：即使在右值引用限定的函数中，*this 仍是左值表达式。

相关内容可见 *C++ - RVO* 和[EMCpp](https://cntransgroup.github.io/EffectiveModernCppChinese/3.MovingToModernCpp/item12.html)。

- 说明符 override：声明该函数覆盖了基类的某个虚函数。如果不是，编译器会报错。
- 说明符 final：阻止类的进一步派生 和 虚函数的进一步重写。

> ```cpp
> struct FFunctor {
> 	void operator()() & { cout << "&      " << '\n'; }
> 	void operator()() && { cout << "&&     " << '\n' }
> 	void operator()() const& { cout << "const& " << '\n'; }
> 	void operator()() const&& { cout << "const&&" << '\n'; }
> } a;
>
> (a)();  // &
> std::move(a)();  // &&
> std::as_const(a)();  // const&
> std::move(std::as_const(a))(); // const&&
> ```
>

**this 指针有什么用**

成员函数代码只有一份，但类对象可以有很多个。（非静态）成员函数内通过 this 来判断自己操作的是哪一个对象。
代码中，通过 this 可以指定某个成员变量，可以返回指向自身的指针或引用（比如实现链式调用）。

this 实际上是成员函数的一个形参，在调用时将对象的地址作为实参传入。

> **为什么 this 不是引用？**
> 历史原因，C++ 引入指针时还没有引入引用。

**delete this**

成员函数中可以释放 this，就像在成员函数外一样。
[注意释放后](https://isocpp.org/wiki/faq/freestore-mgmt#delete-this)，

- 不能访问该对象的任何成员变量及虚函数（虚指针也会被删除）。
    但仍可调用不使用成员变量的函数。
- 不能再访问和使用 this 指针（已经是悬挂指针了）。

注意栈上的/临时作用域内的/全局的类对象不能 delete 它（包括 delete this），否则将导致被 delete 两次，因为它会在作用域结束后会自行释放。
换句话说，**delete 的目标必须是用 new 分配的**，不包括 全局的/new[] 的/原地 new 的。

**成员函数中 this 不能为 null**

> https://www.zhihu.com/question/589273713

通过值为空指针的类对象指针 调用类的成员函数是 UB，即使成员函数内并没有用到 this 指针（比如只输出一句话）。
编译器会假设不存在 UB，即认为成员函数内 this 不可能为 null。

> 标准的内在逻辑是，调用成员函数之前无论如何都要先确定一个合法的 receiver 对象，即使这可能毫无意义（实际并不会做）。
> 函数调用表达式是作用在对象的引用上，而空引用是 UB，自然传空 this 是 UB。

## 模板

**泛型**

泛型编程指：以独立于某种特定类型的方式编写代码。使用时，需要提供具体的类型。
模板是泛型编程的基础，是创建类或者函数的蓝图或公式。通过给定蓝图或公式足够的信息，能将它真正的转变成具体的类或函数。这种转变通常发生在编译时。

**模板的优缺点**

优点：

- 可内联，提高运行效率：
    在 C 中，谓词（比如 sort 中的 cmp）只能通过一个函数指针传递（类型为`int (const void*, const void*)`），编译器不能在编译期识别谓词是什么，所以没法内联，只能在每次比较两个对象时，将其放入栈然后调用 cmp 函数。
    而使用模板做谓词时，编译器会为每种类型实例化一个函数，它的实现是编译期就能确定的，因此可以内联。
- 类型安全，并且能在编译期检查。

缺点：

- 程序的体积增大，编译时间延长：
    需要实例化每个模板（为每个重载生成一份代码）。
- （当模板定义需要在外部文件使用时）不能将实现与定义分离，导致头文件依赖更严重、编译时间更长。
- 报错信息复杂。

> 关于模板声明与定义，见 *模板的实例化*。

**类型擦除 (type erasure)**

> 与（运行时）多态相比，模板的优点之一是可以保留对象的类型信息：多态只能传递一个基类，编译器不能知道它具体是哪个类，会导致一些不便，比如：无法拷贝它（因为不能在编译时确定 要调用哪个类的拷贝构造，除非尝试每个子类能否 dynamic_cast？）（但可以实现 clone 虚函数实现拷贝，可见 *Codes - Any、function*）。
> 而模板将它的参数类型作为了实际类型的一部分，但这也可能导致不便，比如`vector<int>`与`vector<char>`是两种完全不同的类型，不能像基类一样用一种类型代表它们两个，不能将这两种不同类型放入一个容器里（给它们一个共同的基类可以解决，但又回到了多态）。
> 要解决这个问题，可以不暴露 T、不将 T 的类型作为它类型的一部分。**隐藏一部分或整个类型，就是类型擦除**。
> 很多情况下接口并不关心具体的类型是什么，或者根本不需要这个类型，就可以用类型擦除，使接口有更好的通用性、延展性，消除耦合和重复代码。
> 通过它可以在一个容器里放各种类型的对象，但不要求这些对象有一个共同基类，也不会丢失它们的类型信息（不需要向下转型，判断它们是哪个子类）。

类型擦除可**将其实际类型隐藏，全部抽象成一种类型**。

类型擦除有多种方式：

- 通过（运行时）多态（继承和虚函数）。
    缺点是效率略低；不能完全擦除类型（基类型仍然存在）；必须通过继承进行；会增强类型之间的耦合。
- 通过模板。
    类型之间不存在耦合，但是类型仍然存在，虽然 T 可以接收任意类型，但也只能表示一种类型，在编译时就要确定。
    而且用了模板就需要到处都用，有传染性。
- 通过可以存放多种类的类型，比如 union、**variant**。
    但能使用的类型必须一开始就定义好，不能随意使用任意类型。
- 通过某种通用类型，比如 **any**、std::function、void*（但不是类型安全的）、Java 的 Object。
    完全消除了其类型，但使用时必须先自行转换成其原本的类型。

类型擦除的一种实现方式：
定义一个泛型类，比如叫 Any。在内部定义抽象基类，定义各个纯虚接口。在内部定义模板子类（有必要可以多个），通过 T 传入任意类型，调用 T 的方法实现虚接口。Any 的构造函数通过模板将任意类型 T 传入子类。
例子可见 *Codes - type_erasure、any、function*。
这本质还是多态，就是用模板减少重复代码，再用外部类的模板构造函数自动推导类型，不用每次定义都像`Any<xx>`这样写明类型？

缺点：

- 类型擦除和类型恢复可能影响效率。
- 可能不能确定被擦除的对象占用的空间，所以只能把对象放在堆上。
- 类型擦除一般是运行期的行为，会丢失一部分类型信息，所以无法内联。

> 另一种介绍（参考[这里](https://akrzemi1.wordpress.com/2013/11/18/type-erasure-part-i/)）：
>
> 模板和 void\* 都可以表示任意类型，但因为模板的缺点，如果对运行效率不敏感，可以用 void*（或一种 Any 类型）表示任意类型，只写一个函数，避免大量函数重载。这个 void\* 就是一种类型擦除。
> 但是，它不是类型安全的（它完全没有类型，全靠用什么类型的指针解释），如果程序员传递了错误的谓词，编译器也不会给警告，导致 UB。比如：
>
> ```cpp
> int i_bigger (const void * a, const void * b) {
> 	return *(int*)b - *(int*)a;
> }
> int c_bigger (const void * a, const void * b) {
> 	return -strcmp ((const char *)a, (const char *)b);
> }
> bool search_in_C (std::vector<int> & vec, int val) {
> 	// 应该是i_bigger()，但void*可以接收任意类型，编译器也不知道参数里的void*实际会是什么类型，所以能编译
> 	bsearch(&val, vec.data(), vec.size(), sizeof(int), c_bigger);
> }
> ```
>
> 可以通过面向对象和虚函数来实现类型安全：创建一个公共基类（比如 Object），让每个类都继承 Object，实现`Compare(const Object*)`（内部需要 dynamic_cast 尝试转成当前类，再比较。cast 失败就 throw）。这样就能自动调用正确的 cmp，并且两个对象不可比较时也会给错误。
> 但这也会带来一些问题：1. 所有自定义对象都要继承 Object，但第三方库的类型显然很难做到。2. 假如一个类既需要做排序，要继承上面的 Object，也需要做类似的其它功能，要继承其它的 xxObject，就不得不多继承多个基类。3. 如果多继承的基类中，有相同的两个函数，还会产生冲突。4. 类型间不可比较这种错误，只能在运行时检查出来，不能在编译期发现。
>
> 所以还是要用模板，但是要用一种不需要太多实例化的方式。后面的不看了，见上链接。

**模板**

[模板类](https://zh.cppreference.com/w/cpp/language/class_template)（类模板）：在 class 声明前添加`template <typename/class...>`告诉编译器这是一个模板类，并指定模板的参数。
定义模板类时，需要指定模板参数，其余与普通类相同。
模板类名（如 vector）不是类名，只是能用来实例化类。带完整模板参数的才是类名（如 vector<int>）。
类模板的实例化不会实例化没有被使用的成员函数。

[模板成员](https://zh.cppreference.com/w/cpp/language/member_template)（成员模板）：任何不是局部类的 class、struct 或 union 的成员，都可以是模板成员。

[*模板函数*](https://zh.cppreference.com/w/cpp/language/function_template)（函数模板）：在函数声明前添加`template <typename/class...>`。
模板函数的使用与普通函数相同。可以不指定模板参数，编译器会从函数的形参和实参中，解析推导模板实参。但当编译器没有足够的信息推导模板类型时，需要显式指定。

[*模板变量*](https://zh.cppreference.com/w/cpp/language/variable_template)（变量模板，C++14 起）：在变量声明前添加`template <typename/class...>`，使用模板类型定义它。

类的成员函数也可以是模板函数，称为成员模板。它不能是虚函数，与类有独立的模板参数。
当在类外定义成员模板时，必须依次写出类模板和成员模板的模板参数（template <...> 分两行写，然后再 Node<T>::Func(...)）。

在链接时，不同翻译单元生成的相同实例化被合并。

> 模板函数自带 inline，因为其定义只能放在 .h 中。

**模板参数**

[模板参数](https://zh.cppreference.com/w/cpp/language/template_parameters)可以是类型参数，也可以是非类型模板参数 (non-type template parameter, *NTTP*)，但不能使用浮点和类，如：`template<typename T, int size = 5>`。非类型参数可看做类中的一个常量，也会用于标识这个类。
参数还可以是模板（不完整的类型，称为模板模板参数），如：`template<template<typename> typename F = vector>`。
但注意，`template<typename> class F`代表一个参数列表为 typename 的模板，不能接收参数列表不同的模板。可以用`template<typename...> class F`或`template<int...> class F`等。

模板参数可以给出默认值，与函数类似，需要从右往左给出。
与函数默认参数不同，默认值表达式可以使用前面的模板参数。

```cpp
template<typename T, bool = (sizeof(T) <= 8)>
class A {};

A<int> a;
```

不同模板参数的类（函数），就是不同的类（函数），它们有自己独立的 static 成员变量与函数（因为是从一个类模板生成的新的类）。
类模板的 static 成员变量与函数只有在使用时才会实例化。

模板参数可以用于表达式，而不只是用做类型。编译器会尝试所有可能的模板重载，以找到能够匹配的（存在多个时 CE），并自行推断模板参数的值。

```cpp
template <auto> // C++17 起，auto 可自动推导非类型模板参数的类型，但只能确定一种。此处是 int，如果定义不匹配 int 则 CE
class A {};

template<int I, int J>
void f(A<I+J>);
template<int K, int L> // 重载
void f(A<K-L>);

ff<0,0>(A<0>{}); // 均可匹配，CE
ff<2,1>(A<1>{}); // 只能匹配第二个
```

> 模板的非类型模板实参 只能接收以下类型：整型和算术类型、函数指针、成员指针、对象指针、对象左值引用。
> 当使用对象指针或左值引用时，实参必须是静态存储期对象，才可在 *C++ - 编译期取地址*（指针实参也可以是求值为 nullptr_t 的常量表达式）。
>
> 只要传给模板的模板实参不同，就会实例化出不同的模板对象，即使它们可能有相同的函数类型。

**模板友元**

相同模板类的不同实例化属于不同类，因此不能互相访问私有成员。
可声明各实例化为友元：

```cpp
template <class T>
class A {
    template <class U>
	friend class A;
public:
    template <typename U>
	void set(A<U>& a) { // 接收同模板的任意实例化
		v_ = a.v_;
	}
private:
    int v_{1};
};

A<int> a;
A<double> b;
b.set(a);
```

注意，此处友元声明的模板参数与限制必须与原声明完全对应，包括 requires。

```cpp
template <typename T, typename Allocator = std::allocator<T>>
	requires SVectorRequire<T, Allocator>
class SVector {
	template <typename U, typename U_Allocator>
		requires SVectorRequire<U, U_Allocator>
    friend class SVector;

  	// ...
};
```

**模板实参推导**

> https://zh.cppreference.com/w/cpp/language/template_argument_deduction
>
> auto 与此相同。

设 T 是模板形参，P 是依赖 T 的函数形参（比如 T、T&、const T），A 是要传递的实参。
推导的目的是决定 T 的一个取值，使得特化后形参 P 的类型与实参 A 的类型相同（A 会在推导前进行调整）。
需要注意的部分：

- 如果 P 不是引用类型，如果 A 是 U 的数组或数组引用，则 T 是 U\*；如果 A 是函数，则 T 是函数指针；否则 T 是 remove_cvref_t<decltype(A)>（不会附加 cv 与 &）。
- 如果 P 是引用，则按照 P 引用的类型推导。比如：`f(const T&)`使用 const T 进行推导，传递 const int，则 T 就是 int。
    如果 P 是到无 cv 限定 T 的右值引用，即万能引用 T&&，且实参 A 是左值，则将 A 的左值引用替换 A 进行推导，即推导出的 T 会带左值引用（这是 *万能引用* 和 *完美转发* 的原理）。
    除了万能引用且实参为左值时，T 本身都不会带 &。
- 如果 P 不是引用，且 A 有 cv 限定，则推导时会忽略 A 的顶层 cv。
- 如果 P 有 cv 限定，则推导时会忽略 A 的顶层 cv。

如果上述推导不能得到与实参相同的形参类型，则考虑如下规则：

- 如果 P 是引用，那么推导出的 A（即 P 引用的类型）可以比 A 有更多的 cv 限定。
- A 可以具有另一种指针或成员指针类型，并可以通过限定性转换或函数指针转换转换到推导出的 A。
- ...

**类模板实参推导** / **CTAD**

C++17 前，必须声明每个模板实参才能实例化一个类模板。
[CTAD (class template argument deduction)](https://zh.cppreference.com/w/cpp/language/class_template_argument_deduction) 在 C++17 引入，允许编译器通过初始化表达式和推导指引，推导出缺失的类模板实参。

比如：`pair p(1, 1.)`可以推导出 pair 是`pair<int, double>`，`lock_guard lk(mu)`可以推导出是`lock_guard<mutex>`。

> CTAD 可能导致问题？`-wctad`可以开启警告。

**推导指引**

用户可以定义[推导指引 (deduction guides)](https://zh.cppreference.com/w/cpp/language/class_template_argument_deduction)，在未指定类模板实参时，CTAD 会自动根据推导指引推导模板实参。
可以为特定类型建立指引，使用特殊的模板实参；也可以为某个模板类型建立指引。

格式为：

```cpp
explicit说明符(可选) 模板名(形参列表) requires子句(可选) -> 简单模板标识;

// 如果形参列表需要使用模板参数，则在此声明
template <模板形参列表> requires子句(可选)
explicit说明符(可选) 模板名(形参列表) requires子句(可选) -> 简单模板标识;

// 简单模板标识：模板名  <模板实参列表 (可选)>
```

当初始化表达式与形参列表匹配时，将使用简单模板标识中指定的模板参数作为模板实参。
当声明 explicit 时，该推导指引不能在复制初始化语境中使用，但会被考虑，如果选中该指引则 CE（与构造函数中的 explicit 相同）。

`模板名(形参列表)`类似一个函数声明，但它不是函数也没有函数体。
推导指引不会被名字查找所找到。除了在推导类模板实参时与其它推导指引构成重载外，它不会参与重载决议。

其它要求：

- 必须与类模板有相同的作用域。如果是成员类模板，则要求有相同的访问权限。
- 不能在同一翻译单元中，为同一类模板多次声明形参列表相同的推导指引。

```cpp
// 接收引用时，将去掉引用的类型作为模板实参 T
template <class T>
reference_wrapper(T&) -> reference_wrapper<T>;

template <class T>
struct S {
	S(T) {}
};

explicit S(bool) -> S<int>;
S(int) -> S<int>;
template <class T> S(T) -> S<double>;

S a{true}; // T 是 int
// S b = {true}; // CE，匹配到 explicit S(bool) -> S<int>
S c = {'a'}; // T 是 double
```

推导指引常用于：为特定类型或特定样式类型指定推导结果；**为不存在对应构造函数的构造方式指定推导结果**（如果没有推导指引，编译器可以根据匹配的构造函数进行推导，因此可认为编译器会根据构造函数生成隐式推导指引）。

```cpp
template<class... Ts>
struct overloaded: Ts... { using Ts::operator()...; };

// 推导指引（C++20 起不需要）
// overloaded 本身只有 overloaded() 和 overloaded(const overloaded&) 两个构造函数，因此默认只能推导这两种情况
template<class... Ts>
overloaded(Ts...) -> overloaded<Ts...>;
```

**模板的特化 / 实例化 / 编译**

当提供了模板实参，或函数模板和类模板的模板实参被推导出时，它们会替换模板形参，产生模板的一个特化 (specialization)。
特化可以显式提供，也可以隐式生成特化。
在要求完整对象类型的语境中使用某个类模板特化，或在要求函数定义存在的语境中使用某个函数模板特化时，除非模板已经被显式特化或显式实例化，否则模板被隐式实例化 (instantiation)。

编译器遇到模板定义时并不会立刻生成代码并编译，只有实例化（显式或隐式）时才会生成模板的代码。因此大多数编译错误只能在实例化期间检查。

通常当调用函数时，编译器只需要知道函数的声明；当使用类对象时，只需要知道类的定义，不需要成员函数的定义。因此函数声明与定义、类定义与成员函数定义可以分离。
但模板的定义必须在隐式实例化点可见，即为了隐式实例化，编译器必须知道模板的定义。
因此模板定义一般放在头文件中，以保证同一翻译单元内的模板使用能看到模板的定义、能进行实例化。

> 一个程序内（可能包含多个翻译单元），同一个模板的显式实例化和显式特化不能同时出现，除非显式实例化对显式特化是可见的（可达）。
> 13.9.4.7：一个程序内，显式特化的声明 必须对所有会进行隐式实例化的模板特化的使用可见。即隐式实例化后，不能再出现显式特化的声明或定义。
>
> [13.9.1.6](https://eel.is/c++draft/temp.spec#general-6)：显示实例化和显式特化的声明中使用的名字，不会进行通常的访问权限检查，除非名字位于函数体、默认实参、成员声明等少数场景内。
>
> 模板包含两种名字：不依赖模板参数的名字、依赖模板参数的名字。
> 使用模板时，所有不依赖模板参数的名字都必须是可见的；当模板被实例化时，模板的完整定义必须是可见的。这要由模板的提供者保证。
> 用来实例化模板的函数、类型、使用到的与类型相关的运算符的声明，都必须是可见的；传递给模板的实参必须支持模板中的操作，并能正常工作。这要由模板的使用者保证。

> 只在当前文件内使用的，比如类的私有模板函数，可以放在 .cpp。
> 类模板实例化时，只有实际使用到的成员函数，才会被实例化（即使模板实参在某个函数中无法执行，只要不调用它就没事）（显式实例化例外）。
>
> 当需要将实现放在头文件时，可以将实现写在另一个文件（比如`map.inl`，叫内联函数源文件；不用 .cpp 是因为 .cpp 需要配置不链接它生成 .o），然后在头文件的下面（或任意位置？）`#include "map.inl"`（在预处理时会被替换成代码），以使头文件更简洁（虽然没有本质区别，依然一起编译）。
> 比如：[folly](https://github.com/facebook/folly/blob/main/folly/AtomicHashMap-inl.h)。

> gcc 中不能在类内特化某个函数成员模板，因为 gcc 中模板特化只能在命名空间作用域，不能在类作用域。
> 标准是允许的，clang 也可以。gcc 只能用 if constexpr 代替。

**显式实例化**

如果没有显式实例化，只有当模板被使用且需要它有完整类型时，才会进行隐式实例化。
因此模板的同一种特化可能被实例化多次：当多个独立编译的源文件（不同的编译单元）使用了相同模板、提供了相同实参时，每个文件中都会有该模板的一个实例，为了避免*程序膨胀*，链接器会合并相同的符号，但会导致编译时间变长。
一般问题不大，但大系统中影响可能比较严重，可通过[显式实例化](https://zh.cppreference.com/w/cpp/language/class_template)避免。

> 在同一编译单元中，同一个模板实例只会被实例化一次。
> gcc 中模板实例化生成的实体为弱符号，因此在链接时，重复实例化不会出错。
>
> 除非显式特化或实例化，模板只会在使用该模板的一个特化且“需要有完整定义的类型或类型的完整性会影响程序语义”的上下文中被隐式实例化。换句话说，只使用模板某个特化的指针或引用并不会导致模板实例化，自然也不会检验模板实例化后的定义和内部的 static_assert。

显式实例化有两部分：

- 一个文件进行**显式实例化定义**，触发实例化、生成代码。
    格式为`template declaration`，要给出所有模板实参。

```cpp
// a.cc 可以单独写一个实例化文件
template class Node<int>; // 显式实例化定义

template void f<int>(int);
```

- 其它文件使用 **extern 进行实例化声明**。

```cpp
// b.cc
extern class Node<int>;
Node<int> x; // 目前没有 Node<int> 定义，会在链接时查找

extern template void f(int);
f(1);

// 也可以创建一个函数声明，显式实例化该声明，与普通函数类似，会创建符号，等链接时填充
template <class T> void f(T); // 模板声明
template void f<int>(int); // 显式实例化该声明
f(1);
```

extern 模板声明表示在程序的其它位置，一定有该实例化的一个非 extern 定义，可以直接使用，所以编译器不会对 extern 进行实例化，但会为它创建符号。
可以有多个文件使用 extern 声明，但给出定义的只能且必须有一个。它们必须被链接到一起。

注意：

- 由于编译器在使用模板时自动实例化，所以 extern 声明必须出现在所有使用该实例化的代码前，否则仍会进行实例化。
- **显式实例化定义会实例化类的所有成员**（包括可能没用到的成员函数），因为编译器不知道其它程序会使用哪些成员。这与隐式实例化不同。
    因此，用来显式实例化的类型，必须能用于模板的所有成员（包括函数）。

**显式特化**

模板类型会对所有类型生效，但有时我们想对某种类型进行特殊处理，就可以用显式特化 (specialization)。特化的模板类是与原模板独立的实现，不会继承成员定义等内容。
显式特化可以分为特化和[偏特化](https://zh.cppreference.com/w/cpp/language/partial_specialization)。只有类和模板变量能偏特化。给出特化声明后，该模板类就被**显式特化**。
对于要特化的模板参数，不在`template`中给出。对于类，在类的声明后添加要特化的类型；对于函数，直接写明返回值和参数类型即可；对于变量 TODO。

> 特化必须在导致隐式实例化的首次使用之前，在每个发生这种使用的翻译单元中声明。
> 只有声明没有定义的模板特化，可以像其他不完整类型一样使用，即如果只使用特化的引用或指针，即使没有特化定义也可以。它们会在需要完整类型时被隐式实例化。

注意，主模板声明不能在 template 外写模板参数，因为这种就是特化。
所有特化的尖括号中（`struct Name<...>`）都必须满足最初的、唯一的主模板声明中的模板参数。
**特化的 template 中的参数，不需要与模板声明匹配，可用于将尖括号（`Name<...>`）中的实参绑定到某些参数上，这些参数会自动推导、得到合适值。**

```cpp
template <int pos, typename L, typename R> // 模板类声明
struct Split;

template <int now, int... L, int... R> // 特化的模板参数不需要与声明匹配，用于获取类型或值使用
struct Split<now, List<L...>, List<R...>> { // 该处需要与声明匹配（注意 List<...> 是类型！）
    // 通过 List<L...>，可以得到 L... 的值（List 无模板参数的话也不是完整类型）
	using next = Split<now - 1, List<L..., List<R...>::value>, typename List<R...>::PopFrontT>;
	using newL = typename next::newL;
	using newR = typename next::newR;
};
```

特化指将类模板中的所有参数进行具体化，并给出与原模板不同的实现。
此时没有模板参数，但还是要用`template <>`告诉编译器这是模板的特化。
例：

```cpp
template <class T>
class stack {};

template <> // 对于类，在声明后指定。如果需要根据特化参数推导类型，可放在 <...> 中
class stack<bool> {};

template <class T>
T max(const T t1, const T t2) {
	return t1 < t2 ? t2 : t1;
}

template <> // 对于函数，在返回值和参数中不用模板即可
const char* max(const char* t1,const char* t2) {
   return (strcmp(t1,t2) < 0) ? t2 : t1;
}
```

偏特化/部分特化指将模板中的部分参数进行具体化（其实与特化没有区别？只是有一些和原来一样的模板参数）。
没有具体化的模板参数，既要写在`template <...>`中，也要写在声明后；具体化的参数同上，写在声明后。
函数模板不能偏特化，只能全特化，想要偏特化只能通过仿函数，TODO 可参考[这里](https://zhuanlan.zhihu.com/p/509671916)。
例：

```cpp
template <class T, class Allocator>
class vector {};

template <class Allocator>
class vector<bool, Allocator> {};
```

> 不同的特化间可以继承，以减少重复代码（但很丑）。
>
> 为什么使用特化/偏特化，不用虚函数？
> 一点是特化的语义更明确，这是一个新的类，数据定义与原模板类不同；另一点是虚函数是运行时多态，在运行期间要访问虚函数表、找到要调用的函数，而特化是在编译时就能确定调用的函数地址，效率更高。

**函数特化的匹配规则**

> 函数模板与非模板函数可以重载。部分内容见 *C++ - 重载决议*，*Quiz - 251.*。

最特化的优先匹配，即类型最精确的优先使用。
对于函数，可能还存在重载的非模板函数。选择要调用哪个函数的规则为：确定参数类型，**先找出能够完全匹配的函数（模板可以匹配任意类型，只要满足约束），如果有多个，优先选择非模板函数，其次是最特化的模板函数，最后是无特化的**（准确来说是最特殊的、可接收类型最少的模板函数最优，见 *Quiz - 251.*）。
例：

```cpp
template <class T> void f(T);
template <> void f(int);
void f(int);

f(1.0); // 参数为double，只有f(T) 完全匹配，因此选择f(T)（在编译时会生成f(double) 函数）
f(1); // 参数为int，都可匹配，所以选择非模板重载
f<>(1); // <> 是空的模板参数列表，所以只会选择模板函数；而特化f(int) 比更f(T) 更合适
```

非模板函数与具有相同类型的模板特化始终不同。即使具有相同类型（返回值、形参列表均相同），不同函数模板的特化（可以是模板参数名字不同）也始终互不相同，可以用显式模板实参列表进行区分。

**模板函数重载**

当模板参数用于函数返回值或参数的表达式中时，它们会作为函数签名的一部分，以便于重载。
函数仍然签名相同的两个模板函数，自然无法重载，会发生重定义。

```cpp
template <auto>
class A {};

// #1
template<int I, int J>
void f(A<I+J>) { // void f(A<(I + J)>)
	cout << __PRETTY_FUNCTION__ << '\n';
}
template<int K, int L>
void f(A<K-L>) { // void f(A<(K - L)>) 签名不同，可重载
	cout << __PRETTY_FUNCTION__ << '\n';
}
template<int K, int L>
void f(A<K+L>) {} // 签名与 void f(A<(I + J)>) 一致，因此重定义

// #2
template<int I>
void f(A<I>);
f(A<1>{});

template<int I> // 重载，但如果不写明模板参数、自动推导，仍会选择 A<I>
void f(A<I+1>);
f<0>(A<1>{});

template<int I> // 重载，同上
void f(A<I+2>);
template<int I> // 与上功能等价但签名不等价，虽然能重载，但程序非良构。很可能因重定义而 CE
void f(A<I+1+1>);
```

**注意，模板递归必须要有边界处理，否则会不断递归、尝试解析类型！或者可能未定义。**

```cpp
// 如果不特化 MaxAlign<T>，则 MaxAlign<T> 无法确定类型，因为此时 MaxAlign<T, Ts...> 还未定义完成
template <typename T, typename... Ts>
struct MaxAlign {
	static constexpr size_t value = alignof(T) >= MaxAlign<Ts...>::value ? alignof(T) : MaxAlign<Ts...>::value;
};

template <typename T>
struct MaxAlign<T> {
	static constexpr size_t value = alignof(T);
};
```

**auto 定义参数 / 简写函数模板 / 简写模板**

[C++20 起](https://zh.cppreference.com/w/cpp/language/function_template)，当函数声明或模板函数声明的形参列表中出现占位符类型 auto 时（前面可以加 concept 约束），该声明将声明一个函数模板，并在模板形参列表中，为每个占位符加一个模板形参。
C++14 起，就可以使用这种方式定义*泛型 lambda*（为 operator () 附加模板参数）。

```cpp
void f1(auto); // 与 template<class T> void f(T) 相同
void f2(C1 auto&); // 如果 C1 是概念，则与 template<C1 T> void f2(T&) 相同
void f3(C2 auto...); // 如果 C2 是概念，则与 template<C2... Ts> void f3(Ts...) 相同

template<class T, C U>
void g(T x, U y, C auto z); // 与 template<class T, C U, C W> void g(T x, U y, W z); 相同

template <typename T>
concept C = requires(T x) {
				x.size();
             } || std::is_same_v<T, int>;
// concept 放在 auto 前
void foo(C auto x) {}
// 以下将生成不同的函数模板
foo(1);
foo(std::string{});
```

```cpp
	cout << __FUNC__ << '\n';
}
// void f3(const T&) [with T = int; typename std::enable_if<is_same_v<T, int>, int>::type <anonymous> = 0]
// 签名不同于
// void f3(const T&) [with T = double; typename std::enable_if<is_same_v<T, double>, int>::type <anonymous> = 0]
template <typename T, std::enable_if_t<is_same_v<T, double>, int> = 0>
void f3(const T&) {
	cout << __FUNC__ << '\n';
}
```

注意如果是成员函数，enable_if 条件使用的需要是函数模板参数，而非类，这样才能让 enable_if 的条件在函数模板实例化时计算，而不是在类模板实例化时计算。

```cpp
// CE: Failed requirement '5 <= 4';
template <int L>
class C {
    template <std::enable_if_t<(L <= 4), int> = 0>
    void f() {}
};
// ok
template <int L>
class C {
    template <int _L = L, std::enable_if_t<(_L <= 4), int> = 0>
    void f() {}
};
C<5> c;

```

但是，enable_if 只能判一些 bool 值，无法验证表达式是否合法？
此时可以使用 [void_t](https://zh.cppreference.com/w/cpp/types/void_t)，见 *declval*。
void_t 与 enable_if 类似，也可启用多个：

```cpp
template <typename T, typename = void>
struct test : N<1> {};

template <typename T>
struct test<T, std::void_t<decltype(std::declval<T>().begin()), decltype(std::declval<T>().end())>>
    : N<2> {};

template <typename T>
struct test<T, std::void_t<decltype( ++std::declval<T&>() )>>
    : N<3> {};
```

**declval**

[std::declval<T>](https://zh.cppreference.com/w/cpp/utility/declval)将 T 转换为引用类型，能使 *decltype* 中的操作数不经过 T 的构造函数，就能使用 T 的成员以获取类型。
如果 T 不是 void，返回 T&&，否则返回 void。
注意，能调用的成员也取决于 T：**当传入 T 时，生成的 T&& 为右值，无法调用 & 成员函数**（右值只可绑定 const& 和 &&）；当传入 T& 时，T& && 实际为 T&，才可调用 & 成员函数。

通常用于模板，获取模板参数 T 的成员；或是直接生成 T 类型的对象。因为 T 可能没有默认构造函数。
它只能用于不求值语境，不需要有定义。不能求值包含此函数的表达式。

```cpp
struct A {
    A() = delete;
    int foo() const;
};

// decltype(A().foo()) x = 1; // CE：没有默认构造
decltype(std::declval<A>().foo()) x = 1; // ok
```

可用于 SFINAE 检测表达式是否合法（类似 concept）（更多见 *requires 表达式*）：

```cpp
// 不支持前置自增
template <class, class = void>
struct has_pre_increment_member: std::false_type {};

// 特化识别支持前置自增的类型
template <class T>
struct has_pre_increment_member<T,
			std::void_t<
				decltype( ++std::declval<T&>() ), // 注意前缀++需要T&对象，T&&没法调用++！
				decltype(std::declval<T>().begin()), // 可验证多个
				typename T::container_type, // 类型要求
				Int<T> // 类型要求
			>
		>: std::true_type {};

template <typename T>
concept C = std::is_same_v<T, int>;

template <C T> // 只能接收满足 C 的类型
struct Int {};
```

但是，void_t 配 decltype 只能验证表达式合法，不能判一些 bool 值？不能像 concept 一样可连接多个 requires 表达式。
但和 enable_if 搭配可以（利用 void_t 先将表达式合法性转为 bool 值）：

```cpp
template <typename T, typename = void>
struct HasMember: std::false_type {};

template <typename T>
struct HasMember<T, std::void_t<decltype(std::declval<T>().value)>>: std::true_type {};

// 通过 enable_if 与 void_t 搭配，可以同时约束表达式有效性和 bool 值
template <typename T, typename = void> // 注意默认类型要与下面的特化相同？
struct HasConstMember: std::false_type {};

template <typename T>
struct HasConstMember<T, std::enable_if_t<HasMember<T>::value && std::is_const_v<T>>>: std::true_type {};
```

也可以用于仿函数和 lambda：`decltype(std::declval<F>()(...))`。这相当于实例化可调用对象，并调用其 operator () 成员函数。可获取使用某种参数的情况下，仿函数的返回值类型。如：*variant* 的 *visit* 实现。

```cpp
auto o = overloaded{
    [](int arg) { return 1; },
    [](const std::string& arg) { return "abc";},
};

decltype(std::declval<decltype(o)>()("abc")) p;
cout  << typeid(p).name() << '\n';

// 或者封装一下
template <typename F, typename... Args>
struct return_type;

template <typename F, typename... Args>
struct return_type<F(Args...)> {
    using type = decltype(std::declval<F>()(std::declval<Args>()...));
};
```

**typename / template 消歧义**

typename 用来定义模板，或告诉编译器 后面的标识符是一个类型，而不是变量或函数，用来消除歧义。

在类模板定义内（包括成员函数和嵌套类），某些名字可以推导为指代*当前实例化*的名字，此时不需要写出这些模板实参和 typename 消除歧义。

```cpp
template<class T>
class A {
    A* p0; // 指代 A<T>
};
template<class T>
class A<T*> {
    A* p0; // 当前实例化，指代 A<T*>
    A<T*>* p1; // 当前实例化
    A<T>* p2;  // 不是当前实例化
};
```

在模板的声明或定义中，不是当前实例化、且取决于某个模板形参名字的成员，不会被认为是类型名（认为是成员变量），除非使用 **typename** 消歧义，或它已经被设为类型名（如用 typedef 声明或是基类名）。
类似的，模板定义中，不是*当前实例化*的成员的待决名同样不会被认为是模板名，除非使用 **template** 消歧义，或它已被设立为模板名。
例子见 [*当前实例化*](https://zh.cppreference.com/w/cpp/language/dependent_name)。
正确地多加 typename / template 不是错误。

类似的，调用（模板？）成员函数时也可能需要 template 来消歧义（避免被认作成员变量）。

> 简单示例：
> 当只使用模板定义时，会产生如下的歧义：
>
> ```cpp
> template<typename T>
> void f(T x) {
> 	T::iterator* it; // error
> 	// typename T::iterator it; // ok
> }
> ```
>
> 只有知道了 T 的具体类型后，编译器才能知道 T::iterator 是不是一个类型（而非成员变量从而认为是`x * it`），才能允许这样一个变量声明。
> 为了减少代码的歧义，需要在 T::iterator 前加上 typename 来声明这是一个类型：`typename T::iterator it;`。
>
> 在模板定义中可代表某种类型，`template<typename T>`与`template<class T>`是一样的，保留后者是为了兼容。
> 最初只有 class，引入 typename 是因为 class 也是类定义，可能导致混淆，尤其是声明一个标识符为类型时。

**待决名** (**dependent name**)

在模板定义中，某些类型和表达式的含义可以在不同的实例化间有所不同，称为[待决名](https://zh.cppreference.com/w/cpp/language/dependent_name)（就是其结果取决于模板参数，没法在实例化前就确定）。
待决类型、类型/值待决表达式见 cppref。

> 比如 Foo<T>::Bar，Bar 可能被定义成一个类型，也可能被某个显式特化定义成变量，在 T 不确定前无法确定其含义。
> 不管它是类型还是一个数字变量，`Foo<T>::Bar<0> x` 都是一个合法表达式：如果是`int Bar = 1;`，那么表达式可视为`(1 < 0) > x`。

在模板的声明或定义中，不是当前实例化的成员、且取决于某个模板形参的名字（即待决名）不会被认为是类型，除非使用关键词 typename 或它已经被设立为类型名（例如用 typedef 声明或用作基类名）。

类似的，不是当前实例化的成员的待决名不会被认为是模板名，除非使用关键词 template 或它已经被设立为模板名。
比如：`s.template foo<T>();`（foo 是一个模板函数）。

> 如果想在这种不确定的场景让编译器把某个标识符当成模板，需要加 template；如果想在这种不确定的场景让编译器把某个标识符当成类型，需要加 typename。
>
> 模板中会进行[两阶段名字查找](https://zh.cppreference.com/w/cpp/language/two-phase_lookup)：
> 对于在模板定义中使用的非待决名，会在检查该模板的定义时，进行无限定的名字查找（这与普通函数一样，无法找到在当前定义后再声明的东西）。
> 对于在模板定义中使用的待决名，它的查找会推迟到确定它的模板实参的时候（即实例化时）（因此有可能找到在它定义后再声明的东西）。
>
> 如果某个基类取决于某个模板形参（即基类是待决名），那么无限定名字查找不会检查它的作用域（不管是在定义处还是在实例化时）。

**形参包 / 可变模板参数 / 变参模板**

C++11 起，带省略号的参数称为[参数包/形参包](https://zh.cppreference.com/w/cpp/language/parameter_pack)，模板形参包能接收任意个模板实参（包括0个），函数形参包能接收任意个函数实参。
在模板参数前加省略号，它就是一个可变模板参数，能够接收任意个、任意类型的参数。
展开的方式有很多，见 *C++ - 一些代码 - 模板使用 - 形参包示例*。

**`((expr), ...);`**可以展开形参包并依次执行表达式。

```cpp
template<class... T>
int func(T... t){
	(f(t + 1), ...);
}
// 多个 &x 获取多个指针
int (*ptrs[10]);
((ptrs[size++] = &args), ...);

// 使用 addressof 可以避免有 operator& 重载
int (*ptrs[sizeof...(args)]) = {
    std::addressof(args)...
};
```

如果 expr 内含有二元运算符，则称为[折叠表达式](https://zh.cppreference.com/w/cpp/language/fold)（C++17 起），会将包内的各个实参依次进行二元运算得到最终值。

```cpp
template<typename T, typename... U>
concept IsAnyOf = (std::same_as<T, U> || ...);

template<typename... Args>
int sum(Args&&... args) {
    return (args + ... + (1 * 2)); // 额外值 1*2 需要有括号
}
sum(1, 3, 6); // 1 + 3 + 6 + 2
```

但是，不能直接获取参数包中的每个参数（如 args[i] 是不行的，需要 C++23），只能通过递归或折叠表达式展开参数包的方式来遍历获取。

递归展开参数包（C++11 可用）：
多加一个模板参数，每次处理一个参数，将剩下的递归处理。

```cpp
template<class T, class ...Args>
void F(T value, Args... args)
{
	cout << value << " ";
	F(args...);
}
// 为了让其递归终止，还需要一个无参的函数
void F()
{
    // end
}
// 为了让其递归终止，只含一个模板参数的函数也可，但 F 不能接收0个参数
template<class T>
void F()
{
    // end
}
```

这样的缺点是，如果调用 F 时没有传入参数，会匹配无参的、结束递归的 F，而不是模板函数。如果想要不管什么情况都要匹配模板函数，需要再加一个外层函数。
使用`sizeof...`判断个数来结束递归是不可行的，原因没看懂。

如果参数类型都相同，可通过初始化列表来获取参数：

```cpp
template<class ...Args>
void ShowList(Args... args)
{
	int arr[] = { args... }; // 注意只能传入int，且至少1个
	for (auto e : arr)
		cout << e << " ";
}
```

通过这种方法和逗号表达式，也可以不通过递归就在 C++11 执行多语句，而不需要折叠表达式展开：

```cpp
template <class... T>
void f(T&&... vals) {
    // C++17
	((cout << vals << ' '), ...);

    // C++11
    // 优化后似乎不会调用 array 构造？
	std::array<int, sizeof...(vals)>{(cout << vals << ' ', 0)... };
    // 或用普通数组
    int _[]{ (cout << vals << ' ', 0)... };
}
```

普通的可变参数，如 printf，需要用 va\_start 之类的宏实现？见 *C++ - 变长实参*。

> 使用省略号做形参的函数（如`f(...) {}`），始终在重载集中，可匹配任意参数，但具有最低重载优先级。
>
> 如果形参包的每个合法的特化，都要求传入空形参包，那么程序非良构 (ub)，不要求诊断（难以检查？）。见 quiz *228.*。

**奇异递归模板模式 (Curiously Recurring Template Pattern) / CRTP**

[CRTP](https://zh.cppreference.com/w/cpp/language/crtp) 指一个类A有一个基类，这个基类是类A本身的模板特化。

用于静态多态，可以像多态继承与虚函数一样，由基类暴露接口、由派生类实现，为子类引入同名但不同实现的函数，但调用时没有虚函数开销、可以内联。
也可以称为静态分发、函数注入或 mixin 模式，可以让基类给子类注入函数。此时继承不是 is-a 的关系，而是 -able 或 include。

但是这不是真正的多态，CRTP 的子类并不是继承同一个父类，子类之间并没有什么关系，只是有相同接口。
如果想要在一个容器内存储 CRTP 的不同子类，需要用 variant 包含对应类型（或用 any，或为 Base 再定义一个公共基类），但这也会引入虚函数。

```cpp
template <class Derived>
struct Base {
    void name() { // 注入同接口但实现不同的函数
        (static_cast<Derived*>(this))->name(); // 此时 Derived 是不完整类型，但可以使用 Derived*
    }
    std::unique_ptr<Base> clone() const { // 注入实现相同的 clone 函数
        return std::make_unique<Derived>(static_cast<const Derived&>(*this));
    }
    int x;
};
// 隐式实例化 Base<D1>
// 但因为未使用到 Base::name，所以不会实例化 Base::name、不会因为子类 name 尚未声明而出错
// 在后续调用 Base::name 时，才会实例化该函数，并且已经能找到子类 name 的定义
struct D1 : Base<D1> {
    void name() { puts("D1"); }
};
struct D2 : Base<D2> {
    void name() { puts("D2"); }
};

template<typename T>
void f(Base<T>* b) {
	b->name();
}
int main() {
    // 必须定义子类，不能直接使用父类模板定义对象，否则它不会包含子类对象、也不会有继承，自然也无法将 this 转为子类（它没有子类，this 只是指向 Base，不会指向 D1）
    // Base<D1> b1; b1.name(); // 未定义行为，这只是一个 Base 对象，不存在 D1 子类对象
    // Base<D2> b2; b2.name(); // 未定义行为
    D1 d1{}; d1.name();
    D2 d2{}; d2.name();

	vector<std::variant<D1, D2>> v {d1, d2};
	auto lam = [](auto&& d) { f(&d); };
	for (auto& d: v) {
		std::visit(lam, d);
	}
}
```

`enable_shared_from_this<T>`就是为子类注入了函数`shared_ptr<T> shared_from_this()`。它有一个 weak_ptr 成员，会在构造 shared_ptr 时初始化，在调用 shared_from_this 时生成一个 shared_ptr。

其它例子：

- 子类都实现了 operator <，在基类可以给所有子类定义 operator ==（内部也 static_cast 然后调用 <）（[链接](https://godbolt.org/z/a91Ev1Pvx)）。
- 为类注入单例（[链接](https://godbolt.org/z/odM4xazfz)）。
- 为类注入 COW（[链接](https://github.com/apache/doris/blob/master/be/src/vec/common/cow.h#L85)）。

> 模板类的函数在被调用前，不会进行实例化，所以能在 Derived 定义前，就在函数中使用它的成员或函数。
> 即使函数中使用了模板实参不包含的成员或函数，只要不调用该函数、不实例化，就不会出错。
>
> 因为各“子类”是独立的，所以 CRTP 也不需要同一接口的返回值相同（“基类”中用 auto 声明返回值即可，会实例化独立的模板类）。
>
> 在多重继承中，需要多个模板嵌套定义，很不方便。

**状态元编程** / **STMP**

> https://zhuanlan.zhihu.com/p/646752343
>
> STMP 未被标准支持，可能是非良构的，并且容易导致 odr violation（违反 ODR），比如：如果 A, B 都使用了 STMP 实现的计数器，C 先 include A 再 B，D 先 include B 再 A，那么编译后的 C 和 D 中的 A, B 得到的计数器值是不同的（注意 include 是文本替换，再编译）。

C++ 常规的常量表达式求值既不依赖、也不能改变全局的状态，它们类似一个纯函数，只要输入确定，输出就确定。模板元编程也是如此。
状态元编程 (stateful template meta programming) 能够在编译期的常量求值中引入全局状态，使同样的常量表达式可以求值出不同结果。
比如：实现一个编译期计数器：

```cpp
constexpr int a = f();
constexpr int b = f();
constexpr int c = f();
static_assert(a == 1 && b == 2 && c == 3); // 常规的模板变成无法实现，f 将始终得到相同结果
```

**STMP 的原理：通过控制模板实例化来控制函数何时定义；根据函数是否定义返回不同的值。**

友元函数可以在类外部声明，在类内部定义；如果这个类是一个类模板，那么只有在实例化类时才会定义该函数：

```cpp
auto flag();

template <bool val> // 任意模板参数
struct flag_setter {
	friend auto flag() {}
};

int main() {
	// flag();  // error，无法推导 auto
	flag_setter<true>{};
	flag();  // ok，实例化了 flag_setter 并定义了 flag
}
```

注意函数的返回类型需要为 auto，这样当不存在类的实例化和 flag 的定义时，调用 flag 会因为没有定义、无法确定返回值类型而先推导失败（属于 SFINAE，不是严重错误）；如果不是 auto 而是固定类型，则会因为没有定义而直接导致编译失败（不是推导失败，属于错误），并且依然能找到之后实例化后的 flag。（不确定？）
**推导失败不是错误，仍可继续执行；推导时只会考虑当前位置及以前的函数定义，之后再定义函数不会影响 auto 推导和找不到定义。函数未定义是错误，直接导致编译失败；检查函数定义时会检查整个翻译单元中是否有该函数。**

通过这个逻辑并检查`flag()`是否是合法表达式，就可以实现0到1的计数器：

```cpp
auto flag();

template <bool val>
struct flag_setter {
	friend auto flag() {}
};

template <bool condition = requires { flag(); }>
consteval int value() {
	if constexpr(!condition) {
		constexpr flag_setter<false> tmp;
		return 0;
	} else {
		return 1;
    }
}

int main() {
	constexpr auto a = value();
	constexpr auto b = value();
	static_assert(a == 0 && b == 1);
}
```

> 模板实例化时会进行[两阶段名字查找](https://zh.cppreference.com/w/cpp/language/two-phase_lookup)：在第一阶段检查语法是否正确；在第二阶段（即实例化时）替换模板参数，再检查是否正确。
> （某些编译器将所有检查放在第二阶段）
> 所以某些不依赖模板参数的错误，在实例化前就可以被找到，导致 CE。
> 所以上述代码可能要修改`flag()`，让它依赖待决名，避免直接因为 flag 未定义而报错：
>
> ```cpp
> auto flag(int);
>
> template <bool val>
> struct flag_setter {
> 	friend auto flag(int) {}
> };
>
> template <int arg = 0, bool condition = requires { flag(arg); }> // arg 可以是任意模板参数
> consteval int value() { ... }
> ```

通过类模板`reader<N>`将友元函数 flag 扩展到 N 个（每个 reader 有一个），然后检查`flag(reader<N>{})`是否定义，就可以实现 0~N-1 的计数。

```cpp
template <size_t N>
struct reader {
    friend auto flag_counter(reader<N>); // 声明
};

template <size_t N>
struct setter {
	friend auto flag_counter(reader<N>) {} // 定义
};

template <size_t N = 0,
			auto tag = []{},
			bool condition = requires { flag_counter(reader<N>{}); }>
consteval auto next() {
	if constexpr (!condition) {
		constexpr setter<N> tmp; // 定义 flag(reader<N>)
		return N;
	} else { // 注意这里要 else，如果没有，也会在 condition 为 false 时生成该分支代码
		return next<N + 1>(); // 可以优化成指数级增长和检查
	}
}

int main() {
	constexpr auto a = next();
	constexpr auto b = next();
	constexpr auto c = next();
	static_assert(a == 0 && b == 1 && c == 2);
}
```

这里 flag 是在类内声明和定义的友元函数，只能通过 ADL 找到，也不会影响命名空间。

注意函数 next 中需要一个默认参数`auto tag = []{}`。
常量表达式是纯的，即如果函数的各实参都相同，它的结果也应当相同，所以编译器可以缓存对同一函数的调用结果，避免多次调用，不符合我们的预期。因为不同位置创建的 lambda 都是不同的类型（即使看起来相同），并且函数的默认参数会在每次执行时求值，所以**使用一个 lambda 作为默认模板实参，可以在每次调用函数时产生不同的模板实参、不同的函数和常量表达式**，从而避免编译器缓存结果。
使用`source_location::line`作为默认实参，也可以实现同样的效果。

## 约束 概念


**约束与概念 (constraints, concept)** / **concept**

[约束](https://zh.cppreference.com/w/cpp/language/constraints)指定了对模板实参的一些要求。可关联到涉及模板的类或函数，用于选择最恰当的函数重载和模板特化。
约束是实现 SFINAE 的简单方式。通过约束，可以更容易、更直观地发现问题（将错误限制在类型推断阶段，不进行实例化，避免大量的编译错误），并便于编写代码（比如如果约束 T 包含某成员，则 IDE 会给出提示）。
一个概念 (concept) 定义了一种约束。*requires 子句*也能代表一种约束。

C++20 前，通过 SFINAE 也能实现约束（见下）。

concept 通过 *requires 表达式*定义，可以放在模板实参前，约束该实参必须满足某些条件，否则编译错误。
它与 requires 子句类似，但 concept 只需一次定义就可用于修饰不同模板函数的参数。

```cpp
// 概念 Hashable 只能被满足以下条件的类型 T 满足：
// 对于 T 类型的值 a，表达式 std::hash<T>{}(a) 可以编译，且它的结果可以转换到 std::size_t
template<typename T>
concept Hashable = requires(T a) {
    { std::hash<T>{}(a) } -> std::convertible_to<std::size_t>;
};

template<Hashable T> // 受约束的模板函数
void f(T) {}

// requires 子句也可进行相同的约束
template<typename T>
    requires Hashable<T> // HashTable<T> 是一个 bool 值
void g(T) // requires Hashable<T>; // 放在函数后也可
{}

using std::operator""s;
struct A {};

f("abc"s); // ok，string 满足 Hashable
// f(A{}); // 错误：A 不满足 Hashable

// concept 的实例化可作为 bool 值使用
template<typename T>
concept is_destructible = requires(T v) { v.~T(); };

cout << is_destructible<int> << '\n'; // true，有伪析构函数

template <typename T>
struct is_destructible_or_array {
	static constexpr bool value = is_destructible<T> || std::is_array_v<T>;
};
```

concept 定义时可包含多个模板参数，当位于 requires 子句之外、约束类型时，该类型将作为第一个参数，剩余的实参才需要自己指定。
它可以指定非类型模板参数，但没什么用。

```cpp
template <std::same_as<int> T>
void f(T x); // 与 requires same_as<T, int> 相同

// 用于简写函数模板时，也放在 auto 的前面
void f(std::same_as<int> auto&& x);

// 另一个例子
template<class T, class U>
concept Derived = std::is_base_of<U, T>::value;

template<Derived<Base> T> // T 被 Derived<T, Base> 约束
void f(T);
```

在重新声明模板时，声明的约束必须与之前使用的约束相同，否则非良构。

**requires 表达式**

requires 分为两种：requires 子句、requires 表达式。
[requires 表达式](https://zh.cppreference.com/w/cpp/language/requires) 是 bool 类型的纯右值表达式，可定义一种约束，赋给 concept。

> 准确来说，concept 的初始化方式与 requires 子句相同？即可包含若干初等表达式（包括 requires 表达式和 bool 值）的逻辑连接。
>
> ```cpp
> template <typename T>
> concept C = requires(T x) {
> 				x.size();
> 				x[0];
> 			} || std::is_same_v<T, int>;
> ```

它可以使用模板参数、函数参数列表中的参数、任何可见的声明。
表达式不会被求值，只检查正确性。

格式为：`requires (参数列表) { 要求序列 }`。
参数列表可选，可用于为类型模板参数定义实例，以便在要求序列中使用。

要求序列有四种：

- 简单要求：要求某个表达式必须合法，如：`requires (T a, T b) { a+b; cout<<a; }`。
- 类型要求：包含 typename 和一个类型，要求该类型是有效的，如：某个特化是否存在。
- 复合要求：`{表达式} (noexcept) (-> 进一步的类型约束)`，要求某个表达式必须合法（同简单要求）；如果包含 noexcept，则表达式必须 noexcept；如果有进一步的类型约束（另一个 concept），则约束表达式的值的类型。
- 嵌套要求：包含其它 requires 表达式。

SFINAE 检查表达式合法的方式：

```cpp
// 基本是通过 decltype 和逗号表达式，依次验证表达式合法性，最后返回合适的类型

// #1 利用 void_t 等在模板参数中添加表达式
// 不支持前置自增
template <class, class = void>
struct has_pre_increment_member: std::false_type {};

// 特化识别支持前置自增的类型
template <class T>
struct has_pre_increment_member<T,
			std::void_t<
				decltype( ++std::declval<T&>() ), // 注意前缀++需要T&对象，T&&没法调用++！
				decltype(std::declval<T>().begin()), // 可验证多个
				typename T::container_type // 类型要求
			>
		>: std::true_type {};

// #2 同上
template <typename T, typename = decltype(cout << std::declval<T>())>
	// requires requires (T x) { cout << x; }
void print(T x) {
 cout << x << '\n';
}

// #3 利用函数重载，给匹配参数类型（即匹配 decltype 表达式）的函数特定的返回类型
// 判断 T 是类 string
template <typename T>
class is_std_string_like {
	template <typename U>
	static auto check(U* p)
		-> decltype(p->find('a'), p->length(), p->data(), int());

	template <typename>
	static void check(...);

 public:
	static constexpr const bool value =
		is_string_v<T> ||
		std::is_convertible_v<T, std::string_view> ||
		!std::is_void_v<decltype(check<T>(nullptr))>;
};

// 等价于
template <class T>
concept check = requires(T* p) {
	p->find('a');
    p->length();
    p->data();
};

template <class T>
struct is_string_like {
	static constexpr bool value = check<T> || is_string_v<T> || std::is_convertible_t<T, std::string_view>;
};
```

可以出现在 [constexpr if](https://zh.cppreference.com/w/cpp/language/if) 中判断类型是否满足某约束。

```cpp
template <typename T>
struct A {
	using Type = decltype([]() -> auto {
        // constexpr if 中，被舍弃(false)语句中的 return 不参与返回类型的推导
		if constexpr (requires(T x) { x + 1; }) {
			return 1;
		} else if constexpr (requires(T x) { std::sort(x.begin(), x.end()); }) {
			return std::vector<int>{};
		} else {
			return std::string{};
		}
	}());
};
A<double>::Type i = 1;
A<std::vector<int>>::Type v{};
A<void>::Type s{};
```

**requires 子句**

[requires 子句](https://zh.cppreference.com/w/cpp/language/constraints)通过 requires 引入，用来指定对 模板实参 的约束。
可以放在模板形参列表的后边，或作为函数声明符放在函数定义后、函数体前。

子句中，requires 后的表达式必须是常量表达式，并且是以下形式之一：

- 是初等表达式 (primary expressions)，包括 **requires 表达式（或一个 concept）**、字面量、单个标识符、lambda、任何括号包含的表达式。
- 通过 && 连接的初等表达式。
- 通过 || 连接的上述表达式。

最初的模板声明中的 requires 子句，将对后续的所有特化都生效。

```cpp
template <class T>
//  requires is_same_v<T, int> // ok
//	requires is_same_v<T, int> || is_same_v<T, double> // ok
	requires (!is_same_v<T, int>) // 带 ! 需要括号
void f(T);

// error：does not match any template declaration
template <>
void f(int);

// 子句后可以接 requires 表达式（定义了一个匿名 concept？）
template <typename T>
	requires requires(T x) { x.length(); } || std::is_same_v<T, int>
void g(T) {}
```

**same_as**

> is_same_v<T,U>是一个原子约束，is_same_v<U,T>也是一个原子约束，这两个原子约束是不同的原子约束，因此same_as必须重复is_same_v<U,T>和is_same_v<T,U>，使得两个原子约束都为same_as子约束，从而same_as<U,T>和same_as<T,U>的子约束为相同的两个约束，而不是让same_as只有一个偏序的子约束，从而调换顺序后变成不同的子约束。
> 如果两个约束之间没有蕴含关系，那么就无法判两个约束谁更严格，从而无法进行重载决议；假设有一个T同时满足两个约束，如果两个约束之间没有蕴含关系，那么就无法判两个约束谁更严格，从而无法进行重载决议。
