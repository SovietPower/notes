搜索过程：

- 信息采集：在互联网中发现、搜集信息和数据。通常，这个步骤是通过爬虫(Crawler/Spider)抓取网页来实现的。每个独立的搜索引擎都有自己的网页抓取程序爬虫。 爬虫顺着网页中的超链接，从这个网站爬到另一个网站，通过超链接分析连续访问抓取更多网页，被抓取的网页被称之为网页快照。
- 建立索引库：对收集到的信息进行提取和组织建立索引库。最重要的预处理工作是提取关键词，建立索引库和索引。根据应用场景的不同，其他可能的处理还包括去除重复网页、分词(中文)、判断网页类型、分析超链接、计算网页的重要度/丰富度等。
- 提供检索服务：由检索器根据用户输入的查询关键字，提供检索服务。接受到关键词后，系统在索引库中快速检出文档，进行文档与查询的相关度评价，对将要输出的结果进行排序，并将查询结果返回给用户。通常，为了用户便于判断，除了网页标题和 URL 外，还会提供一段来自网页的摘要及其他信息。

 **全文搜索引擎**
　　全文搜索引擎是名副其实的搜索引擎，国外具代表性的有Google，国内著名的有百度。它们都是通过从互联网上提取的各个网站的信息（以网页文字为主）而建立的数据库中，检索与用户查询条件匹配的相关记录，然后按一定的排列顺序将结果返回给用户，因此他们是真正的搜索引擎。
　　从搜索结果来源的角度，全文搜索引擎又可细分为两种，一种是拥有自己的检索程序（Indexer），俗称“蜘蛛”（Spider）程序或“机器人”（Robot）程序，并自建网页数据库，搜索结果直接从自身的数据库中调用，如上面提到的7家引擎；另一种则是租用其他引擎的数据库，并按自定的格式排列搜索结果，如Lycos引擎。
      在搜索引擎分类部分提到过全文搜索引擎从网站提取信息建立网页数据库的概念。搜索引擎的自动信息搜集功能分两种。一种是定期搜索，即每隔一段时间（比如Google一般是28天），  蜘蛛搜索引擎搜索引擎主动派出“蜘蛛”程序，对一定IP地址范围内的互联网站进行检索，一旦发现新的网站，它会自动提取网站的信息和网址加入自己的数据库。另一种是提交网站搜索，即网站拥有者主动向搜索引擎提交网址，它在一定时间内（2天到数月不等）定向向你的网站派出“蜘蛛”程序，扫描你的网站并将有关信息存入数据库，以备用户查询。由于近年来搜索引擎索引规则发生很大变化，主动提交网址并不保证你的网站能进入搜索引擎数据库，目前最好的办法是多获得一些外部链接，让搜索引擎有更多机会找到你并自动将你的网站收录。
　　当用户以关键词查找信息时，搜索引擎会在数据库中进行搜寻，如果找到与用户要求内容相符的网站，便采用特殊的算法——通常根据网页中关键词的匹配程度、出现的位置、频次、链接质量——计算出各网页的相关度及排名等级，然后根据关联度高低，按顺序将这些网页链接返回给用户。这种引擎的特点是搜全率比较高。





### 整体架构

1. 查询理解 (Query Understanding, QU)。
    1. 词法分析：去停词，分词。
    2. 查询变换/改写：拼音转文字，同义词替换（归一化），改写，纠错，补全，扩展，提升匹配度。
    3. 短句分析：计算变换后的各个 term 的权重，决定哪个 term 影响更大，提升查询相关性。
    4. 实体识别：识别品牌名、商品名、地理信息等实体信息，更精细地匹配用户意图。
    5. 意图识别，类目预测：识别用户要查询的类目/感兴趣的信息，保证最后结果的相关性。
2. 召回。
3. 粗排。
4. 精排。
5. 重排。





### 查询理解 (QU)

> https://zhuanlan.zhihu.com/p/393914267

对 Query 进行深入理解，保证召回的数量和最终排序精度。

去停词：去除停止词，比如“的”。

归一化：过滤特殊符号，统一大小写，繁体转简体，全角转半角。

**query 改写**

相同的意图往往有不同的 query 表达方式，仅依靠字面分词匹配，会损失一定的召回，在候选量有限的情况下非常容易出现空结果或少召回。查询改写使用同义词替换原 query 的不同词，生成多个同义 query，可以解决该问题。

改写后会存在同义或近义的多个 query，一般 query 判别器会对候选的 query 进行排序，然后选取 TopK 的相似候选 query 集合再进行召回。召回时候选 query 中相同的分词可以只召回一次，比如`华为手机`可以被改写成`huawei手机`，在查询时可以`手机 and (华为 or huawei)`来进行查询。

**term weighting**

term 的权重可以分为两类：

- 静态 term weight：权重不随上下文而改变。
    简单，无需复杂算法，但准确性低，对长尾 term 效果差。
- 动态 term weight：从 query 整体考虑每个 term 的权重。
    需要通过特征工程、深度学习等方式对权重进行拟合。



### 相关性匹配

> https://zhuanlan.zhihu.com/p/351163627

倒排只能找到包含 term 的 doc，但不能保证这些 doc 的相关性。为了在语义层面上识别结果是否相关，还需要相关性匹配。

相关性计算可以看作是 query 和 item 的匹配分的计算，query-item 相关性分可以由单个或多个匹配特征分融合得到。
在搜索的不同阶段，一般会根据实际情况采用不同的方法进行相关性度量。比如：在候选较多的粗排阶段，一般采用简单的匹配特征做排序（如类目匹配、BM25、TFIDF 表示匹配分）；而在候选较少的精排阶段，可以采用复杂特征分或多特征分融合的方式，甚至使用在线的深度匹配模型。

> 搜索一般有明确的查询意图，所以相关性更重要，非常影响体验。
> 相关性一般不考虑用户行为和统计特征，只考虑准确度。

搜索相关性特征可以分为三类：

- 结构化匹配特征：query-item 类目匹配；属性信息匹配（品牌、型号等）。
    计算复杂度低。
- 文本匹配特征：term 匹配数、query term 匹配率、item term 匹配率（匹配的 term 在商品中的 term 占比）（可以根据 term 带权重），编辑距离，BM25，紧密度，PMI 等。
    简单有效。
- 语义匹配特征：对于同一语义的多种表述，即使引入同义词信息有时也无法有效处理。语义匹配很重要。
    - 隐语义表示匹配：将文本表示成隐语义向量，并通过向量距离来度量文本之间的相似度。
        比如：LDA，word2vec，点击图表示匹配。
    - 深度语义匹配：通过神经网络进行文本的隐语义表示匹配。
        包括：表示型语义向量匹配，交互型语义向量匹配。
    - 多模态语义匹配。

可以通过用户行为来拓展文本匹配特征。比如：对于用户查询和点击的`<query, item>`对，通过协同过滤等方式，可以得到相似的 query 列表 (query2query)；聚合 key query 的所有相似 query，然后进行分词，用统计或关键词提取的方法，选择最关键的一批 term 作为 key query 的表征 term，然后对所有 query 的表征 term 进行加权求和、归一化得到 query 的最终向量表示；item 侧也可以做同样处理得到向量表示，然后可使用向量距离计算 query 与 item 的相似度。





### 查询推荐

query 推荐有两个目标：

- 提升渗透率，将用户引导到成交效率更高的搜索场景，提升搜索活跃度。比如：搜索底纹、query push（通过近期行为、热榜主动推送）、风向标。
- 搜索效率优化，节省用户输入成本，细化搜索词，预测用户搜索兴趣，同时挖掘用户需求，并进一步拓宽用户搜索宽度。比如：Query sug、历史搜索、猜你想搜、细选、纠错、空结果反馈等。

**Query Suggestion**

根据用户当前搜索词前缀，实时猜测用户希望搜索的 query，进行 query 推荐。
实现方式包括：召回阶段根据事先建立好的前缀词典召回候选 query；或使用相似 query 算法召回（基于内容或行为的 q2q），兼容拼音、错别字等，一定程度放松前缀一致的要求，排序阶段可简单按照候选 query 历史搜索 pv、uv 或点击/成交率进行排序，也可基于简单特征和模型（如 LR）进行 qeury 点击率预估。
因为对实时性要求高，不会用复杂模型。

**猜你想搜**

根据用户当前搜索 query，推荐相似、略发散的 query。对于泛 query 则可以推荐更加收敛的 query（如细选）。通常直接目标为坑位点击率，间接目标可以根据需要设计为搜索丰富度（如人均搜索类目数）、整体成交效率（如人均成交）等。

**i2q**

可以根据用户行为统计 item 历史的搜索 query，按点击率、成交率、曝光数等维度排序可以得到 i2q 结果；如果没有行为数据，可以参考 i2tag2q，如使用 item 的 title 的核心 tag 召回候选 query。







### 其它

**搜索的评价标准**

精准性、时效性、响应速度、权威性。

**精准词**

与产品匹配度极高的词（一般就是准确的产品名字），也是用户名搜索频率最高的一些词。

**核心词**

对应方向中搜索量高的词，可能不精确，但搜索频率非常高。

**长尾词**

长尾关键词 (Long Tail Keyword) 是组合关键词，不是目标关键词，但与其相关也可带来流量。
长尾词比较长，可能由多个词和短语组成。许多平台的下拉框推荐词就是长尾词。
与尾对应的是头，头部指的是流量大、需求高的词，尾部的需求是个性化、零散、小量的。

长尾词的搜索量相对精准词/核心词更低且不稳定，但目的性更强、意图明确、能更准确地描述产品，且竞争强度更小，客户的转化率要高很多。
长尾效应指：将市场细分到很细很小时，虽然单个领域的效益很低，但因为这些领域足够多，这些细小市场的累计也会带来明显的效益。

### 召回

排序最底层是召回层 (recall, matching)，该层候选一般是千万到亿级别的商品，主要目标是从海量的商品池中尽可能多地召回与 query 相关的商品。由于候选较多，该层算法的核心是策略简单、计算复杂度低，同时尽可能保证搜索查询和候选商品的相关性。召回层也需要过滤并选出适量的商品交给下游，如依据简单的类目匹配、商品发布时间、历史销量等简单维度进行截断。

> 召回后可能还有过滤层，基于业务规则进行过滤，比如：内容有黑词、无库存等。
> 过滤也可能放在粗排后。

电商搜索召回架构往往同推荐系统一样采用多路召回架构，多路召回后将结果归并、截断，得到最终的召回结果送给下游。
多路召回包括：

- **文本召回**：最基础的召回是基于倒排索引的文本召回，即通过 query 词直接检索搜索引擎，召回包含所有 term 词的商品。

> 由于召回要求 term 精准命中，常常会出现召回能力不足的问题，因此通常配合扩召回策略。常见的扩召回（基于文本）从两个维度出发：query 侧和物料侧（商品）。query 侧的扩召回一般就是 query 理解中的改写。物料侧的扩召回通常是扩充召回字段，如：增加商品详细描述、商品品类、属性等基础文本字段，或通过相似商品的基础文本信息作为本商品的扩充索引字段。

- **结构化召回**：根据商品结构化信息召回。

商品结构化信息包括类目、属性等，属性有多个维度，比如：品牌、型号、尺寸、适用性别年龄等。
通常需要 query 和 item 端同步进行优化：query 侧需要识别出目标的类目或属性 ID，商品侧需要建立相应的 ID 索引。

该方法除了用于基础召回，还用于搜索导航（根据层级类目导购、根据属性进行细选）等。

- **个性化召回**：基于文本和结构化的召回是基于对文本的内容理解，完成 query 和商品的匹配；个性化召回则是基于用户行为的召回方法。

常见的个性化召回方法有：

1. q2i2i：获取当前 query 下曝光、点击、成交的 item，然后通过 i2i 的方法（协同过滤、swing 等）扩召回更多候选商品。通过类目限制（要求 trigger item 和 cand item 的类目一致）或其它离线相关性度量方法来保证召回商品和搜索 query 的相关性，也可以直接通过 query 和召回商品的相关性度量保证相关性。
2. u2i2i：通过 user 历史曝光、点击 item，召回更多的候选商品，并通过相关性策略进行不相关商品过滤。
3. u2s2i：召回 user 偏好的 seller 下和 query 相关的商品，作为 query 的候选商品。

相对于文本召回，个性化召回的优势为：不需要精准的 term 命中即可召回相关的商品，且商品天然具有高点击/成交属性。
缺点为：如果相关性把握不好，会引入一些影响搜索体感的 bad case。

- **向量召回**：基于文本的召回要求 query term 和商品信息的 term 精准命中，向量召回则可以实现基于隐式语义的相关召回。











### 粗排

粗排层 (pre-ranking) 也叫海选层。该层的目的是从上万级别的候选中筛选出合规、相关且成交效率高的topN商品，目标为通过计算复杂度低的排序算法实现 CTR 或 CVR 等目标的预估。
由于对计算性能要求较高，粗排的经典做法是使用带简单交叉特征的 LR 排序模型或者深度双塔结构做排序模型进行效率目标预估，单个样本的特征少。而在一些相关性要求较高的场景下，粗排阶段也会涉及一些相关性匹配策略。

粗排中的排序分主要考虑三点：

1. 业务规则分，如：运营圈品提权、低信誉商家降权等。
2. 相关性分，如：类目匹配分、关键词匹配分等。
3. 成交效率相关的排序分，如：静态商品/卖家分、CTR、CVR 等。

它们会有不同的权重。前两点用于约束，能否选出高成交效率的商品主要依赖第三点。

出于计算性能考虑，粗排模型的选择有限，最经典的有：商品静态分、线性回归、DSSM 双塔模型。

- 静态排序分：是最简单的效率排序分，比如：静态商品质量分（标题质量、图片质量、价格分等，通常由内容理解、历史统计信息得到）；商品/卖家效率分，仅考虑商品和卖家等维度特征的 CTR/CVR 预估分（可以用 LR, DNN 等模型预估）；商品粒度分（新品、需扶持商品分等）。
    通常可以离线计算，极大节省线上预测的开销，但有时效性差、预估分不够精准的问题。
- LR：逻辑回归模型预测效率高，适合粗排。可以综合考虑 user、query、context 和 item 的特征，性能允许的情况下也可使用更多的交叉特征。
    相比静态分，LR 更加准确，但模型容量有限。
- 双塔模型：与 LR 类似，也可综合考虑各特征，同时可通过深度网结构络提升特征表达能力。

### 精排

精排层 (ranking) 部分的候选约在百到千，目标往往和粗排一致（预估 CTR、CVR 等），主要区别在于预估模型复杂度更高、单个样本的特征更多、计算密度大，如经典的大规模交叉特征的 LR、GBDT+LR 或深度排序模型。
通常是用 pointwise 方式对商品的 CTR/CVR 进行预估、打分。











### 重排

由于粗排和精排多是用 point-wise 方式对商品的 CTR/CVR 进行预估（仅考虑 query 本身和商品的关系，可以看做是每个 item 的分类），常常未考虑商品之间的关系 (pair-wise) 或未考虑整体排序 (list-wise) 的合理性，并且精排候选可能上千，无法应用更复杂的模型，因此有时需要对精排结果进一步重排。
重排 (reranking) 的目标一般有三种：

1. 实现更精准的打分以获得全局视角下更好的排序效果，该目标实现方式往往使用更复杂的模型、考虑 list-wise 方式的排序或是基于交互反馈在线强化学习等。
2. 满足特定业务诉求/特殊整体性的排序需求，常见的整体性需求包括：搜索结果的多样性（如价格、地域、品牌、风格、卖家、描述等同属性的打散，避免同质化）、发现性（帮用户发现多种商品、新的兴趣）、异质内容的混排调控（如商品、广告、视频内容等物料的混排）、流量调控等。
3. 关注序和上下文。

## 预估



### 预估模型

CTR 预估模型是针对每个用户的每个搜索词，预测搜索结果 (doc) 潜在点击量的模型。通过 CTR 预估模型，能够优化搜索结果的排序，提升搜索效果和业务转化。

好的 CTR 模型应该在提高 CPM 的同时，尽可能保证 CPC 不涨。即应该能更准确地预估点击率，避免不必要的曝光，但不会导致点击量过多下降。
曝光量和点击率应该保持平衡：盲目提升曝光量能够提高点击量，但无法有效/同比提升点击量，会降低转化率、降低广告方的 ROI、间接影响投放方的收益；单纯追求高点击率可能导致曝光量不足，影响平台的收益。

> CPC：每次点击计费（平均点击价格） = 广告主支出 / 点击量。
> CPM：每千次展现计费（每千次投放价格） = 广告主支出 / 曝光量 \* 1000。
>
> 如果广告目标是提高品牌曝光，可以适当以提高曝光量优先；如果广告目标是促进转化，则应更注重点击率的优化。

**模型**

> 预估模型的输入可以是用户 ID + 物料 ID 列表，输出是每个物料的预估结果（打分）（中间会对物料进行特征提取？）。预估结果一般是浮点数或浮点数集合，主要功能包括特征计算，模型计算，以及样本特征的 DUMP 管理。
>
> 模型的特征数据一般是 sample/slot/feature 三层嵌套结构，即每个候选可包含多个 slot，每个 slot 可产出多个特征。

> 模型压缩主要有三个方向：行即特征数量；列即 embedding 向量的维度；值的精度 (FP32, FP16, Int8)。



**模型训练**

> 模型可能包含三步：公共样本补全/物料样本补全（预处理）、公共特征计算/物料特征计算（特征工程）、本地模型推理/远程调用推理。
>



**Pointwise**, **Pairwise**, **Listwise**

> https://zhuanlan.zhihu.com/p/659467763
>
> CTR 预估中，三者都有可能更好，需要看实际情况。

- Pointwise：模型的训练目标是单个文档的用户反馈（点击率/转化率/相关性等）。

训练数据以单个文档为单位，不考虑其它在同一列表中一起展示的文档。训练就是二分类，预测一个文档是否会被点击。
模型推断对每个文档独立处理，结果通常是一个有含义的概率值（比如 predictedCTR）

- Pairwise: 模型的训练目标是把好的文档排在差的文档的前面。

训练数据以 (好文档, 差文档) pair 的形式构造。
训练好的模型，推断依然对每个文档独立处理，预测的分数没有特定数学含义，但是模型会尽量使得好的文档的分数比差的文档的分数高。

Pairwise 尽量把用户正反馈的文档 排在用户负反馈的文档前面，因此最契合排序目标。
可以拟合到更有区分度的函数，区分两个可能本身区分度就不高的文档。并且它是一个相对更困难的问题，会强迫模型去学习真正有价值的东西，而不是走捷径，因此可能效果更好、泛化能力更强。

- Listwise: 模型的训练目标是整个文档列表的排列尽可能最优（最优的定义可以采用类似NDCG之类的指标）。

训练数据以列表为单位。
模型推断可能每个文档独立进行，也可能对整个列表进行（复杂度较高，较少采用）。







### 异质训练

> 深度模型的在线推理效率是至关重要的，需要在一定延迟约束下，进行大量深度学习的计算，为此在推理异构计算方面RTP集成了专门的异构推理计算加速引擎，可以同时支持FPG、CPU、GPU、阿里NPU等各种各样的异构硬件，为业务在深度学习领域提供了巨大的算力。

- CPU 主要包括：计算单元 (ALU)、控制单元 (control unit)、寄存器、高速缓存。
- GPU 包括大量的 ALU 和超长的流水线（和一个共用的内存？）。适合处理大量类型统一的数据。
- TPU (Tensor Processing Unit，张量处理单元) 是专用于深度学习的芯片。包括用于高效执行矩阵运算的张量核心、高带宽的存储系统以及专用的硬件加速器。针对机器学习中的张量运算进行硬件优化，能够大幅提高深度学习模型的训练和推理速度。
- NPU (Neural Processing Unit，神经处理单元) 是专用于执行神经网络计算的硬件加速器。包括神经网络加速引擎、专用的存储和数据传输设计，以及针对神经网络计算的功能单元。
    专门针对神经网络计算进行硬件优化,能够高效地执行神经网络的推理计算，且具有低功耗的特点，非常适合部署在移动设备和边缘设备中。
- GPGPU (General-Purpose computing on Graphics Processing Units)：通用图形处理器，利用 GPU 来加速处理 CPU 的通用计算任务，而不仅仅是图形处理。

> GPU 通常使用图形 API 如 OpenGL 或 DirectX 进行编程，GPGPU 则使用通用并行编程模型如 CUDA 或 OpenCL 进行编程。
> GPU 更注重图形处理的硬件架构，如纹理采样器、光栅化单元等；GPGPU 更注重通用并行计算的硬件架构，如更多的计算单元、内存带宽等。

### 术语

**指标**

**CTR** (Click Through Rate)：点击率(点曝比) = 点击量 / 展现量。代表用户是否对推荐的内容感兴趣。

**CVR** (Conversion Rate)：转化率 = 转化次数 / 点击量。一个统计周期内，完成转化行为的次数占推广信息总点击次数的比率。

**CPC** (Cost Per Click)：每次点击计费（平均点击价格） = 广告主支出 / 点击量。
计费：广告主广告投放成本，即平台的收益。
广告通常按点击计费，因此平台更倾向展示 CTR 多的商品，以提高收益。

**CPM** (Cost Per Mille)：每千次展现计费（每千次投放价格） = 广告主支出 / 曝光量 \* 1000。
相同流量情况下，CPM 高的广告收入更多。

**eCPM** (Effective Cost Per Mille)：千次展示平台实际收益 = 平台收益 / 曝光量 \* 1000。
eCPM 通常比 CPM 更低，因为广告收益需要扣除渠道分成、流量成本等费用。
结合了曝光量和收益，用来评估广告投放的盈利/流量变现能力。

**CPA** (Cost Per Action)：每次转化计费 = CPC / CVR。

**CPV** (Cost Per View)：每次展示计费（一次独立 IP 展示）。

**PV 单价**：每 PV 的收入，衡量页面流量变现能力。

**RPS** (Revenue Per Search)：每次搜索产生的收入，衡量搜索结果变现能力指标。

**ROI** (Return On Investment)：投资回报率/投入产出比。如：广告产生的总销售金额 / 广告花费。



**基础**

**PV**：页面浏览量。

**UV**：页面不重复的访客数。

**GMV**：商品交易总额（一段时间内）。一般也会包括未支付和未最终成交的订单金额。

**ATV** (Average Transaction Value)：客单价，每个顾客平均购买的商品金额，即平均交易金额。
电商推荐注重该指标。

view/impression：展示/展现量。

**ADimp** (AD impression)：单个广告的展示次数。

**ADPV** (Advertisement Page View)：载有广告的 PV 流量。

**DAU** (Daily Active User)：日活跃用户数量。

**MAU** (Monthly Active User)：月活跃用户数量。

**ACU** (Average Concurrent Users)：平均同时在线人数。

**PCU** (Peak Concurrent Users)：最高同时在线人数。

**DSR** (Detail Seller Rating)：卖家服务评分系统。

**DMP** (Data Management Platform)：数据管理平台。



**其它**

**session**：用户在一个时间段内（如 10min）的搜索行为可以认为是一个 session，同一个 session 内的搜索词有较大的概率是同类目的。
通过大量用户在一个 session 内的搜索词，可以建立相关性。

**pv**：属性。如：property: 性别, value:女。

**RT** (Return Time)：响应时间。
