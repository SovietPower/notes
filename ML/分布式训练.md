由于当今的GPU拥有大量的显存，因此在实际场景中（不包括图深度学习）只有数据并行这种并行训练策略值得推荐？



### 具体方式

一轮批量训练：

1. 每个计算服务器 (worker) 读取小批量数据中的一部分。
2. 每个 worker 将自己的数据切分到各个 GPU 内存上。
3. 每个 worker 从参数服务器获取模型参数，将参数从内存复制到每个 GPU 上。
    参数服务器可以有多个，分别保存一部分的模型参数。训练时 worker 向所有的服务器分别请求参数。
4. 每个 GPU 计算损失和梯度。
5. 每个 worker 将自己所有 GPU 的梯度求和，一起传回服务器。
6. 每个参数服务器对不同 worker 的梯度求和，更新参数，将参数重新广播给各 worker 和 GPU。

这里所有 worker 都是同步计算同一个批量，称为同步 SGD。
假设有 n 个 GPU，每个 GPU 每次处理 b 个样本，那么同步 SGD 等价于在单 GPU 运行批量大小为 n*b 的 SGD。
在理想情况下，n 个 GPU 可以得到相对个单 GPU 的 n 倍加速。

模型需要有较小的通信计算比。通信量即 model size，计算量即 FLOPS。

> 在参数服务器汇总梯度时，汇总过程和参数发送（通信）是可（一定程度）并行的：
> 反向误差传递时，每计算完一层的梯度、更新完该层参数，都可以进行发送。





> 分布式训练可以增大 batch size：原先单机但 GPU 1s 可以计算 1 个样本的误差；分布式后 n 台机器、每台机器 m 个 GPU，1s 可以计算 n*m 个样本的误差。因此可以用很短的时间完成一个 batch 的误差计算，然后（先在本地汇总一次，减少发送量）给服务器汇总和梯度下降。
>
> batch size 达到一定程度，每个batch内的样本的多样性不会比之前有多大增长，对梯度的贡献也不会比之前的batch大多少，但是大的batch size 会带来更多的训练时间，就造成了训练有效性（训练效率）下降：
> 假设总共有10000个样本，如果一个batch是100，那么一个epoch可以迭代100次梯度；如果一个batch是1000，那么一个epoch只能迭代10次梯度，如果想要收敛，意味着需要更多epoch。如果两者得到的梯度差不多，那么后者显然会影响训练效率。
>
> 比如：图片分类，如果分 10 类，那 batch size 一般最好在 10n、20n 以内。
>
> 如果数据集足够大、多样性好，或优化算法足够好，那么可以使用更大的 batch size 训练、不会影响梯度和效率。
